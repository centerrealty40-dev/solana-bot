# Solana Alpha — операционная модель релизов и изменений

**Продукт:** Solana Alpha (исследовательская платформа и будущий торговый контур на Solana).  
**Статус документа:** нормативный (обязателен для любых изменений, затрагивающих исполнение стратегий, журнал сделок, деплой или параметры риска).  
**Связанные артефакты:** [`VERSION`](./VERSION), [`CHANGELOG.md`](./CHANGELOG.md), [`PARALLEL_WORKFLOW.md`](./PARALLEL_WORKFLOW.md) (несколько агентов и один деплой), [`DIAGNOSTIC_SCRIPTS.md`](./DIAGNOSTIC_SCRIPTS.md) (классы A/B/C для `scripts-tmp/`).

---

## 1. Цели и инварианты

### 1.1 Зачем этот документ

- Чтобы **мелкая правка** (ладдер, таймаут, фильтр входа) не «пропадала» из‑за второго источника правды, ручного патча на сервере или деплоя без фиксации в git.
- Чтобы **рестарт процесса** (PM2, VPS, авария) не менял поведение открытых позиций из‑за неполного восстановления состояния.
- Чтобы для каждой выпущенной версии было ясно: **что изменилось**, **как откатить код**, **что откат кодом не чинит** (данные, журналы).

### 1.2 Инварианты безопасности (нарушение — стоп фиче до исправления)

| ID | Инвариант |
|----|-----------|
| **I1** | Боевые параметры стратегии задаются только в **версионируемом репозитории** (`ecosystem.config.cjs`, `.env*` шаблоны, код с дефолтами). Нет «забытых» правок только на VPS. |
| **I2** | После загрузки журнала состояние открытых позиций **воспроизводимо**: повторный старт процесса на том же JSONL даёт тот же набор `ladderUsedLevels`, `remainingFraction`, ног DCA и т.д., что соответствует цепочке событий (см. §5). |
| **I3** | Любое изменение, влияющее на деньги или риск, проходит **минимальный чеклист** (§10) и фиксируется в **CHANGELOG** с номером версии (§4). |
| **I4** | Дашборд и маркетинговые тексты **не являются** источником правды для параметров стратегии; они либо генерируются из того же конфига, либо содержат явную пометку «см. ecosystem». |
| **I5** | Каждый путь к спецификации в **[`specs/INDEX.md`](../specs/INDEX.md)** (колонка со ссылкой на `.md`) **обязан существовать в том же коммите** ветки интеграции (обычно `v2`). Ссылка без файла в git — **дефект процесса**, а не «локальный черновик». |
| **I6** | Перед пометкой задачи выполненной и push в **`v2`** рабочее дерево **не содержит неучтённых** изменений в зоне продукта (`docs/strategy/**`, `src/**`, закреплённые утилиты в `scripts-tmp/`): всё либо **закоммичено**, либо **удалено**, либо явно игнорируется `.gitignore`. Иначе следующий агент и сервер расходятся с тем, что «кажется» сделанным по INDEX/дашборду. |
| **I7** | Файл **`VERSION`** и **новая** запись релиза в **`CHANGELOG`** для выката на **`v2`** меняет **только интегратор**, в **одном** интеграционном коммите **после** слияния задач в `v2`. Исполнители в ветках **`task/*`** **не бампят** semver и не выбирают следующий PATCH/MINOR «на глаз» по локально устаревшему `v2`. Перед выбором номера интегратор выполняет **`git fetch origin`** и сравнивает с **`origin/v2`** (или иной **согласованной** веткой интеграции): следующий semver считается от **удалённой** головы, не от непулленной локальной копии. |
| **I8** | Одновременно допускается до **пяти** параллельных исполнителей (агентов/веток) при соблюдении [`PARALLEL_WORKFLOW.md`](./PARALLEL_WORKFLOW.md): изолированные ветки или **`git worktree`**, **один** поток слияния в `v2`, **одна** версия продукта на выпуск, **один** деплой на этот SHA. |

## 2. Классы изменений

Используйте класс при заведении задачи и в сообщении коммита (`chore|fix|feat(scope): …`).

| Класс | Примеры | Версия (semver) | Обязательные проверки |
|-------|---------|-----------------|------------------------|
| **P0 — критический риск / деньги** | исполнение ордеров (live), ключи, перевод средств, изменение семантики выхода | **MAJOR** или **MINOR** + пояснение | полный чеклист §10, ревью, при live — поэтапный rollout |
| **P1 — логика paper/trading** | трекер, ладдер, SL/TP, восстановление состояния, impulse, liq-watch | **MINOR** или **PATCH** | `npm run typecheck`, релевантные тесты, смок после деплоя |
| **P2 — конфиг стратегии** | env в ecosystem для конкретного PM2-приложения | **PATCH** (если только числа) или **MINOR** (новые ключи) | сверка с §3, запись в CHANGELOG |
| **P3 — наблюдаемость / UI** | дашборд, тексты, метрики без изменения контрактов API | **PATCH** | регрессия отображения критичных полей |
| **P4 — документация / процесс** | этот документ, runbook | **PATCH** | ссылка в CHANGELOG |

Правило: при сомнении между классами выбирайте **более строгий** (больший scope версии и проверок).

## 3. Единый источник правды (SSOT)

### 3.1 Параметры стратегий (Oscar / Dno / Diprunner и др.)

- **Первичный SSOT:** блоки `env` в **`ecosystem.config.cjs`** для соответствующего `name: 'pt1-…'`.
- **Запрещено:** дублировать числовые параметры в `dashboard-paper2.html` или иных статических текстах без автоматической сверки. Допустимо: описание качественное («ладдер по остатку») + отсылка «актуальные числа — в ecosystem для `pt1-oscar`».
- **Желательная эволюция (не блокер для 1.0.0):** вынести параметры в один YAML/JSON в репозитории и собирать ecosystem или отдавать их через API дашборда — один файл, одна генерация.

### 3.2 Состояние позиций и журнал

- **SSOT истории:** append-only **JSONL** по пути `PAPER_TRADES_PATH` (например `data/paper2/pt1-oscar.jsonl`).
- **Производное состояние:** объект `OpenTrade` в памяти процесса должен быть **восстановим** из журнала после рестарта (replay событий `open`, `partial_sell`, `dca_add`, … до актуальной открытой позиции). Логика replay централизована в загрузчике store (например `store-restore.ts`); изменения в трекере, добавляющие новые поля состояния, **обязаны** сопровождаться обновлением replay или явным событием snapshot (§5.3).

### 3.3 Схема БД и миграции

- SSOT схемы — миграции в репозитории; версия схемы фиксируется в платформенных правилах при необходимости. Изменения, затрагивающие Postgres под Solana Alpha, помечаются в CHANGELOG отдельным подпунктом «Миграции».

## 4. Версионирование продукта (semver)

### 4.1 Где живёт версия

- Файл **[`VERSION`](./VERSION)** содержит ровно одну строку: **`MAJOR.MINOR.PATCH`** (без префикса `v`).
- История изменений — **[`CHANGELOG.md`](./CHANGELOG.md)**.
- Семантика следует [Semantic Versioning 2.0.0](https://semver.org/lang/ru/).

### 4.2 Когда повышать MAJOR

Повышайте **MAJOR**, если любое из нижеперечисленного **верно**:

- Меняется **контракт событий JSONL** так, что старый журнал без миграции несовместим с новым кодом (переименование обязательных полей, иная семантика `sellFraction` и т.п.).
- Меняется **поведение, влияющее на деньги**, так, что откат без изменения данных даёт неожиданный риск (live-контур).
- Удаляются или переопределяются публичные контракты, от которых зависят внешние потребители (если появятся).

### 4.3 Когда повышать MINOR

- Новая **фича**, обратно совместимая: новый флаг env с безопасным дефолтом, новый вид события JSONL (старое ПО читает журнал как раньше).
- Новая стратегия или новый PM2-процесс в том же продукте.
- Расширение replay без поломки старых файлов.

### 4.4 Когда повышать PATCH

- Исправления ошибок, рефакторинг без изменения контрактов.
- Только **тексты дашборда**, если не меняют SSOT параметров.
- Точечные правки ecosystem **в пределах того же смысла** (например bugfix порога).

### 4.5 Связь с git-тегами

- Рекомендуемый формат аннотированного тега: **`sa-alpha-X.Y.Z`** (префикс продукта, чтобы не путать с тегами монорепозитория).
- Тег ставится на **тот коммит**, в котором одновременно:
  1. обновлён файл `docs/strategy/release/VERSION`,
  2. добавлена запись в `docs/strategy/release/CHANGELOG.md`,
  3. (если применимо) прошли проверки §10.

### 4.6 Синхронизация с `package.json`

- Поле **`version`** в `package.json` может оставаться отдельным (npm-пакет).
- Рекомендация: при **MAJOR** релизе продукта опционально поднять и npm-версию для ясности; при **PATCH** продукта npm не трогать, если нет публикации пакета.

### 4.7 Параллельная работа и монополия на semver

- Полная схема веток, слотов агентов, локальных **`git worktree`** и чеклист интегратора — в [`PARALLEL_WORKFLOW.md`](./PARALLEL_WORKFLOW.md).
- **Исполнитель** коммитит только код/спеки по задаче; **интегратор** после merge в `v2` делает ровно **один** релизный коммит (`VERSION` + `CHANGELOG`), затем push и деплой — см. **I7**.
- **Опционально** при плотном окне параллельной работы: интегратор может **зарезервировать** следующий номер отдельным коммитом на `v2` (`chore(release): reserve X.Y.Z`, только `VERSION` + черновик строки в `CHANGELOG` или пометка «слот зарезервирован»), чтобы исполнители не гадали номер; резерв снимается тем же интегратором при финализации записи или отменяется revert’ом. Это не замена **I7**: «резерв» всё равно делает интегратор после **`git fetch`**.

## 5. Рестарт, replay и снимки состояния

### 5.1 Обязательный контракт

После **любого** рестарта процесса `papertrader` для данной стратегии:

1. Загрузчик читает JSONL с начала (или с оптимизированной точки, если позже появится индекс — но результат должен быть эквивалентен полному replay для открытых mint).
2. Для каждой открытой позиции восстановлены как минимум:
   - ноги и средние (`legs`, `avgEntry`, `totalInvestedUsd`, …),
   - **`ladderUsedLevels`** и **`remainingFraction`** согласованы с цепочкой `partial_sell`,
   - **`dcaUsedLevels`** и ноги DCA согласованы с `dca_add`,
   - прочие поля, влияющие на решение трекера, если они сериализуются в события.

### 5.2 Запрещено

- Полагаться только на последнюю строку `open` без учёта последующих `partial_sell` / `dca_add` при восстановлении (источник класса багов «ладдер снова с первой ступени»).

### 5.3 Опционально для live (этап 2)

- Периодические **`position_snapshot`** в JSONL с полным сериализованным `OpenTrade` + хеш конфигурации — ускоряют старт и упрощают аудит. Введение snapshot — изменение контракта; оформляется как **MINOR** с описанием в CHANGELOG.

## 6. Процесс внесения изменения (сквозной)

1. **Задача:** класс P0–P4, ссылка на spec/issue при наличии.
2. **Реализация:** только в git; правки на сервере без коммита запрещены (кроме аварийного hotfix с последующим немедленным переносом в репо в течение согласованного SLA). При **нескольких параллельных исполнителях** — ветка **`task/*`** или отдельный **`git worktree`** по [`PARALLEL_WORKFLOW.md`](./PARALLEL_WORKFLOW.md); без изменения `VERSION`/`CHANGELOG` в ветке задачи (**I7**).
3. **Локально:** `npm run typecheck`; релевантные `npm run test`.
4. **Документация версии:** bump `VERSION` по правилам §4; запись в `CHANGELOG` (что, зачем, откат). При параллельной работе этот шаг выполняет **только интегратор** на уже слитом **`v2`**, после **`git fetch`** и выравнивания с **`origin/v2`** (**I7**).
5. **Коммит:** осмысленное сообщение; для кросс-продуктовых изменений — соблюдение правил монорепозитория.
6. **Push** в согласованную ветку (например `v2`).
7. **Закрытие задачи (до или в том же merge, что push в `v2`):**
   - **`npm run check:hygiene`** — автоматическая проверка **I5** + формата [`VERSION`](./VERSION); перед push в `v2` — **`npm run check:hygiene:integration`** (ещё и **I6**, чистое `git status`).
   - `git status` без сюрпризов в `docs/strategy/**` и `src/**`.
   - Быстрая сверка: все `.md` из [`INDEX.md`](../specs/INDEX.md) в этом коммите присутствуют в дереве (**I5**).
   - Утилиты под `scripts-tmp/`, которые **должны** пережить сброс сервера (`git reset --hard`), — с осмысленным именем (как остальные `paper2-*.mjs`), без «потерянных» только на диске ноутбука или VPS.
8. **Деплой:** по §7; зафиксировать **SHA** на сервере после выката.
9. **Пост-деплой:** чеклист §10 для затронутых сервисов.

**Почему отдельный шаг 7:** документ с самого начала требовал SSOT в **git** и совпадения **SHA** на сервере (§7), но не фиксировал **чистоту рабочей копии** и **целостность INDEX**. В результате возможны ситуации «в списке спек уже есть ID, файла в репозитории нет» или «на ноутбуке лежит ROADMAP/strategy-скрипт, а в origin их нет» — это не нарушает I1–I4 явно, но ломает предсказуемость для следующего исполнителя. **I5–I6** закрывают этот зазор.

## 7. Деплой

### 7.1 Принципы

- Деплой = **доставка конкретного коммита** из git на сервер, а не «скопировали файл с ноутбука».
- После деплоя команда проверки: `cd /opt/solana-alpha && git rev-parse HEAD` совпадает с ожидаемым SHA.
- Запрещено **`git reset --hard`** без понимания, что будут потеряны незакоммиченные изменения на сервере; для рутины использовать известный хороший коммит или тег.

### 7.2 PM2

- Перезапуск затронутых процессов с **`pm2 restart … --update-env`** и при смене env — **`pm2 flush`** по политике проекта, чтобы не тянуть старый кеш env.
- После изменения списка приложений — **`pm2 save`**.

### 7.3 Резервное копирование перед рискованным деплоем

- Копия актуального JSONL затронутых стратегий и при необходимости дамп/снимок конфигурации.

### 7.4 SSH-доступ агента / автоматизации (единственный канон)

Чтобы исполнители (в т.ч. ИИ-агенты в Cursor) **не спрашивали владельца «как зайти на сервер»**, используется **только** связка ниже. Иное (`User botadmin`, DNS `*.hstgr.cloud`, другие ключи из `~/.ssh/config` на рабочей станции) для этого хоста **не подставлять** — конфиг пользователя может указывать другого пользователя или ключ, не совпадающий с `authorized_keys` у **`root`** на VPS.

| Параметр | Значение |
|----------|----------|
| Хост | **`187.124.38.242`** (только IP). |
| Пользователь SSH | **`root`**. |
| Ключ | Приватный **`c:/Users/cente/.ssh/botadmin_187_auto`**; в каждой команде явно **`-i c:/Users/cente/.ssh/botadmin_187_auto`**. |
| Каталог продукта на сервере | **`/opt/solana-alpha`**. |
| PM2 и файлы рантайма | как прежде под пользователем **`salpha`**: `sudo -u salpha pm2 …` |

Примеры:

```text
ssh -i c:/Users/cente/.ssh/botadmin_187_auto -o StrictHostKeyChecking=accept-new root@187.124.38.242 "<cmd>"
scp -i c:/Users/cente/.ssh/botadmin_187_auto <local> root@187.124.38.242:<remote>
```

Если вход по этим параметрам невозможен — **остановка** и эскалация владельцу (новый IP, ключ, сеть); самовольно менять пользователя или ключ **запрещено**.

## 8. Откат

### 8.1 Откат кода

1. Определить тег или SHA целевой версии (из `CHANGELOG` или `git tag -l 'sa-alpha-*'`). Для **точечного** отката одной «кривой» правки на `v2` допустим **`git revert <merge-commit или SHA>`** без смены тега — затем новый PATCH в `CHANGELOG` или объединение с следующим релизом по политике окна.
2. На сервере: выкатить дерево на этот SHA (тем же механизмом, что и обычный деплой).
3. Перезапустить те же PM2-приложения, что и при исходном релизе.
4. Проверить §10.

### 8.2 Откат конфигурации без отката кода

- Возможен точечный revert коммита, менявшего только `ecosystem.config.cjs`, с `pm2 reload --update-env`.

### 8.3 Откат ≠ восстановление журнала

- Откат бинарника/скрипта **не удаляет** и **не правит** строки JSONL. Если некорректные события уже записаны, нужен отдельный пост-мортем (ручная коррекция под контролем, или закрытие позиции вручную в live — по отдельной процедуре).

## 9. Тестирование и качество

### 9.1 Минимум для изменений трекера / restore

- Юнит-тест или интеграционный тест на **фикстурном JSONL**: «после replay открытая позиция совпадает с эталоном».
- При изменении парсера ладдера — тест на строку `PAPER_TP_LADDER`.

### 9.2 Регрессия дашборда

- Smoke: ключевые API отдают 200, таймлайн строится без падения.

### 9.3 Диагностические скрипты (анализ сделок / JSONL)

- Любой новый вспомогательный скрипт для разбора открытых/закрытых сделок оформляется по **[`DIAGNOSTIC_SCRIPTS.md`](./DIAGNOSTIC_SCRIPTS.md)**: класс **A** (утилита в репо) или **B** (одноразовый, префикс `_` + `.gitignore`), без смешения в одном файле.
- Запрещено оставлять неотслеживаемые «просто полезные» `.mjs` в `scripts-tmp/` без решения A/B и без завершения задачи (удаление или коммит).

## 10. Чеклист после деплоя (заполнять по факту)

**Стратегия / процесс:** `pt1-___` / deploy SHA `________`

- [ ] `git rev-parse HEAD` на сервере = ожидаемый SHA.
- [ ] `pm2 list` — нужные процессы `online`.
- [ ] В логах старта нет необработанных исключений за первые N минут.
- [ ] Для paper: heartbeat / JSONL пишется, размер файла растёт при активности.
