# Параллельная работа нескольких агентов и контролируемый деплой

Цель: много мелких правок в коротком окне **не превращаются** в хаотичные перезаписи сервера и «пропавшие» изменения.

Связанные документы: [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md), [`VERSION`](./VERSION), [`CHANGELOG.md`](./CHANGELOG.md).

---

## 1. Две роли

| Роль | Что делает |
|------|------------|
| **Исполнитель** (агент / разработчик) | Пишет код, коммитит в **свою** ветку задачи; может долго держать изменения **только локально** (без push), пока задача не готова. **Не** меняет [`VERSION`](./VERSION) и **не** добавляет запись релиза в [`CHANGELOG.md`](./CHANGELOG.md) — инвариант **I7** в [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md). **По умолчанию не деплоит на прод** сам по себе. |
| **Интегратор** (вы или назначенный агент) | Сводит ветки в `v2`, решает конфликты, гоняет `typecheck`/smoke, **один раз** поднимает версию (`VERSION` + `CHANGELOG`), **один раз** деплоит на сервер и фиксирует SHA. Перед bump — **`git fetch origin`** и выравнивание с **`origin/v2`** (**I7**). |

Если интегратор один и тот же человек — правило всё равно полезно: **деплой = отдельное осознанное действие**, не каждый коммит.

## 2. До пяти параллельных исполнителей на одном компьютере

**Лимит:** одновременно не более **5** активных линий работы (веток или worktree). Больше — резко растёт риск пересечений по файлам и дублирования semver без выигрыша в скорости.

### 2.1 Именование веток (слоты)

Фиксируйте **слот** `1…5` на время задачи:

| Слот | Имя ветки (шаблон) |
|------|---------------------|
| 1 | `task/agent-1-<кратко>` |
| 2 | `task/agent-2-<кратко>` |
| … | … |
| 5 | `task/agent-5-<кратко>` |

Правила:

- Пока ветка не **смержена в `v2` или не закрыта** (`git branch -d`), **не переиспользовать** тот же номер слота для другой задачи — так проще помнить «кто где».
- Старые схемы `task/a-<кратко>` / `task/b-<кратко>` остаются допустимыми, если слотов два; при **трёх и более** параллельных линиях переходите на **`task/agent-n-*`** во избежание путаницы.

### 2.2 Локальные изменения без путаницы

- **Коммиты локально** в своей ветке — нормально: история уже структурирована; когда задача стабильна — **`git push origin task/agent-n-...`**.
- **Несколько корней репозитория** (например несколько окон Cursor): предпочтительно **`git worktree`** — отдельная папка на каждый слот, чтобы агенты не перезаписывали друг у друга несохранённые файлы в одном дереве.

Пример (слот 3, база `v2`):

```text
git fetch origin
git worktree add ../solana-alpha-agent3 origin/v2
cd ../solana-alpha-agent3
git switch -c task/agent-3-liq-watch-tune
# работа, коммиты; по готовности:
git push -u origin task/agent-3-liq-watch-tune
```

Удаление worktree после merge: `git worktree remove ../solana-alpha-agent3` (из основного клона).

### 2.3 Что исполнителю нельзя

- Менять **`docs/strategy/release/VERSION`** и **`CHANGELOG.md`** под релиз (**I7**).
- Пушить в **`v2`** напрямую без роли интегратора (если не договорились иначе).
- Деплоить на прод из своей ветки.

## 3. Рекомендуемая схема веток (минимум столкновений)

1. **База:** ветка **`v2`** — всегда то, что *должно* быть на сервере после интеграции (или то, что уже задеплоено).
2. **Работа агента в слоте *n*:** ветка `task/agent-n-<кратко>` от **свежего** `origin/v2` (`git fetch` перед созданием ветки).
3. Другие слоты — так же, от той же головы `origin/v2`, если задачи независимы; если задача B зависит от незапушенной A — согласовать порядок merge с интегратором (временный merge `task/agent-1-*` в `task/agent-2-*` или ожидание push A).
4. Исполнители **пушат только в свои ветки на remote**, не в `v2` напрямую (если не договорились об обратном).

**Интегратор** раз в час (или после пачки задач):

```text
git fetch origin
git checkout v2 && git merge --ff-only origin/v2
git merge task/agent-1-...   # порядок по приоритету / зависимостям
git merge task/agent-2-...
# … до пяти веток за цикл
npm run typecheck
# при необходимости npm run test
```

Если два агента трогали одни файлы — конфликт решает интегратор **один раз**, а не два независимых деплоя.

После зелёных проверок — **один** коммит **`VERSION` + `CHANGELOG`**, затем push `v2` и **один** деплой на этот SHA (см. §5).

---

## 4. Почему нельзя «каждый сам деплоит» при параллельной работе

- Деплой часто делается как **`git reset --hard origin/v2`**: побеждает **последний** запушенный коммит на `v2`. Если А запушил, Б запушил, А задеплоил старый SHA — на сервере окажется не то, что «последнее на GitHub».
- Два деплоя подряд без синхронизации дают гонку: неясно, какой SHA реально на VPS.
- Версия продукта (`VERSION`) и `CHANGELOG` разъедутся с реальностью, если их бампает каждый по отдельному усмотрению.

**Вывод:** либо **один канал деплоя** (интегратор / CI), либо жёсткое правило: перед деплоем всегда `git fetch && git rev-parse origin/v2` и деплой **только этого SHA**.

## 5. Окно «час мелких правок», потом «организованно»

Практичный ритм:

| Фаза | Действия |
|------|----------|
| **Набор изменений** | Агенты пушат коммиты в **свои** ветки (или в общую `integration/hourly-N` — см. ниже). На сервер **не трогаем**. |
| **Слияние** | Интегратор мержит ветки в `v2` (или делает один squash-merge PR). |
| **Проверка** | `npm run typecheck` (+ тесты по затронутым областям). Обязательно **`npm run check:hygiene`** — целостность ссылок из [`specs/INDEX.md`](../specs/INDEX.md) и формат [`VERSION`](./VERSION) (**I5** в [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md)). Перед **`git push origin v2`**: **`npm run check:hygiene:integration`** (чистое дерево, **I6**). |
| **Версия** | Один коммит (можно последним в серии): обновить [`VERSION`](./VERSION), дописать блок в [`CHANGELOG.md`](./CHANGELOG.md) со списком изменений за окно (можно кратко: «batch: см. коммиты с SHA₁..SHA₂»). |
| **Публикация** | `git push origin v2`. |
| **Деплой** | Один прогон на сервер на **тот же SHA**, что только что на `v2`. Записать SHA в тикет/чат. SSH для агентов: **§7.4** в [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md) (канон `root` + IP + ключ, без вопросов владельцу). |

Так вы получаете: **много коммитов в истории**, но **одна зафиксированная версия продукта** и **одно состояние сервера** на конец окна.

## 6. Упрощённый вариант: общая ветка интеграции

Если не хотите плодить `task/*` на каждого агента:

- Создайте ветку **`integration/wip`** от `v2`.
- Все агенты пушат только туда (по очереди или с короткими коммитами).
- Раз в час интегратор: `integration/wip` → PR или merge в `v2`, затем bump версии, push, деплой.

Минус: чаще конфликты в одной ветке. Плюс: один поток, проще для «много мелочей».

## 7. Версия vs коммиты

- **Коммиты в git** — полная история; откатить можно на любой SHA.
- **Версия в `VERSION`** — **маркетинговый/операционный снимок** «что мы считаем выпущенным». Имеет смысл бампать **не каждый коммит**, а **на границе интеграции** (конец часа, конец спринта, перед live).

Допустимо:

- за час — 10 коммитов в `v2`;
- затем **один** коммит `chore(release): bump 1.0.2` с обновлением `CHANGELOG` списком изменений.

Либо: последний коммит в серии уже содержит bump — главное, чтобы **перед деплоем** на сервере была зафиксирована одна цифра версии, совпадающая с `CHANGELOG`.

**Точечный откат одной ошибочной правки** после merge в `v2`: предпочтительно **`git revert`** (отдельный merge-коммит или конкретный SHA), затем при необходимости новый **PATCH** в [`CHANGELOG.md`](./CHANGELOG.md) и деплой нового HEAD — см. §8.1 в [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md). Так история остаётся линейной и понятной для следующего окна агентов.

## 8. Чеклист интегратора перед деплоем

- [ ] **`git fetch origin`** выполнен; локальный **`v2`** выровнен с **`origin/v2`** до слияний и bump (**I7**).
- [ ] `git pull` / merge завершён, рабочее дерево чистое.
- [ ] `npm run typecheck` (и тесты по необходимости) зелёные.
- [ ] `npm run check:hygiene` зелёный; непосредственно перед push в `v2` — `npm run check:hygiene:integration`.
- [ ] `VERSION` и `CHANGELOG` отражают **этот** выпуск (или явно помечено «без bump — только деплой SHA», если политика временно ослаблена).
- [ ] Запланирован **один** деплой; после него `git rev-parse HEAD` на сервере записан.

---

## 9. Если всё же нужен срочный одиночный фикс на прод

1. Минимальный коммит в `v2`, push.
2. Деплой **сразу**.
3. В течение 24 ч дописать в `CHANGELOG` под **PATCH** с датой (ретроспективно) или следующим bump объединить с батчем — но **не оставлять** прод без записи в журнале релизов дольше согласованного SLA.

## 10. От полных локальных черновиков до чистого push

Цель — после серии параллельных работ получить **непересекающиеся** коммиты на `v2`, один semver и возможность отката.

| Шаг | Кто | Действие |
|-----|-----|----------|
| 1 | Исполнитель | В слоте `1…5`: ветка `task/agent-n-*`, коммиты локально и/или push в remote-ветку; **без** `VERSION`/`CHANGELOG`. |
| 2 | Исполнитель | Перед долгой паузой: **`git fetch origin`** в своём клоне/worktree, при необходимости **rebase** на `origin/v2`, чтобы уменьшить конфликты при интеграции. |
| 3 | Интегратор | **`git fetch`**, merge веток в `v2` в согласованном порядке, проверки §8. |
| 4 | Интегратор | Один коммит **`VERSION` + `CHANGELOG`**, **`npm run check:hygiene:integration`**, **`git push origin v2`**. |
| 5 | Интегратор | Один деплой на SHA = `origin/v2`; зафиксировать SHA. |
| 6 | Все | По завершении слота: удалить ненужные локальные ветки/worktree; на remote ветки `task/*` можно удалить после merge (политика команды). |

Полный откат окружения к прошлому выпуску — по тегу **`sa-alpha-*`** или SHA из [`CHANGELOG.md`](./CHANGELOG.md); операционно §8 в [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md).
