# Solana Alpha — норматив: диагностические скрипты (paper / стратегии)

**Продукт:** Solana Alpha.  
**Статус:** нормативный (обязателен для агентов и разработчиков при добавлении или завершении работы со вспомогательными скриптами анализа сделок и журналов).  
**Связано:** [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md), журналы paper [`specs/INDEX.md`](../specs/INDEX.md) (W6.3c), каталог **`scripts-tmp/`**, корневой **`.gitignore`**.

---

## 1. Цель

Разделить **повторяемые утилиты** (часть операционного контура) и **одноразовые расследования**, чтобы:

- на сервере после `git reset --hard` не пропадали нужные инструменты;
- не копился неотслеживаемый мусор в репозитории;
- не было спора «это продукт или временный файл».

---

## 2. Три класса

| Класс | Назначение | Жизненный цикл |
|-------|------------|----------------|
| **A — утилита репозитория** | Диагностика или отчёт, который **повторится** (разные стратегии, недели, агенты). | Коммит в **`scripts-tmp/`**, осмысленное имя (§3). При необходимости — npm-скрипт в **`package.json`** (§4). |
| **B — сессия / расследование** | Ответ на **конкретный** инцидент (одна сделка, один mint, один день). | Префикс **`_`** и шаблон в **`.gitignore`** (§5); после задачи — **удалить** или **промотировать в A** (§6). |
| **C — фрагмент без файла** | Одноразовый запрос (SQL, пять строк Node в чате). | **Не коммитить**; результат фиксировать в тикете/отчёте, если нужен аудит. |

**Обязательное правило исполнителя:** в начале задачи явно указать класс **A / B / C**. Если со временем выясняется, что **B** используется снова — инициировать промоцию в **A** в том же или следующем merge в `v2`.

---

## 3. Класс A — именование и содержимое

1. **Расположение:** только **`scripts-tmp/`** (или иной каталог, явно согласованный в spec; по умолчанию — `scripts-tmp/`).
   - **Исключение:** утилита, которая **импортирует продакшен-модули из `src/`** (например общий **`qnCall`**, типы live), может лежать в корневом **`scripts/*.ts`** и запускаться через **`tsx`** — как **`scripts/check-release-hygiene.mjs`** для релизной гигиены. Имя и шапка файла обязательны; npm-скрипт — по §4. Не смешивать с классом **B** (префикс **`_`** остаётся под `scripts-tmp/`).
2. **Имя файла:** префикс предметной области + глагол + объект, без «мусорных» префиксов `tmp-`, `new-`, `test2-`.
   - Для paper2 / JSONL стратегий: **`paper2-<действие>-<объект>.mjs`** (или `.ts`, если запуск через `tsx` осознанно зафиксирован).
   - Для коллекторов / инфры: существующие соглашения репозитория (`collector-*`, `orca-collector.mjs` и т.д.).
3. **Шапка файла (обязательно):** кратко — назначение; входы (`argv`, переменные окружения); пример команды; какие пути JSONL / какие стратегии (`pt1-oscar` и т.д.); ограничения (только read-only к журналам, если так).
4. **Поведение по умолчанию:** скрипт **не** должен мутировать боевые журналы без явного флага и предупреждения в шапке; предпочтительно только чтение и stdout.

---

## 4. Класс A — регистрация в npm (рекомендация)

Если утилиту запускают **чаще одного раза в месяц** или её должен вызывать агент по короткой команде:

- добавить скрипт в **`package.json`** вида `"paper2:<кратко>": "node scripts-tmp/paper2-….mjs"` (или существующий нейминг вроде `paper2:healthcheck`).
- в **первом** коммите, вводящем утилиту в постоянный контур, желательна строка в **[`CHANGELOG.md`](./CHANGELOG.md)** под **PATCH** («добавлена утилита X для …»), если утилита влияет на операционные решения или разбор инцидентов.

---

## 5. Класс B — одноразовые скрипты

1. **Имя:** **`_<тип>-<slug>.mjs`** (или другой префикс **`_`**, согласованный с корневым **`.gitignore`**).
2. Допустимые шаблоны игнора перечислены в **`.gitignore`** (`scripts-tmp/_diag*`, `_audit_*`, `_smoke*` и т.д.). Если нужен **новый** шаблон префикса — добавить строку в **`.gitignore`** в **том же коммите**, что и первое использование (или не использовать новый префикс).
3. **Запрещено:** ссылаться из **[`specs/INDEX.md`](../specs/INDEX.md)** или из нормативных spec на файлы класса **B**.
4. **Завершение задачи:** удалить файл **B** с рабочей машины или промотировать в **A** (§6). На сервере не держать единственную копию логики расследования только как неотслеживаемый файл — после `git reset --hard` она исчезнет.

---

## 6. Промоция B → A

Условия:

- тот же вопрос возник **повторно** или гарантированно возникнет у другой стратегии;
- скрипт **обобщён** (нет захардкоженного mint без `argv`);
- добавлены шапка (§3), при необходимости npm-скрипт (§4).

Действия: переименовать без ведущего **`_`**, убедиться, что путь **не** попадает под игнор, **`npm run check:hygiene`**, коммит в `v2`.

---

## 7. Сервер

- Доставка утилит **A** на VPS — только через **git** (тот же коммит, что и приложение), см. [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md) §7.
- Выполнение: от **`root`** SSH по §7.4 того же документа; если скрипт трогает только данные под `salpha`, по необходимости `sudo -u salpha` (права на файлы журналов).

---

## 8. Нарушения (дефект процесса)

- Утилита **A** без шапки и без понятного имени — доработать или понизить до **B** с `_` и игнором.
- Файлы класса **B** закоммичены в git «случайно» — удалить из истории при необходимости или переименовать в **A** по правилам §3.
- Ссылки из normative docs на несуществующие скрипты — то же семейство проблем, что **I5** в [`RELEASE_OPERATING_MODEL.md`](./RELEASE_OPERATING_MODEL.md).
