# W8.0 — Live Oscar: автоматический торговый бот (спецификация внедрения)

**Статус:** нормативный черновик для реализации (до merge кода — предмет ревью и bump версии продукта по [`../release/RELEASE_OPERATING_MODEL.md`](../release/RELEASE_OPERATING_MODEL.md)).

**Цель:** процесс **`live-oscar`** на том же сервере и той же Postgres, что бумажный **`pt1-oscar`**, с **параллельной** работой (четыре процесса: три paper + один live). Логика входа/выхода и параметры стратегии — **паритет с Oscar paper** (SSOT в репозитории); исполнение — **реальные swap-транзакции** с жёсткими лимитами и отдельным кошельком.

**Не входит в v1 этой spec:** мультикошельки, кросс-чейн, MEV-бандлы, собственный роутер вместо Jupiter.

---

## 1. Принципы и инварианты

| ID | Инвариант |
|----|-----------|
| **L1** | Бумажный **`pt1-oscar`** не получает секреты подписи; live-процесс — единственный носитель **`LIVE_WALLET_SECRET`** (или эквивалент). |
| **L2** | Журнал сделок и состояние открытых позиций live **отделены** от paper (`LIVE_TRADES_PATH` ≠ `PAPER_TRADES_PATH`). Replay после рестарта — по **live** JSONL, без смешения с paper. |
| **L3** | Решение «открыть / добавить DCA / закрыть часть» для live принимается только если проходят **и** логика стратегии (как у Oscar), **и** исполнительные гейты (лимиты потерь, свежесть цены, slippage, симуляция при dry-run). |
| **L4** | Postgres-снимки используются для **отбора кандидатов и контекста** (dip, объёмы, возраст пары). **Цена для расчёта размера ордера, TP/SL-проверок и фактического swap** перед отправкой — из **свежего** источника (Jupiter quote ± ончейн-верификация по политике §5). |
| **L5** | Лимиты проигрыша и дневные потолки заданы в **ENV** и не переопределяются без деплоя (нет «тихих» правок только на VPS). |

## 2. Архитектура процессов (один сервер)

```
Коллекторы → Postgres (pair_snapshots, …)
                 │
    ┌────────────┼────────────┐
    ▼            ▼            ▼
 pt1-* paper   pt1-oscar    live-oscar
 (JSONL)       (JSONL)      (JSONL + Keypair + RPC send)
```

- **Чтение БД:** те же таблицы снимков, что использует discovery Oscar (источник пула, mint, ликвидность в момент записи коллектором).
- **Запись БД live:** только явно спроектированные таблицы (аудит исполнения, опционально); не перезаписывать данные коллекторов и не путать SSOT paper.

Новый блок PM2 в **`ecosystem.config.cjs`**, например `name: 'live-oscar'`, `script` тот же entrypoint, что papertrader, либо отдельный **`live-trader`** entry point — решение реализации; в спецификации требуется **один** исполняемый процесс на стратегию live.

## 3. Переменные окружения (обязательная схема)

### 3.1 Идентичность и режим

| Переменная | Назначение | Пример |
|------------|------------|--------|
| `LIVE_STRATEGY_ENABLED` | Глобальный предохранитель | `1` / `0` |
| `LIVE_EXECUTION_MODE` | `dry_run` \| `simulate` \| `live` | см. §8 |
| `LIVE_STRATEGY_PROFILE` | Якорь паритета с paper | `oscar` |
| `LIVE_TRADES_PATH` | Абсолютный путь JSONL live | `data/live/pt1-oscar-live.jsonl` |

Паритет параметров с **`pt1-oscar`**: либо дублирование всех `PAPER_*` релевантных ключей с префиксом `LIVE_*`, либо **`LIVE_INHERIT_ENV_FILE`** указывает на общий фрагмент + только overrides — но итог должен быть **верифицируемым диффом** в git.

### 3.2 Кошелёк и RPC

| Переменная | Назначение |
|------------|------------|
| `LIVE_WALLET_SECRET` | Base58 secret key или путь к файлу keypair (предпочтительно путь + chmod 600) |
| `LIVE_RPC_URL` | QuickNode (или другой) HTTPS RPC для `sendTransaction`, `simulateTransaction`, account reads |
| `LIVE_RPC_TIMEOUT_MS` | Верхняя граница ожидания |

Отдельный RPC endpoint для live допускается (лимиты / метрики), но не обязателен.

### 3.3 Жёсткие лимиты риска (обязательны к реализации)

| Переменная | Смысл | Рекомендация для канарейки |
|------------|-------|----------------------------|
| `LIVE_MAX_POSITION_USD` | Нотационный потолок на одну открытую позицию | ≤ разумной доли депозита |
| `LIVE_MAX_OPEN_POSITIONS` | Одновременно открытых mint | `1` на первую фазу |
| `LIVE_MAX_TRADES_PER_DAY` | Лимит сделок (open + adds + partials считаются по политике в коде) | мало, например `5` |
| `LIVE_MAX_DAILY_LOSS_USD` | При накопленном реализованном убытке за календарный UTC-день — **стоп новых входов**, опционально принудительное закрытие | задать явно |
| `LIVE_MAX_SESSION_DRAWDOWN_USD` | От пика equity сессии (с момента старта процесса или с midnight — зафиксировать в коде одним способом) | задать явно |
| `LIVE_KILL_AFTER_CONSEC_FAIL` | Подряд неуспешных отправок / симуляций | например `3` → пауза до ручного сброса или `LIVE_STRATEGY_ENABLED=0` |

Дополнительно (желательно):

| Переменная | Смысл |
|------------|-------|
| `LIVE_MAX_REALIZED_LOSS_USD` | Пожизненный / с эпохи журнала потолок реализованного минуса |
| `LIVE_MIN_WALLET_SOL` | Не торговать, если SOL на комиссии ниже порога |

Все пороги читаются при старте и на каждом тике **до** постановки ордера; при срабатывании — событие в JSONL (`risk_limit`, `execution_blocked`) с причиной.

## 4. Какие API использовать для реальной покупки и продажи

### 4.1 Jupiter (агрегатор маршрутов)

Текущий paper-модуль уже использует **quote** (`lite-api.jup.ag/swap/v1/quote` или override — см. `price-verify.ts`). Для live нужны **два шага**:

1. **Quote** — актуальные `outAmount` / `priceImpactPct` / маршрут для заданных `inputMint`, `outputMint`, `amount`, `slippageBps`.
2. **Swap transaction** — POST на endpoint вида `…/swap/v1/swap` (актуальный URL из [документации Jupiter](https://station.jup.ag/docs/apis/swap-api)) с телом: `quoteResponse`, `userPublicKey`, опции приоритетной комиссии, `wrapAndUnwrapSol`, и т.д. Ответ — **сериализованная транзакция** (base64), которую подписывает `LIVE_WALLET_SECRET`.

**Рекомендация:** для продакшена получить **API key** Jupiter (Developer Platform), вынести base URL в ENV (`LIVE_JUPITER_SWAP_API_URL`), чтобы не упираться в лимиты lite-api и иметь предсказуемый SLA.

Итоговая цепочка buy (SOL → token):

- Убедиться в лимитах риска §3.3.
- Получить свежий quote (§5).
- Вызвать swap API → получить tx.
- Подписать локально.
- `simulateTransaction` через QuickNode (§8.2).
- `sendTransaction` с skipPreflight по политике (обычно preflight включён на первой фазе).

Продажа (token → SOL): симметрично, `inputMint` = mint токена, `outputMint` = wrapped SOL, `amount` = реальный баланс токена (с учётом decimals), прочитанный с сети через RPC.

### 4.2 Solana RPC (QuickNode)

Обязательные методы:

| Метод | Назначение |
|-------|------------|
| `getLatestBlockhash` | Свежий blockhash для подписи |
| `getBalance` / `getTokenAccountsByOwner` | Проверка SOL и ATA перед сделкой |
| `simulateTransaction` | Оценка успеха, logs, compute units **до** live отправки |
| `sendTransaction` | Фактическая отправка |
| `signatureSubscribe` или polling `getSignatureStatuses` | Подтверждение финализации (политика: confirmed vs finalized) |

Дополнительно для **верификации цены вне БД** (согласовано с W7.6):

- Чтение состояния пула через QuickNode (программа Raydium/Orca/Meteora и т.д.) — если в коде уже есть декодеры (например Whirlpool), использовать их как **второй источник** рядом с Jupiter для порога расхождения.
- Если декодера нет для конкретного `source` — полагаться на Jupiter quote + `priceImpactPct` + лимиты slippage (явно задокументировать снижение гарантий).

### 4.3 Priority fee

Использовать существующий механизм paper (`PAPER_PRIORITY_FEE_*`) как образец: для live — отдельные ключи `LIVE_PRIORITY_FEE_*` или наследование с override. Значение попадает в инструкции транзакции Jupiter swap (параметры swap API / отдельная set compute unit price — в зависимости от того, как Jupiter формирует tx).

## 5. Реальность данных Postgres vs «мгновенная» цена сети

### 5.1 Что даёт БД

- Снимки цен и ликвидности с **задержкой** относительно чейна (интервал коллектора, очередь, запись в PG).
- Для Oscar это **достаточно** для: фильтрации кандидатов, dip-windows, whale-контекста, возраста пары.

### 5.2 Чего БД **не** достаточно для live

- Точное исполнение TP/частичных продаж по «цене снимка минутной давности» может дать **проскальзывание иное**, чем ожидал трекер paper.
- Решение: **перед каждым live-действием** (open, DCA add, partial sell, full close):

  1. Запросить **Jupiter quote** с тем же notional / direction, что планируется.
  2. Сравнить исполняемую цену с **порогами стратегии** (например TP относительно средней входа) с явным допуском `LIVE_MAX_PRICE_DISAGREE_PCT` между «ценой трекера» (обновлённой через quote) и целевым уровнем.
  3. Если snapshot в БД старше `LIVE_MAX_SNAPSHOT_AGE_MS` для этого mint/pair — **не использовать** его как якорь для размера; только quote + on-chain balance.

### 5.3 Политика «цена для трекера»

- В paper уже есть путь «подставить Jupiter цену при открытии» — для live сделать **обязательным**: запись в журнал фактических `executedPriceUsd`, `quotePriceImpactPct`, `slippageBpsUsed`, `priorityFeeMicrolamports` на каждое исполнение.

## 6. Паритет со стратегией Oscar (бумажной)

- Те же `PAPER_POST_*`, `PAPER_DIP_*`, DCA/TP/ladder/trail/timeout, safety, price verify, impulse, liq-watch — в виде **копии ENV** для live или общего импортируемого профиля.
- Отличия только: блок §3, исполнение §4, события JSONL live (поля `execution`, `txSignature`, `simulationCu`).
- Запрещено: «вручную» ослаблять фильтры только на live без commit в git.

## 7. Контракт JSONL (расширения)

Минимальные новые поля / виды событий (точная схема — на этапе реализации + bump MAJOR/MINOR по [`RELEASE_OPERATING_MODEL.md`](../release/RELEASE_OPERATING_MODEL.md)):

- `execution_attempt`: intent (buy/sell), mint, intendedUsd, quote summary, mode (`dry_run` | `simulate` | `live`).
- `execution_result`: signature или `simulated: true`, ошибка, фактические amounts.
- `risk_block`: какой лимит сработал.

Replay открытых позиций обязан учитывать **фактические** остатки токенов с цепи при старте (reconcile) — отдельный подпункт реализации (если JSONL и chain расходятся после сбоя).

## 8. Режимы dry-run и оценка slippage / priority fee / попадания в цену

### 8.1 `LIVE_EXECUTION_MODE=dry_run`

- Выполнять quote + swap API → получить **неподписанную** или подписанную tx **без** `sendTransaction`.
- Логировать: маршрут, `priceImpactPct`, ожидаемый outAmount, оценку priority fee из кэша/тикера.
- Сравнить с «целевой» ценой стратегии: попал ли mid/executable в коридор **как если бы** исполнили сейчас.

### 8.2 `LIVE_EXECUTION_MODE=simulate`

- Подписать tx и вызвать **`simulateTransaction`** на QuickNode.
- Записать consumption CU, ошибки программы, изменение балансов из meta.
- Не отправлять в сеть. Использовать для регрессионных тестов и дневных лимитов «стресса» без капитала.

### 8.3 `LIVE_EXECUTION_MODE=live`

- Полный путь: simulate (опционально, configurable) → send → confirm.
- Жёсткий slippage cap из ENV; при деградации quote — отмена.

### 8.4 Оценка попадания в цену при продаже/покупке

Для каждой попытки:

1. `targetPriceUsd` из логики трекера (от средней входа и уровня ladder).
2. `executablePriceUsd` из quote (out/in с учётом decimals и SOL USD).
3. Условие прохода:  
   `abs(executablePriceUsd - targetPriceUsd) / targetPriceUsd <= LIVE_TP_FILL_TOLERANCE_PCT`  
   (для покупки — аналогично относительно лимита входа).

Если не проходит — событие `execution_skip` с причиной `price_not_in_band`.

## 9. Пошаговый rollout (обязательный)

1. **Только `dry_run`**, неделя наблюдения: сравнение с paper Oscar по тем же mint (без ключа).
2. **`simulate`** на микро-суммах кошелька (SOL только на комиссии).
3. **`live`** с `LIVE_MAX_POSITION_USD` минимальным, `LIVE_MAX_OPEN_POSITIONS=1`, жёсткий дневной loss cap.
4. Расширение лимитов только после отчёта и bump версии.

## 10. Acceptance criteria (измеримые)

- [ ] PM2-процесс `live-oscar` стартует при `LIVE_STRATEGY_ENABLED=0` без подписи транзакций.
- [ ] При срабатывании `LIVE_MAX_DAILY_LOSS_USD` новые `open` не вызывают swap API (тест с подставным счётчиком).
- [ ] Каждый `live` swap имеет запись в JSONL с signature или явной ошибкой симуляции.
- [ ] Перед swap всегда есть Jupiter quote не старше `LIVE_QUOTE_MAX_AGE_MS` (новый ENV).
- [ ] `npm run typecheck` и релевантные тесты проходят; документация и `CHANGELOG` обновлены при merge.

---

## 11. Forbidden actions

- Хранить `LIVE_WALLET_SECRET` в репозитории или в dashboard-статике.
- Использовать один и тот же `PAPER_TRADES_PATH` для live.
- Отправлять транзакцию без проверки лимитов §3.3.
- Полагаться только на цену из PG без свежего quote для live-исполнения.
