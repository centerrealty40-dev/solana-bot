# Spec W7.6 — Impulse Confirm Entry Path (PG-сигнал → QuickNode on-chain → согласование с Jupiter)

**Продукт:** `solana-alpha` (paper trader, post-lane / dip).  
**Статус:** норматив по **контракту и intent**; **реализация** — в `src/papertrader/pricing/impulse-confirm.ts` (+ `orca-whirlpool-spot.ts`), интеграция в `src/papertrader/main.ts`, конфиги `PAPER_IMPULSE_*` / `PAPER_ENTRY_IMPULSE_PG_BYPASS_DIP`. С этим документом расхождения по DEX-веткам: ончейн layout реализован **только** для **Orca Whirlpool**; для остальных источников — Jupiter-коридор (см. `impulseAllowJupiterOnlyUnsupported`).  
**Связанные спеки:** W6.3b (discovery), W6.3c (executor), W7.2 (safety), W7.4 (Jupiter pre-entry verify), W7.5 (liq watch).  
**Цель документа:** зафиксировать полный пошаговый поток исполнения, граничные случаи, политики отказа и наблюдаемость для контура «импульсный триггер по дельте снапшотов + быстрое подтверждение цены через платный QuickNode + вторичная проверка Jupiter».

---

## 1. Зачем этот контур

### 1.1 Проблема

- Дискавери и дип-гейт по умолчанию используют **агрегаты по окнам** (например 120–720 мин) над `*_pair_snapshots` и минутную сетку записи коллекторов.
- Пользователь хочет **дополнительный, редкий** путь: если по **последним двум точкам в PG** виден **резкий сдвиг цены**, инициировать **быстрое уточнение «живой» цены** (QuickNode RPC / on-chain состояние пула) и принять решение о входе при **согласованности** с допустимым коридором относительно цены снимка.

### 1.2 Цели

1. **Не заменять** текущий дип-детектор целиком, а **добавить опциональный** «контур B» после узкого триггера.
2. Использовать **платный QuickNode** **точечно** (редкие RPC после фильтров), с **жёсткими бюджетами** и деградацией.
3. Сохранить **аудит**: каждая ветка решения должна быть восстановима из логов / JSONL.

### 1.3 Не-цели (явно вне scope первой версии)

- Постоянный WS/stream по всем парам.
- Гарантия исполнения live-сделки (это paper).
- Полная поддержка всех экзотических типов пулов без отдельного исследования layout аккаунтов.
- Автоматический подбор размера позиции от волатильности (можно позже).

## 2. Термины

| Термин | Значение |
|--------|----------|
| **Контур A** | Текущий пайплайн: коллекторы → PG → dip / фильтры → open. |
| **Контур B** | Impulse Confirm Path: триггер по дельте PG → RPC → согласование → решение. |
| **Снимок (snapshot row)** | Строка в `raydium_pair_snapshots` / `meteora_pair_snapshots` / `orca_pair_snapshots` / `moonshot_pair_snapshots` с полем `price_usd`, `ts`, `base_mint`, `pair_address`, и т.д. |
| **Якорная цена снимка** | Значение `price_usd` у **более новой** из двух сравниваемых строк (или явно заданное правило — см. §5.2). |
| **Дельта PG** | Относительное изменение цены между двумя последовательными по времени снимками **одного** `(base_mint, pair_address, source)` или между «предпоследним и последним» в окне. |
| **Live spot (on-chain)** | Цена базового актива к quote/USD, выведенная из **реального состояния аккаунтов пула** на момент RPC-чтения (метод зависит от DEX). |
| **Jupiter reference** | Цена из quote API (как W7.4), используемая как **вторая линия** согласования и проверки исполнимости. |
| **Коридор** | Допустимое отклонение live/Jupiter относительно якорной цены снимка (напр. выше не более +30%, ниже не более −70% — числа конфигурируемые). |
| **Single-flight** | Не более одного активного запроса контура B на ключ `(strategyId, mint, pair_address)` в окне TTL. |

## 3. Архитектура на высоком уровне

```
Коллекторы (≈60s) ──► *_pair_snapshots (PG)
                              │
                              ▼
              ┌───────────────────────────────┐
              │ Контур A: discovery + dip…    │──► eval / eval-skip-open / open
              └───────────────────────────────┘
                              │
                              │  если включён IMPULSE_PATH и выполнен триггер §5
                              ▼
              ┌───────────────────────────────┐
              │ Контур B: Impulse Confirm      │
              │ 5.1 триггер по дельте PG       │
              │ 5.2 admission (бюджет/lock)    │
              │ 5.3 RPC on-chain spot          │
              │ 5.4 Jupiter reference          │
              │ 5.5 коридор + политика AND/OR   │
              │ 5.6 handoff в существующий open│
              └───────────────────────────────┘
```

**Принцип:** контур B **никогда** не опрашивает RPC для всего универса кандидатов; только для записей, прошедших **дешёвые** фильтры и **узкий** триггер.

## 4. Предусловия и входные данные

Перед активацией контура B для конкретной строки кандидата (как в текущем discovery) должны быть известны:

| Поле | Обязательность | Назначение |
|------|----------------|------------|
| `mint` (base_mint) | да | Идентификация токена. |
| `source` ∈ {raydium, meteora, orca, moonshot} | да | Выбор таблицы снапшотов и RPC-маршрута. |
| `pair_address` | да для RPC | Адрес пула / программного аккаунта (см. §8 — уточнение типа пула). |
| `lane`, `features`, фильтры поста | да | Должны уже пройти обычные гейты стратегии (или явное исключение в конфиге). |
| `price_usd` текущего кандидата | да | Точка «сейчас» в логике дискавери (совпадает с последним снапшотом или derived). |

Если `pair_address` отсутствует — контур B **не запускается** (reason `impulse:no_pair_address`).

## 5. Пошаговое исполнение контура B

Ниже шаги **строго упорядочены**. На каждом шаге возможен **ранний выход** с фиксированным `reason`.

### 5.0 Фиче-флаг и конфигурация

- Глобально (per strategy): `IMPULSE_CONFIRM_ENABLED` (предлагаемое имя env, финальное — при имплементации).
- Если выключено — контур B отсутствует, только контур A.

### 5.1 Триггер по дельте двух PG-снимков

**Источник данных:** таблица `*_pair_snapshots`, соответствующая `source`.

**Выбор двух точек (рекомендуемая семантика v1):**

1. Взять **две последние** строки для ключа `(base_mint = mint AND pair_address = :pair)` упорядоченные по `ts DESC`, т.е. `S_new`, `S_prev`.
2. Требование свежести:  
   `now() - S_new.ts` ∈ `[IMPULSE_PG_MAX_AGE_SEC_MIN, IMPULSE_PG_MAX_AGE_SEC_MAX]`  
   (например мин 10 с, макс 120 с — числа **конфигурируемые**; дефолты подбираются после наблюдений).
3. Вычислить относительную дельту (унифицировать направление «пролив»):  
   \[
   \delta_{pg} = \frac{P_{new} - P_{prev}}{P_{prev}} \cdot 100\%
   \]
   где \(P\) = `price_usd` (> 0, иначе триггер не срабатывает).

**Условие триггера (пример из обсуждения пользователя, параметризуется):**

- Если разрабатываем «ловлю пролива», триггер задаётся как **отрицательная** дельта:  
  `delta_pg <= -IMPULSE_PG_MIN_DROP_PCT` (например ≤ −5%).
- Если нужен «любой импульс», возможен модуль `abs(delta_pg) >= threshold` — **отдельный режим**, по умолчанию выкл.

**Нюансы:**

- При одном снимке в окне — триггер **не срабатывает**.
- Если коллектор пишет раз в 60 с, **реальная временная база** между `S_prev` и `S_new` часто ≈ 60 с; окно «10 с–1 мин» тогда относится к **возрасту свежего снимка** относительно wall-clock, а не к межснимочному интервалу. Это нужно явно документировать в §11 (ограничения).
- Дубликаты `ts` или одинаковый `price_usd`: если дельта 0 — триггер не срабатывает.

**Выход:** структура `ImpulseTriggerContext`:

```json
{
  "kind": "impulse_trigger",
  "mint": "...",
  "pair_address": "...",
  "source": "raydium",
  "ts_prev": "...",
  "ts_new": "...",
  "price_prev": 0.012,
  "price_new": 0.011,
  "delta_pg_pct": -8.33,
  "anchor_snapshot_price_usd": 0.011,
  "triggered_at_wall_ms": 1735689600123
}
```

Если триггер не прошёл — контура B нет; продолжается только контур A.

### 5.2 Admission control (обязательно до любого RPC)

Цели: не убить QuickNode и не создать гонки.

1. **Глобальный rate limit** per strategy process: не более `IMPULSE_RPC_MAX_PER_MIN` успешных **попыток** контура B (или только «RPC round-trip» — уточнить при имплементации).
2. **Monthly/daily budget** (опционально интеграция с `qn-feature-usage`: новый feature key `impulse_confirm`).
3. **Single-flight lock:** ключ `(strategyId, mint, pair_address)` — если lock активен (< `IMPULSE_SINGLE_FLIGHT_MS`), шаг завершается с `impulse:busy`.
4. **Cooldown после прошлого триггера** по тому же mint (опционально): `IMPULSE_MINT_COOLDOWN_SEC`.

При любом отказе admission — **не** вызывать RPC; зафиксировать `eval`-подобное событие или расширенный лог (§10).

### 5.3 Запрос live spot через QuickNode (on-chain)

**После успешного admission:**

1. Разрешить `pair_address` → **тип пула** и набор аккаунтов для чтения.  
   - Если тип неизвестен или не поддерживается v1 — `impulse:pool_layout_unsupported` и **выход** (или fallback только Jupiter — политика конфигом `IMPULSE_ONCHAIN_FALLBACK=jupiter_only`).
2. Выполнить **минимально необходимый** RPC:
   - Предпочтительно **один** `getMultipleAccounts` на список pubkey (vault reserves + при необходимости config), либо два вызова если лимиты размера.
3. Декодировать данные программы DEX → **spot price** в тех же единицах, что и `price_usd` в PG (USD/token или quote/token с последующей конвертацией через известный quote mint).

**Таймаут RPC:** `IMPULSE_RPC_TIMEOUT_MS` (например 2500–4000 мс).

**Ошибки:**

- Timeout / 429 / 5xx → один **retry** с jitter backoff `IMPULSE_RPC_RETRY_MS`; после второго фейла → `impulse:rpc_failed`.
- Невалидные данные (нулевая ликвидность, пустые vault) → `impulse:onchain_invalid_state`.

**Результат:** `onchainSpotUsd` (или пара `(spot_raw, quote)` + отдельный FX к USD).

### 5.4 Вторичная проверка Jupiter (quote API)

**Цель:** исполнимость и кросс-проверка; не обязана совпадать с on-chain пипс-в-пипс.

1. Вызвать тот же класс quote, что W7.4 (`verifyEntryPrice` или общий клиент), с параметрами стратегии по slippage/impact.
2. Получить `jupiterPriceUsd` при `kind === 'ok'`, иначе зафиксировать `skipped/blocked` как **reference unavailable**.

**Политика при недоступности Jupiter:**

- `IMPULSE_REQUIRE_JUPITER=1` → без Jupiter **не входим** (`impulse:jupiter_required_failed`).
- `=0` → только on-chain + коридор (рискованнее).

### 5.5 Коридор относительно якорной цены снимка

**Якорь:** `anchor_snapshot_price_usd` = `S_new.price_usd` (рекомендация v1: всегда **новейший** снимок из пары).

**Коридор (конфиг):**

- `IMPULSE_MAX_UP_PCT_FROM_ANCHOR` (например 30) — live spot не должен быть выше якоря более чем на это значение:
  \[
  \frac{spot - anchor}{anchor} \cdot 100 \le MAX\_UP
  \]
- `IMPULSE_MAX_DOWN_PCT_FROM_ANCHOR` (например 70) — не ниже чем на 70% (т.е. допускается большое падение относительно якоря):  
  \[
  \frac{spot - anchor}{anchor} \cdot 100 \ge -MAX\_DOWN
  \]
  (знак и семантику зафиксировать в коде один раз; пользователь имел в виду «не хуже −70% от снимка».)

**Согласование on-chain vs Jupiter (рекомендуемая политика v1 «надёжный режим»):**

- Оба доступны: требовать **оба** внутри коридора **и** взаимное отклонение не более `IMPULSE_MAX_DISAGREE_PCT` (например 3–8%).
- Только on-chain: допускается только если `IMPULSE_ALLOW_ONCHAIN_ONLY=1`.

При нарушении — `impulse:corridor_fail` с деталями `{ anchor, onchain, jupiter, deltas... }`.

### 5.6 Сводка с основным дип-гейтом (политика слияния)

Варианты (выбирается конфигом `IMPULSE_DIP_POLICY`):

| Режим | Поведение |
|-------|-----------|
| `shadow` | Только логировать результат контура B; вход определяет только контур A. |
| `boost` | Если контур B **pass**, можно **ослабить** один параметр дипа (опасно — не рекомендуется в v1). |
| `parallel_or` | Достаточно **либо** классический дип **либо** impulse-pass (строгие лимиты на частоту). |
| `parallel_and` | Требуются **оба** (максимальная фильтрация). |

**Рекомендация для v1:** `parallel_and` или оставить контур A обязательным, а контур B как **дополнительный обязательный** только когда включён отдельный «режим стратегии» — чтобы не ломать текущие pt1-* без явного переключения.

### 5.7 Взаимодействие с W7.4 (порядок вызовов)

Рекомендуемый порядок перед `open`:

1. Safety (W7.2), holders, обычные фильтры.
2. **Классический дип** (контур A) — если политика требует.
3. Если включён impulse и триггер сработал → **контур B** (RPC + коридор + Jupiter политика).
4. **W7.4 `verifyEntryPrice`** как сейчас — финальная проверка исполнимости (или объединить с шагом 5.4, если те же данные — см. §12).

Избегать **двойного** идентичного Jupiter вызова: кешировать результат quote на TTL 1–3 с в рамках одного mint для одного тика.

### 5.8 Принятие решения и запись

- При полном успехе: передать в существующий `makeOpenTradeFromEntry` **effective price**:
  - по умолчанию якорь = `onchainSpotUsd` если прошёл коридор и политика `ENTRY_PRICE_SOURCE=onchain`;
  - или `jupiterPriceUsd` если так задано;
  - **не** использовать сырой PG `price_usd` как единственный источник после контура B.
- Записать в JSONL расширение поля (новый объект) `impulseConfirm: { ... }` на событии `open` и дублировать краткий итог на `eval` если отказ.

## 6. Нюансы по DEX (уровень спеки, без полного layout)

Реализация обязана иметь **таблицу маршрутизации** `source → decoder`. Ниже — что должна содержать каждая строка.

| source | Объект RPC | Примечания |
|--------|------------|------------|
| raydium | pool / vault accounts зависят от продукта (CPMM vs legacy AMM v4) | Версия v1: поддержать только тот тип, который реально приходит в `pair_address` из коллектора; иначе unsupported. |
| meteora | DLMM / pools — layout из программы | То же: только верифицированный путь. |
| orca | whirlpool PDA | Обычно два vault + sqrt price; цена из sqrtPriceX64. |
| moonshot | программа moonshot | Зависит от данных коллектора — если `pair_address` не однозначен, блокировать v1. |

**Правило безопасности:** при малейшем сомнении в декодировании — `impulse:decode_error` и skip.

## 8. Диаграмма состояний (текстом)

```
DISABLED → [контур A только]

ENABLED:
  NO_TRIGGER → контур A
  TRIGGER → ADMISSION_FAIL → skip B + reason
  TRIGGER → ADMISSION_OK → RPC_FAIL → skip open / fallback policy
  TRIGGER → ADMISSION_OK → RPC_OK → JUPITER_FAIL → skip if required
  RPC_OK + JUPITER_OK → CORRIDOR_FAIL → skip
  CORRIDOR_OK → DIP_POLICY_FAIL → skip
  DIP_POLICY_OK → W7.4_FAIL → skip (если block)
  ALL_OK → OPEN (+ impulseConfirm stamp)
```

## 9. Наблюдаемость и аудит

### 9.1 JSONL события

- Новое опциональное поле на `kind: 'open'`: `impulseConfirm: { verdict, trigger, onchain, jupiter, corridor, timings_ms }`.
- При отказе до open: `kind: 'eval-skip-open'` с `reason: impulse:*` (список фиксированных кодов из §5).

### 9.2 Метрики (heartbeat / stats)

- `impulse_trigger_total`, `impulse_rpc_success_total`, `impulse_open_total`, `impulse_skip_total` по reason.

### 9.3 Логи

Структурированный лог на каждом шаге с `mint`, `pair`, `latency`, `credits_estimate` (если интеграция с meter).

## 11. Ограничения и честные оговорки

1. **Минута между снапшотами коллектора** ограничивает то, какие движения вообще попадают в PG-дельту; триггер по PG не является детектором субсекундных событий без учащения записи или вторичного источника.
2. **On-chain spot** отражает пул; **Jupiter** может использовать другой маршрут — поэтому порог `MAX_DISAGREE_PCT` обязателен в «надёжном» режиме.
3. **Экстремальные движения** могут пройти коридор −70% и всё ещё быть «валидными» по формуле — это сознательный риск; можно добавить минимум ликвидности и W7.4 impact как дополнительный потолок.

## 13. Критерии приёмки (когда будете реализовывать)

1. При включённом флаге и синтетических фикстурах PG двух точек триггер вызывает ровно один RPC round-trip на mint (single-flight).
2. При 429 от RPC процесс не зависает; соблюдается backoff и лимит попыток.
3. Любой отказ даёт стабильный `reason` из §10 и попадает в JSONL.
4. Нет регрессии контур A при `ENABLED=0`.
5. QuickNode credit usage за сутки в пределах заданного бюджета на стресс-сценарии (интеграционный тест optional).
