# Спецификация — TP-ladder: доля от **текущего остатка позиции** (mark-to-market)

**Продукт:** Solana Alpha — paper trader (`pt1-diprunner`, `pt1-oscar`, `pt1-dno` и любые стратегии с `PAPER_TP_LADDER`).  
**Статус:** нормативная спецификация поведения + план внедрения/верификации.  
**Связано:** W6.3c executor (`tracker.ts`), JSONL `partial_sell`, дашборд (`dashboard-server.ts`).

---

## 1. Бизнес-смысл (требование)

На каждой ступени TP-ladder параметр `sellFraction` (из `PAPER_TP_LADDER`, формат `pnlPct:sellFraction`) означает:

> **Долю от позиции, которая есть *на момент срабатывания ступени*, оценённую по текущей рыночной метрике** (mcap или цена — как у стратегии), с учётом уже произошедших частичных продаж.

Пример (идеализированно):

- Вход: **$100** по метрике entry.
- При срабатывании ступени **+10%** рыночная стоимость **всего ещё открытого остатка** ≈ **$110**.
- Если `sellFraction = 0.3`, то сделка продаёт **≈ 30% × $110 ≈ $33** выручки (gross/net — по модели costs), а не **30% × $100** от изначального входа.

То же правило применяется **ко всем последующим ступеням**: каждый раз база — **остаток после предыдущих продаж**, по **текущей** цене/mcap на тик трекера.

**Стратегии:** одна и та же семантика для всех трёх (и любых будущих) — общий код в `executor/tracker.ts`, конфиг только задаёт строку лестницы.

## 2. Формализация в терминах модели позиции

В коде позиция описана полями `OpenTrade`:

- `totalInvestedUsd` — суммарный внесённый капитал (open + DCA), **не уменьшается** частичными продажами.
- `remainingFraction` ∈ (0, 1] — доля **изначального эквивалента позиции**, котначало бы соответствовать `totalInvestedUsd`, которая **ещё не закрыта** частичными TP (и финальным close).
- `avgEntry`, `avgEntryMarket` — средневзвешенные цены входа (effective / market).

**Стоимость остатка (mark-to-market), net после модели выходных издержек на текущей метрике:**

\[
V_{\mathrm{rem,net}} = \texttt{totalInvestedUsd} \times \texttt{remainingFraction} \times (\texttt{effectiveSell} / \texttt{avgEntry})
\]

где `effectiveSell` — цена выхода после `applyExitCosts` для **данного** размера сделки (см. §4).

**Выручка частичной продажи:**

\[
\texttt{proceedsUsd} = V_{\mathrm{rem,net}} \times \texttt{sellFraction}
\]

**Бухгалтерская себестоимость проданного куска (для PnL на slice):**

\[
\texttt{investedSoldUsd} = \texttt{totalInvestedUsd} \times \texttt{remainingFraction} \times \texttt{sellFraction}
\]

\[
\texttt{pnlUsd} = \texttt{proceedsUsd} - \texttt{investedSoldUsd}
\]

**Обновление остатка:**

\[
\texttt{remainingFraction} \leftarrow \texttt{remainingFraction} \times (1 - \texttt{sellFraction})
\]

Это эквивалентно «продали `sellFraction` от текущего количества/экспозиции» при линейной модели «цена единообразна по всему остатку».

## 3. Аудит текущей реализации (факт по коду)

Файл `src/papertrader/executor/tracker.ts`, блок TP-ladder:

- Вычисляет `investedSoldUsd`, затем `applyExitCosts(..., investedSoldUsd)` для оценки **effective** цены выхода.
- Считает `remainingValueNet = totalInvestedUsd * remainingFraction * (effectiveSell / avgEntry)`.
- Задаёт `proceedsUsd = remainingValueNet * sellFraction`.
- Обновляет `remainingFraction *= 1 - sellFraction`.

**Вывод:** при непротиворечивых `avgEntry` / метрике трекера движок **уже соответствует** требованию §1: `sellFraction` применяется к **текущему остатку**, оценённому по **текущей** цене на тик.

Дубликат той же логики: `src/scripts/paper2-strategy-backtest.ts` — держать синхронно с `tracker.ts`.

## 4. Нюансы и известные краевые случаи

### 4.1 Порядок вызова `applyExitCosts` и размер сделки для slippage

Сейчас в `applyExitCosts` передаётся `tradeUsd = investedSoldUsd` (себестоимость куска), а не ожидаемая выручка. Для кривой ликвидности это слегка **занижает** оценку размера сделки в USD в прибыльных сделках.

**Рекомендация (minor):** для partial sell передавать в `sideCostBps`/`applyExitCosts` **оценку notional в USD** ближе к `proceedsUsd` (или к среднему между cost и gross proceeds), либо итерацию на 1 шаг — по желанию accuracy.

### 4.2 DCA после частичных TP

После `dca_add` в `tracker.ts` выполняется `ot.remainingFraction = 1`. Если до DCA уже были partial TP, это **ломает** учёт доли остатка относительно «старых» продаж (остаток переопределяется как 100% новой объединённой позиции в терминах fraction, хотя `totalInvestedUsd` и legs отражают DCA).

**Рекомендация:** отдельная задача: при DCA **не** сбрасывать `remainingFraction` в `1` вслепую; пересчитать долю относительно новой базы или явно моделировать количество токенов. До исправления зафиксировать в README: «не рекомендуется комбинировать агрессивный TP-ladder с DCA на той же позиции без ревью».

### 4.3 Восстановление из JSONL

Убедиться, что `store-restore.ts` восстанавливает `remainingFraction` и `ladderUsedLevels` согласованно после рестарта — иначе повторное применение лестницы может теоретически расходиться.

## 5. Наблюдаемость и UX (где «кажется, что баг»)

### 5.1 Дашборд / таймлайн

В `scripts-tmp/dashboard-server.ts` для `partial_sell` в подписи используется **себестоимость** проданного куска (`soldCostUsd = proceedsUsd - pnlUsd = investedSoldUsd`), с текстом вроде «продано $X (cost)».

Для пользователя ожидаемо видеть **~30% от ~$110 ≈ $33** выручки, а не **$30** cost.

**Обязательные правки UX (рекомендуется как часть этой инициативы):**

- В таймлайне показывать **обе** величины или основную — **gross/net выручку**: например `proceedsUsd` из события JSONL и опционально cost.
- Текст: «sell 30% of **remaining notional**» + «≈ $Y proceeds (net)» / «cost basis sold $Z».

### 5.2 Расширение JSONL `partial_sell` (опционально, для аудита)

Добавить поля (обратно совместимо):

- `remainingMtmGrossUsd` — до сделки: `totalInvestedUsd * remainingFraction * (marketSell / avgEntryMarket)`.
- `remainingMtmNetUsd` — после оценки effective цены, до умножения на sellFraction (или зеркально).
- `soldProceedsUsd` — дубликат `proceedsUsd` с явным именем.

Это снимает споры при разборе сделок вне дашборда.

## 6. Критерии приёмки

1. **Математика:** для синтетического тика с фиксированными `totalInvestedUsd`, `remainingFraction`, `avgEntry`, `marketSell` и одной ступенью лестницы — `proceedsUsd / (totalInvestedUsd * remainingFraction * (marketSell/avgEntryMarket))` ≈ `sellFraction` (с допуском на fees).
2. **Последовательные ступени:** после первой продажи вторая ступень использует **уменьшенный** `remainingFraction` и **новую** цену.
3. **Паритет:** одинаковый сценарий в `paper2-strategy-backtest.ts` и live `tracker.ts` даёт те же `proceedsUsd` / `remainingFraction` (при тех же входных ценах).
4. **UI:** таймлайн не вводит в заблуждение «cost» как единственную метрику размера продажи.
5. **Регрессия:** `ladder_retrace` и `trailMode` не ломаются (юнит-тесты на `ladderRetraceTriggered` + новые кейсы на partial sell).
