# Spec W8.0-p5 — Исполняемые гейты риска и капитала (live-oscar, Phase 5)

**Статус:** нормативный черновик для реализации (**до merge кода — ревью + bump `docs/strategy/release/VERSION` по [`RELEASE_OPERATING_MODEL.md`](../release/RELEASE_OPERATING_MODEL.md)**).  
**Продукт:** `solana-alpha`, процесс **`live-oscar`**.  
**Зависит от:**

- [`W8.0_live_oscar_trading_bot.md`](./W8.0_live_oscar_trading_bot.md) — **§3.3** жёсткие лимиты риска, **§3.4** освобождение капитала (**2X**), **§5** свежий quote для MTM;
- [`W8.0_phase1_live_jsonl_contract.md`](./W8.0_phase1_live_jsonl_contract.md) — `liveSchema: 1`, виды **`risk_block`**, **`capital_skip`**, **`capital_rotate_close`** ([`events.ts`](../../../src/live/events.ts));
- [`W8.0_phase4_oscar_parity_adapter_spec.md`](./W8.0_phase4_oscar_parity_adapter_spec.md) — adapter после стратегии; **не менять** семантику `LiveExecutionIntent` / порядок Oscar-гейтов (**P4-I2**);
- Реализация Phase 4: [`phase4-execution.ts`](../../../src/live/phase4-execution.ts), [`main.ts`](../../../src/papertrader/main.ts) (`liveOscar` bundle).

**Не входит в Phase 5:**

- **`sendTransaction`** и подтверждение в сети — **Phase 6** (ниже — как вести §3.4 при **`simulate`** до Phase 6).
- Полный **replay / reconcile** журнала с ончейн-остатками — **Phase 7**.
- Изменение логики discovery/tracker paper (**Oscar**) кроме точек расширения, уже согласованных с Phase 4.

---

## 1. Цели Phase 5

1. Ввести **единый слой** между «решение стратегии разрешило вход/DCA/выход» и вызовом execution adapter Phase 4, который **исполняет** нормы родительской спеки **§3.3–§3.4**.
2. При срабатывании лимита — **детерминированные** записи в live JSONL (**`risk_block`**, **`capital_skip`**) без вызова Jupiter по этому intent (кроме случаев, где MTM **явно** требует quote по §5).
3. Обеспечить **одинаковый порядок** проверок на каждом тике (см. §6), чтобы поведение было тестируемым и совместимым с Phase 6 (тот же порядок перед `send`).
4. В режиме **`LIVE_EXECUTION_MODE=simulate`**: «ротация капитала» из §3.4 трактуется как **симулируемое** полное закрытие выбранной позиции (цепочка Phase 4 sell) + **пересчёт `free_usd`** по данным RPC кошелька (§4.2), без ожидания on-chain подтверждения (**до Phase 6**).

## 3. Объём: переменные окружения (канон из родителя + уточнения)

Минимальный набор (расширение Zod в [`config.ts`](../../../src/live/config.ts)):

### 3.1 Риск §3.3

| Переменная | Тип / семантика | Примечание для Phase 5 |
|------------|-----------------|-------------------------|
| `LIVE_MAX_POSITION_USD` | положительное число; потолок нотации **нового** входа / DCA add | Сравнение с `intendedUsd` **перед** adapter; при превышении — **`risk_block`** или жёсткий clamp — **выбрать один** в реализации и зафиксировать в CHANGELOG. |
| `LIVE_MAX_OPEN_POSITIONS` | целое ≥ 1 | Сравнение с `open.size` в памяти live-oscar (**после** restore Phase 7 — с восстановленным множеством). |
| `LIVE_MAX_STRATEGY_LOSS_USD` | положительное число; порог **совокупного** убытка (realized + unrealized) | Блок **новых** входов и **новых** DCA при нарушении; закрытия по трекеру **не** блокировать, кроме явного флага ниже. |
| `LIVE_KILL_AFTER_CONSEC_FAIL` | целое ≥ 1 или `0` = выкл. | В **`simulate`**: счётчик увеличивается на **каждый** терминальный **`execution_result`** со статусом **`sim_err`** по **обязательному** intent (buy/sell по решению стратегии). В **`live`** (Phase 6+): только по неуспешной **отправке** / финальному отказу — уточнить в Phase 6 спеке; Phase 5 задаёт **минимум** ветку для `simulate`. |
| `LIVE_HALT_CLOSE_ALL_ON_MAX_LOSS` | bool | При `1` и нарушении лимита убытка — инициировать сценарий закрытия всех позиций (**осторожно**); в **`simulate`** — последовательность **sim** sell по каждой открытой позиции + JSONL (**без send**). |

Опционально (как в родителе): `LIVE_MIN_WALLET_SOL` — блок новых входов при низком SOL.

### 3.2 Капитал §3.4

| Переменная | Смысл |
|------------|--------|
| `LIVE_ENTRY_NOTIONAL_USD` | Если задано — **X** для порога свободных средств; иначе **X = LIVE_MAX_POSITION_USD** (или `paperCfg.positionUsd` только для логов — **не** смешивать SSOT; выбрать одно в PR). |

## 4. Вычисление `free_usd` (v1, SOL-only)

**Обязательное минимальное определение для merge Phase 5:**

1. Прочитать **баланс lamports** нативного SOL кошелька (`getBalance` / аналог через существующий RPC-контур).
2. Вычесть **буфер** комиссий + rent (ENV: например `LIVE_FREE_SOL_BUFFER_LAMPORTS` или доля от баланса — зафиксировать в PR).
3. `free_usd = max(0, solAvailable) * solUsd`, где `solUsd` — тот же источник, что и у paper pricing (`getSolUsd()` / refresh), с понятной политикой устаревания (если цена stale — **`risk_block`** или отложить вход — выбрать и описать).

**USDC/USDT** как вход — вне v1 Phase 5 (оставить hook в спеке на Phase 6+).

## 5. Учёт PnL для `LIVE_MAX_STRATEGY_LOSS_USD`

**Realized:** сумма **net** PnL по событиям закрытия, доступным процессу live (**Phase 7** уточнит reconcile); в Phase 5 допускается **in-memory** накопление после старта процесса + опциональное чтение хвоста JSONL при старте (только live-журнал, **не** paper).

**Unrealized:** по каждому `OpenTrade` в памяти — MTM в USD с использованием **свежего** ценового источника (Jupiter quote или эквивалент по §5 родителя). Частота: на том же тике, где принимается решение о новом входе.

**Базовый ноль:** зафиксировать в PR одно из: (a) старт процесса = ноль; (b) событие `equity_baseline` при первом запуске (аддитивно к `liveSchema`, если понадобится — отдельное решение в Phase 5 PR).

## 6. Порядок гейтов (канонический)

Для **нового входа** (`buy_open` после Oscar-гейтов):

1. **Oscar** конвейер (**P4-I2**) — без изменений.  
2. **§3.3:** счётчик консекутивных фейлов; лимит открытых позиций; потолок убытка стратегии; (опц.) минимальный SOL; потолок нотации.  
3. **§3.4:** расчёт `free_usd`; если `free_usd < k·X` — политика ротации (§7).  
4. **Phase 4 adapter** — quote → sign → simulate (или send в Phase 6).

Для **DCA add** (`dca_add`): шаги **2–4** с тем же **X**, что и для добавляемой нотионали (или отдельный потолок — зафиксировать; по умолчанию **тот же** `LIVE_MAX_POSITION_USD` на **долю** DCA).

Для **принудительного** закрытия по ротации капитала: **не** проходят лимиты «нового входа», но проходят проверки, что закрытие **разрешено** стратегией (полное закрытие одной позиции).

---

## 7. Политика ротации капитала (§3.4) в `simulate`

При **`free_usd < k·X`** и сигнале на **новый** вход:

1. Если **нет** открытых позиций → **`capital_skip`** (`insufficient_free_balance_no_positions`); adapter buy **не** вызывать.  
2. Если нет позиции с **строго** положительным unrealized PnL (USD) → **`capital_skip`** (`no_profitable_position_to_close`).  
3. Иначе выбрать позицию с **максимальным** unrealized PnL USD (tie-break: **больший % от входа**, затем **более старая** по `entryTs` — зафиксировать в коде).  
4. Вызвать **Phase 4** полное закрытие (**sell_full**) для этой позиции — только симуляция.  
5. После **`sim_ok`** — пересчитать `free_usd` (§4). Если всё ещё `< k·X` и **`LIVE_CAPITAL_ROTATE_CASCADE=1`** — повторить с шага 3; иначе **`capital_skip`** (`insufficient_free_after_rotation`).  
6. Если после успешной ротации `free_usd ≥ k·X` — разрешить переход к adapter для **исходного** входа.

События: перед симуляцией ротации — осмысленная запись **`capital_rotate_close`** (mint, оценка unrealized, **без** signature в `simulate`; поле `txSignature: null`). После Phase 6 — дополнить signature в спеке Phase 6.

## 8. JSONL (аддитивно)

Использовать существующие виды:

- **`risk_block`**: `limit` — короткий код (`max_open_positions`, `max_strategy_loss`, `consec_sim_fail`, `min_wallet_sol`, …); `detail` — без секретов.  
- **`capital_skip`**: `reason` из набора §3.4 / §7; опционально `freeUsdEstimate`, `requiredFreeUsd`.  
- **`capital_rotate_close`**: `mint`, `unrealizedPnlUsd`, `txSignature` (null в simulate).

При необходимости нового `kind` — обновить W8.0-p1 и bump **`liveSchema`** (нежелательно в Phase 5; предпочесть поля в `detail`).

## 9. Acceptance criteria (измеримые)

- [ ] При превышении **`LIVE_MAX_OPEN_POSITIONS`** новый **`buy_open`** не вызывает adapter; в JSONL есть **`risk_block`**.  
- [ ] При нарушении порога **`LIVE_MAX_STRATEGY_LOSS_USD`** новые **входы / DCA** блокируются; закрытия по трекеру продолжают работать (если не включён halt-all).  
- [ ] При **`free_usd < k·X`** и наличии прибыльной позиции — сначала фиксируется попытка ротации (**`capital_rotate_close`** + цепочка Phase 4 sell), затем либо вход, либо **`capital_skip`** по правилам §7.  
- [ ] При **`free_usd < k·X`** и отсутствии прибыльных позиций — **`capital_skip`** (`no_profitable_position_to_close` или `insufficient_free_balance_no_positions`).  
- [ ] **`npm run typecheck`**; юнит-тесты на порядок гейтов и synthetic wallet balance / MTM (моки RPC + quote).  
- [ ] **`CHANGELOG`** + bump **`VERSION`** (**MINOR** рекомендуется).

## 10. Forbidden actions

- Снижать **`LIVE_MAX_STRATEGY_LOSS_USD`** или отключать §3.3 «тихо» в прод без записи в CHANGELOG и явного решения владельца.  
- Вызывать **`sendTransaction`** из кода Phase 5 (только Phase 6).  
- Использовать paper JSONL для учёта live PnL (**P4-I1**).  
- Блокировать **аварийные** выходы по трекеру (SL / LIQ_DRAIN / …) правилами §3.3 без отдельной спеки.
