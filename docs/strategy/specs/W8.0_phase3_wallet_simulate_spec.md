# Spec W8.0-p3 — Кошелёк + подпись + `simulateTransaction` (live-oscar, Phase 3)

**Статус:** нормативный черновик для реализации (**ещё не выполнено в коде** до явного коммита и bump версии продукта).  
**Продукт:** `solana-alpha`, процесс **`live-oscar`**.  
**Зависит от:**

- [`W8.0_live_oscar_trading_bot.md`](./W8.0_live_oscar_trading_bot.md) — §3.2 кошелёк/RPC, §4.1–4.2 цепочка swap, §8.2 режим **`simulate`**;
- [`W8.0_phase1_live_jsonl_contract.md`](./W8.0_phase1_live_jsonl_contract.md) — `liveSchema: 1`, `execution_attempt` / `execution_result`, `intentId`;
- **Phase 2 (выполнено):** [`W8.0_IMPLEMENTATION_PHASES.md`](./W8.0_IMPLEMENTATION_PHASES.md) — `src/live/jupiter.ts` (quote + unsigned swap build);
- Референс по RPC: [`sim-audit.ts`](../../../src/papertrader/pricing/sim-audit.ts) (W7.8) — вызов **`qnCall('simulateTransaction', …, { feature: 'sim' })`** и разбор ответа.

**Не входит в Phase 3:** `sendTransaction`, подтверждение в сети, паритет трекера Oscar с исполнением (Phase 4), лимиты риска §3.3 как исполняемые гейты (Phase 5), полный `live` режим (Phase 6).

---

## 1. Цели Phase 3

1. **Загрузка keypair** строго по политике доступа (нет чтения секрета при «торговля выключена» или `dry_run`).
2. Построение **реальной** unsigned swap-транзакции Jupiter с **`userPublicKey` = pubkey кошелька** (не placeholder Phase 2).
3. **Подпись** транзакции ключом кошелька (локально, на процессе `live-oscar`).
4. Вызов **`simulateTransaction`** через тот же контур QuickNode, что и W7.8 (**`qnCall`**, feature **`sim`**), с учётом бюджета кредитов.
5. Запись в live JSONL **`execution_result`** со статусами **`sim_ok` | `sim_err`** и трассировкой (`intentId`, CU, ошибка симуляции).
6. Тесты с моками (`fetch`, `qnCall`, ФС keypair) без реальной отправки в mainnet.

## 2. Инварианты (нарушение — блок merge)

| ID | Инвариант |
|----|-----------|
| **P3-I1** | Файл keypair или raw-секрет **не читается** и не декодируется, если **`LIVE_STRATEGY_ENABLED≠1`** или **`LIVE_EXECUTION_MODE=dry_run`**. Допускается наличие переменной `LIVE_WALLET_SECRET` в окружении процесса при **dry_run**, но реализация **не должна** открывать путь к файлу и не должна вызывать `fromSecretKey` до перехода в контекст **simulate/live** с включённой стратегией. |
| **P3-I2** | При **`simulate`** запрещено вызывать **`sendTransaction`** и любые методы, переводящие транзакцию в очередь сети. |
| **P3-I3** | Секрет **никогда** не логируется (ни путь к файлу с полным содержимым, ни base58 строки ключа). Допустимо логировать **pubkey** и факт «wallet loaded». |
| **P3-I4** | Один логический swap intent → один **`intentId`** (UUID v4); пара **`execution_attempt`** → **`execution_result`** с тем же `intentId` (как в W8.0-p1). |

## 3. Переменные окружения

### 3.1 Уже заданные родительской спекой / конфигом

| Переменная | Назначение в Phase 3 |
|------------|----------------------|
| `LIVE_STRATEGY_ENABLED` | Только при **`1`** и режиме **`simulate`** разрешена загрузка ключа и симуляция исполнения (см. §4). |
| `LIVE_EXECUTION_MODE` | Для Phase 3 целевое значение разработки — **`simulate`**; при **`dry_run`** симуляция с подписью **не выполняется** (остаётся поведение Phase 2: quote/build по политике self-test). |
| `LIVE_WALLET_SECRET` | Абсолютный путь к JSON keypair (**предпочтительно**) или raw secret key в формате, поддерживаемом общим загрузчиком (один модуль — см. §5). На VPS: `chmod 600`, владелец совместим с политикой деплоя ([`W8.0_live_oscar_trading_bot.md`](./W8.0_live_oscar_trading_bot.md) §3.5). |

### 3.2 RPC и кредиты QuickNode

Единый HTTP endpoint для **`qnCall`** в репозитории задаётся **`SA_RPC_HTTP_URL`** ([`qn-client.ts`](../../../src/core/rpc/qn-client.ts)).

| Переменная | Обязательность | Описание |
|------------|----------------|----------|
| `SA_RPC_HTTP_URL` | **Обязательна**, если при **`LIVE_STRATEGY_ENABLED=1`** и **`LIVE_EXECUTION_MODE=simulate`** выполняется симуляция | QuickNode (или совместимый) HTTPS RPC URL; без него `qnCall` возвращает отказ (`SA_RPC_HTTP_URL missing`). |

**Опционально (Phase 3.1 или позже):** `LIVE_RPC_HTTP_URL` — если нужен отдельный endpoint для live; если задан, live-модули используют его **вместо** `SA_RPC_HTTP_URL` только для своих вызовов (зафиксировать в коде один приоритет). По умолчанию на одном VPS достаточно **`SA_RPC_HTTP_URL`**.

### 3.3 Параметры симуляции (аналог W7.8)

Ввести или переиспользовать согласованные с W7.8 ключи (префикс **`LIVE_SIM_*`** для изоляции от paper):

| Переменная | Смысл | Рекомендуемый дефолт |
|------------|-------|----------------------|
| `LIVE_SIM_ENABLED` | Мастер-флаг: выполнять ли цепочку sign+simulate при `simulate` | `1` при разработке Phase 3 |
| `LIVE_SIM_TIMEOUT_MS` | Таймаут RPC для `simulateTransaction` | `8000`–`30000` (как минимум не ниже safety-слоёв) |
| `LIVE_SIM_CREDITS_PER_CALL` | Аргумент `creditsPerCall` в `qnCall` для feature `sim` | Выровнять с **`PAPER_SIM_*`** или выделить константу после замера QN |
| `LIVE_SIM_REPLACE_RECENT_BLOCKHASH` | Параметр симуляции Solana | **`true`** (как в `sim-audit.ts`) |
| `LIVE_SIM_SIG_VERIFY` | Параметр симуляции Solana | **`false`** на Phase 3 при типичном Jupiter tx (как в W7.8), иначе документировать включение |

Jupiter API key: **`JUPITER_API_KEY`** — как в Phase 2; заголовок при наличии.

## 4. Конечный автомат «когда грузим кошелёк и симулируем»

```
LIVE_STRATEGY_ENABLED = 0  →  не читать keypair; не подписывать; не вызывать simulate для исполнения.
LIVE_STRATEGY_ENABLED = 1 AND executionMode = dry_run  →  не читать keypair для исполнения (Phase 2 self-test может использовать placeholder по текущей политике).
LIVE_STRATEGY_ENABLED = 1 AND executionMode = simulate  →  загрузить keypair; quote+swap build с реальным pubkey; подписать; qnCall simulateTransaction; записать execution_result.
LIVE_EXECUTION_MODE = live  →  в Phase 3 не реализовывать send (явная ошибка конфига или noop с execution_skip — выбрать одно при реализации; предпочтительно **fail-fast при старте**, если enabled=1 и mode=live без Phase 6).
```

**Рекомендация:** при **`LIVE_STRATEGY_ENABLED=1`** и **`LIVE_EXECUTION_MODE=live`** до появления Phase 6 — **падать при старте** с понятным сообщением (защита от случайной отправки не готовым кодом).

## 5. Загрузка keypair (реализация)

1. **Вход:** строка `LIVE_WALLET_SECRET`.
2. Если строка — путь к существующему файлу (эвристика: начинается с `/` на Unix или содержит `\`/`:` на Windows **или** `fs.existsSync` для переданного значения): прочитать файл **только** в ветке §4 «simulate».
3. Иначе: трактовать как secret key base58 / массив байт JSON — как в утилитах Solana (единый хелпер, например `loadLiveKeypair(secret: string): Keypair`).
4. Верификация: после загрузки проверить `publicKey` не нулевой; не логировать секрет.
5. Ошибки ФС или парсинга → **`execution_skip`** или падение при старте (для simulate-mode старта предпочтительно **fail-fast** при enabled=1, чтобы не крутиться без кошелька).

## 6. Цепочка Jupiter → подпись → симуляция

Порядок строгий:

1. **`liveFetchBuyQuote`** / эквивалент Phase 2 для нужного направления (buy path для канонического self-test).
2. **`liveBuildUnsignedSwapTx`** с **`userPublicKey = keypair.publicKey.toBase58()`** (те же флаги тела swap, что Phase 2 / W7.8: `wrapAndUnwrapSol`, `dynamicComputeUnitLimit: false`, если не согласовано иное в CHANGELOG).
3. Десериализация base64 ответа Jupiter в **`VersionedTransaction`** (или тип, который возвращает `@solana/web3.js` для данного формата).
4. **`transaction.sign([keypair])`** (или корректный API для partial sign — зафиксировать по фактическому формату Jupiter v1).
5. **`qnCall('simulateTransaction', [ serializedBase64, { encoding: 'base64', commitment: 'processed', replaceRecentBlockhash, sigVerify, innerInstructions } ], { feature: 'sim', creditsPerCall, timeoutMs })`**.
6. Разбор результата: наличие **`err`**, **`unitsConsumed`**, первые логи при ошибке — по образцу **`parseSimulateResult`** в `sim-audit.ts`.

При ошибке HTTP Jupiter или swap build — **`execution_result`** со статусом **`sim_err`** или отдельный **`execution_skip`** до попытки симуляции (согласовать с матрицей fsync W8.0-p1); в Phase 3 допускается только **`execution_result`** для единообразия «попытка исполнения».

## 7. JSONL (`liveSchema: 1`)

### 7.1 `execution_attempt` (перед симуляцией)

Уже определён в W8.0-p1. Phase 3 **обязана** заполнять:

- `intentId`, `side`, `mint`, `executionMode: 'simulate'`;
- **`quoteSnapshot`** — как Phase 2 (включая `swapBuildOk`, длину tx при успехе).

### 7.2 `execution_result`

Использовать существующие поля [`events.ts` / `ExecutionResultSchema`](../../../src/live/events.ts):

| Поле | При `sim_ok` | При `sim_err` |
|------|----------------|----------------|
| `status` | `sim_ok` | `sim_err` |
| `intentId` | ✓ | ✓ |
| `simulated` | `true` | `true` |
| `unitsConsumed` | число или `null` | `null` или число, если частично |
| `error` | опущено | `{ message, code? }` из симуляции / RPC |
| `txSignature` | **`null`** (нет отправки) | **`null`** |

Опционально аддитивно (не ломая **liveSchema: 1**): `simWallMs`, `qnReason` при skipped budget — только если расширение согласовано; минимальный набор выше **обязателен**.

## 9. Тестирование

1. **Unit:** загрузка keypair из fixture-файла в памяти / временный файл в тесте; отказ при `dry_run` без чтения.
2. **Unit:** мок `qnCall` — успех и ошибка симуляции → корректный `execution_result`.
3. **Интеграционный опционально:** devnet не обязателен, если моки покрывают цепочку.
