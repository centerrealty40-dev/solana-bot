# Spec W8.0-p1 — Контракт JSONL для live-oscar (Phase 1)

**Статус:** **выполнено в коде** (продукт **1.4.0**, см. [`CHANGELOG.md`](../release/CHANGELOG.md)); нормативный текст остаётся контрактом для replay и последующих фаз W8.0.

**Продукт:** `solana-alpha`, процесс **`live-oscar`** (отдельный от paper).  
**Зависит от:** [`W8.0_live_oscar_trading_bot.md`](./W8.0_live_oscar_trading_bot.md) §7; **реализованный** Phase 0 ([`W8.0_IMPLEMENTATION_PHASES.md`](./W8.0_IMPLEMENTATION_PHASES.md)) — `src/live/store-jsonl.ts`, отдельный `LIVE_TRADES_PATH`.

**Не путать с:** журналом paper (`PAPER_TRADES_PATH`) — смешение файлов **запрещено** инвариантом **L2** родительской спеки.

---

## 1. Зачем

- Единый **контракт строк** live-журнала **до** появления Jupiter/RPC/send (Phase 2–6), чтобы:
  - дашборды и пост-мортемы не ломались при добавлении исполнения;
  - **replay** (Phase 7) знал, какие поля обязательны и как сопоставлять **attempt → result**;
  - расширения были **аддитивными** (новые опциональные поля), а ломающие изменения шли только с новой **`liveSchema`** и записью в CHANGELOG.

---

## 2. Не-цели (Phase 1)

- Реальная подпись, `sendTransaction`, расчёт PnL, reconcile с цепью — **Phase 3+ / Phase 7**.
- Дублирование полного события paper `open` / `close` в live — **optional отдельная спека** (если понадобится зеркалировать в live JSONL); в **p1** достаточно событий исполнения и риска из §4.

---

## 3. Общая оболочка строки (все события)

Каждая строка JSONL — один объект JSON.

### 3.1 Обязательные корневые поля

| Поле | Тип | Описание |
|------|-----|----------|
| `ts` | `number` | Unix epoch **ms**, момент записи (как в paper). |
| `strategyId` | `string` | Идентификатор стратегии live (`LIVE_STRATEGY_ID`, напр. `live-oscar`). |
| `channel` | `'live'` | Явный маркер ветки live (не опираться только на путь файла). |
| `kind` | `string` | Дискриминатор типа события (см. §4). |
| `liveSchema` | `number` | Версия контракта **этой** строки. Для Phase 1 зафиксировать **`1`**. |

### 3.2 Опциональные корневые поля

| Поле | Тип | Описание |
|------|-----|----------|
| `profile` | `'oscar'` | Профиль паритета с paper (пока только Oscar). |
| `executionMode` | `'dry_run' \| 'simulate' \| 'live'` | Режим на момент события (если применимо). |
| `mint` | `string` | Mint SPL при привязке к позиции/сделке. |

### 3.3 Идемпотентность и трассировка

Для связки «намерение → исход»:

| Поле | Где обязательно | Описание |
|------|-----------------|----------|
| `intentId` | `execution_attempt`, и далее в парном `execution_result` / `execution_skip` | Уникальная строка **UUID v4** (RFC 9562) в нижнем регистре с дефисами. Генерируется **один раз** на попытку исполнения. |

**Правило:** для одного логического действия (один swap intent) ровно **один** `intentId`. Повторная запись с тем же `intentId` **запрещена** (ошибка разработки); повтор при сетевом сбое — новый `intentId` + опциональное поле `supersedesIntentId` только в документированных случаях (Phase 6+).

### 4.2 Исполнение (родительская спека §7)

#### `execution_attempt`

Фиксирует **намерение** выполнить swap (ещё без обязательного успеха сети).

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `intentId` | ✓ | UUID | §3.3 |
| `side` | ✓ | `'buy' \| 'sell'` | SOL→token vs token→SOL. |
| `mint` | ✓ | `string` | Токен торговли (не WSOL). |
| `intendedUsd` | опц. | `number \| null` | Нотация в USD, если определима; иначе `null`. |
| `intendedAmountAtomic` | опц. | `string` | Целое в минимальных единицах входного актива (строка, чтобы не терять u64). |
| `executionMode` | ✓ | см. выше | Должен совпадать с `LIVE_EXECUTION_MODE` процесса. |
| `quoteSnapshot` | опц. | `object` | Усечённый снимок quote (Phase 2+ заполняет): см. §5. |
| `targetPriceUsd` | опц. | `number \| null` | Для сверки §8.4 родительской спеки (когда трекер доступен). |

#### `execution_result`

Итог попытки (успех симуляции, успех send, или ошибка).

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `intentId` | ✓ | UUID | Ссылка на `execution_attempt`. |
| `status` | ✓ | `'sim_ok' \| 'sim_err' \| 'sent' \| 'confirmed' \| 'failed'` | Подмножество расширяется в Phase 6 только аддитивно (новые строки enum в CHANGELOG). |
| `txSignature` | опц. | `string \| null` | Base58 подписи; для `dry_run` обычно `null`. |
| `simulated` | опц. | `boolean` | `true`, если результат только из симуляции. |
| `unitsConsumed` | опц. | `number \| null` | CU из simulate. |
| `error` | опц. | `{ code?: number; message: string }` | Человекочитаемо + опционально код. |
| `executedPriceUsd` | опц. | `number \| null` | После исполнения (Phase 4+). |

**Правило порядка:** для одного `intentId` сначала пишется **`execution_attempt`**, затем ровно один терминальный **`execution_result`** или **`execution_skip`** (не оба как терминальные).

#### `execution_skip`

Исполнение не начато или остановлено до swap API / до подписи.

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `intentId` | опц. | UUID | Если skip до генерации intent — допускается **новый** UUID только для skip-строки; предпочтительно всегда создавать intent до gate и передавать тот же id (**рекомендуемая** политика реализации). |
| `reason` | ✓ | `string` | Код причины из фиксированного набора + произвольный суффикс через `:` запрещён без версии схемы; см. §6. |
| `detail` | опц. | `string` | Короткая строка (≤500 символов), без PII и секретов. |

#### `live_reconcile_report` (Phase 7, **`liveSchema: 2`**)

Итог boot **replay + reconcile** и опционально выборочной **`getTransaction`** проверки по журналу. Все прочие виды из §4 при записи по-прежнему используют **`liveSchema: 1`**, пока не зафиксирована миграция.

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `kind` | ✓ | `'live_reconcile_report'` | |
| `ok` | ✓ | `boolean` | Успех SPL reconcile; при `reconcileStatus: skipped` обычно `true`. |
| `reconcileStatus` | ✓ | см. ниже | `'ok' \| 'mismatch' \| 'rpc_fail' \| 'skipped'`. |
| `mode` | ✓ | `string` | Режим **`LIVE_RECONCILE_MODE`**: `report` \| `block_new` \| `trust_chain`. |
| `skipReason` | опц. | `string` | При `skipped`. |
| `mismatches` | опц. | `array` | `{ mint, expectedRaw, actualRaw, note? }[]`. |
| `walletSolLamports` | опц. | `string \| null` | Lamports SOL с кошелька при успешном RPC. |
| `chainOnlyMints` | опц. | `string[]` | SPL mints на цепи без позиции в replay. |
| `journalReplayTruncated` | опц. | `boolean` | Хвостовое чтение журнала (байтовый лимит). |
| `txAnchorSample` | опц. | `object` | `{ checked: number, notFound: string[], rpcErrors: number }` при **`LIVE_RECONCILE_TX_SAMPLE_N` > 0**. |

### 4.3 Риск и капитал (родительская спека §3.3–§3.4)

#### `risk_block`

Срабатывание лимита до постановки ордера.

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `limit` | ✓ | `string` | Идентификатор лимита: `max_strategy_loss` \| `max_open_positions` \| `max_position_usd` \| `kill_consecutive_fail` \| `min_wallet_sol` \| другое — перечисляется в CHANGELOG при добавлении. |
| `detail` | опц. | `object` | Числовые снимки (напр. `lossUsd`, `openCount`) без избыточной точности float в критичных полях — предпочтительно строки или округление в спеке Phase 5. |

#### `capital_skip`

Невозможность входа из-за правила **2X** / свободного баланса.

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `reason` | ✓ | `string` | Значения из родительской §3.4: `insufficient_free_balance_no_positions` \| `no_profitable_position_to_close` \| `insufficient_free_after_rotation` \| … |
| `freeUsdEstimate` | опц. | `number \| null` | Если доступно безопасно оценить. |
| `requiredFreeUsd` | опц. | `number \| null` | Порог **2X** в USD-эквиваленте. |

#### `capital_rotate_close`

Решение закрыть прибыльную позицию для освобождения капитала.

| Поле | Обяз. | Тип | Описание |
|------|-------|-----|----------|
| `mint` | ✓ | `string` | Закрываемая позиция. |
| `unrealizedPnlUsd` | опц. | `number` | На момент решения. |
| `txSignature` | опц. | `string \| null` | После Phase 6 — подпись закрывающего swap. |

## 7. Политика `fsync`

Цель: не терять критичные события при краше после записи.

| `kind` | `fsync` после записи |
|--------|----------------------|
| `live_boot` | да |
| `live_shutdown` | да |
| `execution_result` | да, если `status` терминальный (`confirmed`, `failed`, `sim_err`, …) |
| `risk_block` | да |
| `capital_rotate_close` | да |
| `execution_attempt` | рекомендуется да для live-режима; для чистого `dry_run` допускается нет (конфигурируемо **Phase 8**, по умолчанию да) |
| `heartbeat` | нет |
| `execution_skip` | да, если связано с риском или отказом после расхода RPC-кредита |

Реализация: **`appendLiveJsonlEvent(body, { sync? })`** в `src/live/store-jsonl.ts` + **`liveEventDefaultFsync`** по таблице §7; переопределение `sync` вручную допускается.

## 8. Replay и совместимость (forward-looking, Phase 7)

- Загрузчик **обязан** игнорировать **неизвестные** ключи в объекте (мягкая совместимость).
- Неизвестный **`kind`** → событие пропускается для state-machine с предупреждением в логах (или отдельный счётчик метрик).
- Изменение **семантики** существующего `kind` при том же `liveSchema` **запрещено**; нужен новый `liveSchema` или новый `kind`.

## 11. Forbidden

- Писать события live в `PAPER_TRADES_PATH` или смешивать стратегии в одном `LIVE_TRADES_PATH` без смены `strategyId`.
