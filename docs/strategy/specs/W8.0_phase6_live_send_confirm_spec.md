## 1. Цели Phase 6

1. Включить **`LIVE_EXECUTION_MODE=live`** при **`LIVE_STRATEGY_ENABLED=1`** без fail-fast конфига — с **жёсткими** предохранителями по ENV (см. §3).
2. После прохождения **Phase 5** (и симуляции по §5 при включённом флаге) выполнять **`sendTransaction`** через тот же контур **`qnCall`** / **`SA_RPC_HTTP_URL`**, с учётом кредитов QuickNode (отдельный **feature** или документированное переиспользование — см. §4.3).
3. Дождаться **подтверждения** по политике commitment (**`confirmed`** или **`finalized`**) и записать **`execution_result`** с **`txSignature`**, статусом **`confirmed`** (или промежуточным **`sent`** при явной двухфазной записи — см. §7).
4. Уточнить семантику **`LIVE_KILL_AFTER_CONSEC_FAIL`** для **`live`**: счётчик увеличивается на **терминальный неуспех отправки** или **финальный отказ** после отправки (не на каждый промежуточный timeout опроса статуса — см. §6).
5. Для **ротации капитала §3.4 в `live`**: после **`confirmed`** успешного **sell_full** пересчитывать **`free_usd`** по **RPC** (реальный баланс), затем продолжать или завершать сценарий входа — **без** «виртуального bump» из quote как единственного SSOT (quote — только оценка до подтверждения).

## 2. Инварианты (нарушение — блок merge)

| ID | Инвариант |
|----|-----------|
| **P6-I1** | **`sendTransaction`** вызывается **только** после прохождения всех применимых гейтов Phase 5 для данного intent (**buy_open**, **dca_add**, **sell_*** по решению трекера / ротации). Исключения — отдельная спека + bump **MINOR+**. |
| **P6-I2** | Порядок: **Oscar (P4)** → **Phase 5** → *(опц.)* **`simulateTransaction`** → **`sendTransaction`** → **confirm** → JSONL **`execution_result`**. Не вставлять шаги, меняющие экспозицию, между Phase 5 и подписанной отправкой без явной записи в CHANGELOG. |
| **P6-I3** | Один **`intentId`** на один логический swap; пара **`execution_attempt`** → **`execution_result`** сохраняется (**W8.0-p1**). | **P6-I4** | Секрет кошелька и приватные ключи **не** попадают в JSONL и не логируются (**P3-I3** сохраняется). В журнал допускаются **pubkey**, **signature**, **slot**, **mint**, суммы в безопасном виде. |
| **P6-I5** | При **`LIVE_STRATEGY_ENABLED=0`** или **`dry_run`** — **нет** `sendTransaction` в mainnet-beta (как сегодня для simulate-only веток). |

## 3. Переменные окружения (минимум Phase 6)

Расширение Zod в [`config.ts`](../../../src/live/config.ts); значения по умолчанию должны быть **консервативны** для канарейки.

### 3.1 Режим и блокировка подтверждения

| Переменная | Тип | Смысл |
|------------|-----|--------|
| `LIVE_EXECUTION_MODE` | уже есть | Разрешить значение **`live`** при **`LIVE_STRATEGY_ENABLED=1`** после реализации Phase 6 (снять текущий fail-fast «до Phase 6»). |
| `LIVE_CONFIRM_COMMITMENT` | `'processed'` \| `'confirmed'` \| `'finalized'` | Уровень успеха подтверждения; для прод стартовать с **`confirmed`**, для стресс-тестов **`finalized`** — по решению владельца. |
| `LIVE_CONFIRM_TIMEOUT_MS` | целое ≥ `3000` | Максимальное ожидание финального статуса после **`sent`**; при истечении → **`execution_result`** **`failed`** с кодом причины и **не** считать сделку исполненной для стратегии. |
| `LIVE_SEND_SKIP_PREFLIGHT` | bool | Проброс в RPC **`sendTransaction`**; дефолт **`false`** на канарейке. |
| `LIVE_SIM_BEFORE_SEND` | bool | Если **`true`**, перед каждым send выполнять **`simulateTransaction`** (уже реализованный путь); если **`false`**, только send (не рекомендуется до стабилизации). |

### 3.2 Лимиты повторов и бэкфофф

| Переменная | Смысл |
|------------|--------|
| `LIVE_SEND_MAX_RETRIES` | целое ≥ `0` — повтор **`sendTransaction`** при ошибках класса «transport / rate / blockhash» (не при отказе симуляции программы). |
| `LIVE_SEND_RETRY_BASE_MS` | базовая задержка backoff между повторами. |

Детализация классов ошибок Solana RPC — в PR (таблица **retryable / non-retryable**).

### 3.3 Опционально (не блокер merge)

| Переменная | Смысл |
|------------|--------|
| `LIVE_RPC_HTTP_URL` | Если задан — использовать **только для** вызовов send/confirm live-модуля вместо `SA_RPC_HTTP_URL` (изолировать лимиты); иначе один URL как сейчас. |

## 5. Конвейер исполнения (`live`)

Для каждого intent (**buy_open**, **dca_add**, **sell_partial**, **sell_full**), уже прошедшего Phase 4–5:

1. **Quote + swap build** (существующий Jupiter путь) с реальным **`userPublicKey`**.
2. **Подпись** локально (как Phase 3–4).
3. Если **`LIVE_SIM_BEFORE_SEND=1`**: **`simulateTransaction`**; при **`sim_err`** — **`execution_result`** **`failed`** (или **`sim_err`** если сохраняется двухшаговая семантика — **выбрать одно** и выровнять с dashboard); **не** отправлять tx.
4. **`sendTransaction`**.
5. Запись **`execution_result`** со статусом **`sent`** и **`txSignature`** (опционально как промежуточная строка — если реализация остаётся однострочной до confirm, зафиксировать в CHANGELOG).
6. Цикл опроса до **`LIVE_CONFIRM_COMMITMENT`** или timeout.
7. Финальная запись **`execution_result`** **`confirmed`** с **`txSignature`**, опционально **`slot`**, **`executedPriceUsd`** (если извлекается из meta/logs детерминированно — иначе опционально отложить на Phase 7).

**Ротация капитала в `live`:** каждый **`sell_full`** по выбранному mint должен дойти до **`confirmed`** (или **`failed`** с явным выходом из сценария входа) перед следующим шагом каскада или перед **`buy_open`** исходного сигнала.

## 6. `LIVE_KILL_AFTER_CONSEC_FAIL` в режиме `live`

- **Не** увеличивать счётчик на успешные симуляции с последующим успешным confirm.
- Увеличивать на **терминальный** провал: **`sim_err`** (если включён pre-send sim), **`failed`** после send (ошибка RPC «transaction invalid»), или **timeout confirm** при исчерпании политики retries (одно логическое «падение intent» = один инкремент).
- Успешный **`confirmed`** сбрасывает счётчик (как в simulate для **`sim_ok`**).

Семантика должна быть общей с [`phase5-state.ts`](../../../src/live/phase5-state.ts); при необходимости развести хуки **`notifyLiveExecutionSimErr`** vs **`notifyLiveExecutionSendErr`** — зафиксировать в коде и CHANGELOG.

## 7. JSONL (аддитивно, `liveSchema: 1`)

Использовать существующие виды [`events.ts`](../../../src/live/events.ts):

- **`execution_attempt`** — без изменения обязательной семантики; **`executionMode: 'live'`**.
- **`execution_result`**:
  - **`status: 'sent'`** — после принятия RPC в mempool (или после успешного HTTP без ошибки — уточнить по ответу провайдера в PR);
  - **`status: 'confirmed'`** — после достижения **`LIVE_CONFIRM_COMMITMENT`**;
  - **`status: 'failed'`** — отправка не удалась, confirm timeout, или on-chain err;
  - **`txSignature`**: строка или **`null`** до известности (предпочтительно не писать финальную строку без signature).
- **`capital_rotate_close`**: при **`live`** допускается заполнить **`txSignature`** после confirm ротационного sell.

Запрещено добавлять в JSONL PII и секреты. Любое новое обязательное поле для **`liveSchema: 1`** требует согласования с **W8.0-p1** или bump **`liveSchema`** (нежелательно в Phase 6 без крайней необходимости).

## 8. Взаимодействие со стратегией (paper parity)

- При **`confirmed`** успешном исполнении **buy**/**dca**/**sell** процесс **Oscar** уже мутирует **`open`/`closed`** через существующий поток **`papertrader`** (как в Phase 4–5). Phase 6 **не** меняет условия вызова adapter; только транспорт после гейтов.
- Если **`failed`** после того как paper-логика уже продвинула состояние (краевой случай) — минимальный ответ Phase 6: явное событие JSONL + рекомендация запуска **Phase 7** reconcile; глубокий автоматический rollback не обязателен в p6.

## 9. Acceptance criteria (измеримые)

- [ ] При **`LIVE_STRATEGY_ENABLED=1`**, **`LIVE_EXECUTION_MODE=live`**, валидном ключе и RPC — один полный swap проходит цепочку до **`execution_result`** **`confirmed`** с непустым **`txSignature`** на devnet или mainnet-beta по выбору окружения теста (юнит-тесты с моками RPC обязательны; опционально интеграционный smoke с помеченным mint).
- [ ] При **`LIVE_SIM_BEFORE_SEND=1`** и намеренно битой симуляции — **нет** send, есть **`sim_err`** / **`failed`** по выбранной политике §5.
- [ ] **`npm run typecheck`**; тесты на разбор статусов signature и backoff (моки).
- [ ] **`CHANGELOG`** + bump **`VERSION`** (**MINOR** рекомендуется).
- [ ] `.env.example` содержит ключи §3.

---

## 10. Forbidden actions

- Включать **`live`** по умолчанию в **`ecosystem.config.cjs`** без явного решения владельца и канарейки лимитов §3.3–§3.4.
- Логировать serialized подпись или secret key.
- Считать intent успешным для увеличения экспозиции без **`confirmed`** (или без явной политики §3 для вашего commitment).
- Обходить Phase 5 для «быстрой» отправки.
