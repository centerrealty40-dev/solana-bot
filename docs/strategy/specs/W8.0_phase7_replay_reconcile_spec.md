## 1. Цели Phase 7

1. После рестарта процесса **`live-oscar`** восстановить **достаточное** состояние стратегии (множество **`open`**, журнал **`closed`**, счётчики, релевантные для Phase 5–6 поля **`OpenTrade`**) из **только** live JSONL (**`LIVE_TRADES_PATH`**), без чтения paper JSONL для этого состояния (**L2**).
2. Выполнить **reconcile** с Solana: фактические балансы **SOL** и **SPL** (по отношению к pubkey кошелька live) сопоставить с ожидаемым состоянием, выведенным из replay + подтверждённых swap (**`txSignature`**).
3. Зафиксировать политику расхождений (лог / JSONL / блок новых входов) — одна выбранная модель на релиз + запись в CHANGELOG.
4. Обеспечить **детерминированный** порядок replay (§4) и предсказуемое поведение при неполных журналах (обрыв после `execution_attempt` без пары `execution_result`).

## 2. Инварианты (нарушение — блок merge)

| ID | Инвариант |
|----|-----------|
| **P7-I1** | Состояние позиций live после успешного replay **не** опирается на **`PAPER_TRADES_PATH`** / paper JSONL (**P4-I1** / **L2** сохраняются). Допускается читать **только** общие **`PAPER_*`** параметры стратегии (как сейчас), но не события сделок paper. |
| **P7-I2** | Любое решение «доверять журналу» vs «доверять цепи» при конфликте **документировано** в §6 и в CHANGELOG; молчаливое превышение экспозиции запрещено. |
| **P7-I3** | Replay **идемпотентен**: повторный прогон того же хвоста файла даёт то же восстановленное состояние (при неизменном файле и версии парсера). |
| **P7-I4** | Для режима **`live`** якорём исполнения считается **`execution_result`** со статусом **`confirmed`** и известным **`txSignature`**; **`simulate`** / **`dry_run`** строки **не** порождают ончейн-ожиданий, но могут порождать **in-memory** позиции только если это явно разрешено политикой §5.2 (по умолчанию — **не** восстанавливать «фиктивные» позиции из одного `sim_ok` без последующего подтверждённого live-исполнения). |
| **P7-I5** | Секреты и PII в новых событиях JSONL запрещены (как в Phase 1). |

## 3. Объём replay (минимальный v1)

### 3.1 Источник данных

- Файл **`LIVE_TRADES_PATH`**; кодировка и построчный JSON как у Phase 1.
- Опционально **`LIVE_REPLAY_TAIL_LINES`** или **`LIVE_REPLAY_SINCE_TS`** (ENV) — ограничение объёма для больших файлов; дефолт и семантика — в PR (например «полный файл до N MB, иначе tail»).

### 3.2 Какие события участвуют

Минимум для восстановления торгового контура (уточнить в PR при появлении зеркалирования paper-событий в live):

| kind | Роль в replay |
|------|----------------|
| **`execution_attempt`** | Контекст intent (**mint**, **side**, **`intentId`**, **`intendedUsd`**, quote snapshot по необходимости). |
| **`execution_result`** | Итог: только строки со статусом **`confirmed`** (для **`live`**) увеличивают/уменьшают модель позиции; **`failed`** / **`sim_err`** — откат «ожидаемого» pending для данного **`intentId`** (если pending введён). |
| **`risk_block`**, **`capital_skip`**, **`capital_rotate_close`** | По умолчанию **не** меняют **`open`** напрямую; используются для диагностики и аудита. Исключение — если в Phase 6+ записан **`capital_rotate_close`** с **`txSignature`** и подтверждённым закрытием: трактовать как подсказку к reconcile (§5). |

**Замечание:** сегодня paper **`open`** строится из событий **`open`**, **`dca_add`**, **`partial_sell`**, **`close`** в **paper** JSONL. В live JSONL Phase 4–6 **может не быть** полного дубля этих событий. Phase 7 **обязана** либо:

- **(A)** ввести минимальный набор зеркальных событий live (**`live_position_open`**, **`live_position_dca`**, …) — только с bump **`liveSchema`** и обновлением W8.0-p1, **либо**
- **(B)** детерминированно **выводить** изменения позиций только из **`execution_attempt`/`execution_result`** + статических правил размеров (сложнее; зафиксировать в PR).

**Выбор (A) vs (B)** — до первого merge Phase 7; рекомендация спеки: **(A)** аддитивными `kind` при сохранении обратной совместимости парсера.

### 3.3 Pending intents

Если для **`intentId`** есть **`execution_attempt`**, но нет завершающего **`execution_result`** у конца файла:

- политика по умолчанию: считать intent **незавершённым**, **не** менять **`open`** до разрешения (timeout стартового reconcile → **`failed`** в диагностике или отдельное событие **`reconcile_pending`** — см. §8).

## 4. Порядок и детерминизм replay

1. Сортировка строго по **`ts`** возрастанию; при равном **`ts`** — порядок строк в файле (**стабильная нумерация line index** в памяти для отладки).
2. Индекс **`intentId` → последнее известное состояние** обновляется при каждой паре attempt/result.
3. Запрещено недетерминированное чтение (parallel без порядка слияния).

## 5. Reconcile с цепью

### 5.1 RPC

- **`getBalance`** (pubkey кошелька live) — SOL.
- **`getTokenAccountsByOwner`** или **`getParsedTokenAccounts`** — множество SPL ATA с ненулевым балансом (политика фильтра закрытых счетов — в PR).

Учёт кредитов: тот же **`qnCall`**; feature **`live_send`** или выделенный **`live_read`** — **выбрать один** и задокументировать в CHANGELOG.

### 5.2 Сравнение

- **Ожидаемые** остатки по mint: из восстановленного **`open`** ( **`remainingFraction`**, **`totalInvestedUsd`**, учёт decimals) **или** из упрощённой модели «сырой token amount» — зафиксировать в PR.
- **Фактические** остатки: из RPC.
- Допуск **`LIVE_RECONCILE_TOLERANCE_ATOMS`** или относительный допуск — опционально ENV.

### 5.3 Политики расхождения (`LIVE_RECONCILE_MODE`)

| Режим | Поведение (канон-пример) |
|-------|---------------------------|
| **`report`** | Только JSONL / лог; торговля не блокируется (**не рекомендуется** для prod live без человека). |
| **`block_new`** | Блок новых **`buy_open`** / **DCA** (аналог **`risk_block`** с кодом **`reconcile_divergence`**); выходы по трекеру — по умолчанию **разрешены** (аварийность). |
| **`trust_chain`** | Сброс/коррекция внутренней модели по цепи и продолжение (опасно; только с явным approval в CHANGELOG и лимитами). |

Дефолт для канарейки: **`block_new`**.

## 6. Старт процесса (`live-oscar`)

Рекомендуемая последовательность при **`LIVE_STRATEGY_ENABLED=1`**:

1. Загрузить конфиг + live store path.
2. **Replay** JSONL → **`open`/`closed`** (и прочее минимально необходимое).
3. **Reconcile** с цепью.
4. Применить **`LIVE_RECONCILE_MODE`**.
5. Только затем запускать основной цикл discovery/tracker (как Phase 4–6).

При **`LIVE_STRATEGY_ENABLED=0`** replay может быть пропущен или выполняться в режиме «dry report» — зафиксировать в PR.

## 7. Взаимодействие с Phase 5–6

- **`LIVE_MAX_OPEN_POSITIONS`**, **`LIVE_MAX_STRATEGY_LOSS_USD`** и §3.4 **должны** использовать восстановленное **`open`/`closed`** после replay, а не пустое состояние.
- После **`trust_chain`** или ручного исправления журнала — обязательный повторный старт процесса или явная команда «reconcile-only» (опционально CLI/ENV в PR).

## 10. Forbidden actions

- Смешивать для восстановления **`open`** события paper и live без явной двухфазной спеки + bump **MAJOR/MINOR**.
- Игнорировать **`confirmed`** **`txSignature`** при наличии при **`live`** (цепь должна быть использована в reconcile v1 для проверки критичных путей — хотя бы выборочно или для последних N сделок).
- Автоматически **`trust_chain`** в prod без записи в CHANGELOG и без лимитов.

## 12. Связь с Phase 6

Phase 6 обеспечивает **непротиворечивые** **`txSignature`** для успешных swap; Phase 7 **потребляет** их для связи журнал ↔ цепь. Любое изменение формы **`execution_result`** — согласовать с Phase 6 и **W8.0-p1**.
