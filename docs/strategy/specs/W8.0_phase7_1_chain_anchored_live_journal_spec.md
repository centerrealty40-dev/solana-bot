## 1. Постановка проблемы

**Симптом:** бумажный Oscar фиксирует `open` в **`PAPER_TRADES_PATH`**, а live при **`LIVE_RECONCILE_MODE=block_new`** блокирует новые входы из‑за **расхождения** между:

- ожидаемыми остатками SPL, выведенными из **replay `LIVE_TRADES_PATH`** (`live_position_*`), и  
- **фактическими** остатками на **`LIVE_WALLET`** по RPC.

**Корневая причина (подлежит устранению кодом):** в восстановленном состоянии live возможны **«фантомные» открытые позиции** — запись в журнале live создаёт ожидание баланса **без** достаточного цепочного доказательства или после **не доведённого до согласованного состояния** исполнения (`confirm_timeout`, расхождение notional и т.д.).

**Цель p7.1:** зафиксировать инварианты и поведение так, чтобы **повторение класса рассинхрона** было **структурно затруднено**, а оператор имел **явную диагностику** и **безопасный откат**.

## 2. Цели и метрики успеха

| ID | Цель | Измеримый критерий |
|----|------|---------------------|
| **G1** | Любая позиция в replay **`open`** для live имеет **цепочный якорь** | Для каждого mint в `open` после replay существует непрерывная цепь §5.1 |
| **G2** | Reconcile **не опирается** на противоречивые строки `live_position_*` | Ghost-кандидаты §7 либо исключены из ожидаемых остатков, либо явно помечены событием §8 |
| **G3** | SSOT размера ноги **согласован** между paper и live для процесса `live-oscar` | При старте процесса проверка §6; несогласованность → **`live_boot` предупреждение** + блок новых входов по явному `risk_block` коду |
| **G4** | `confirm_timeout` не оставляет «висящее» несоответствие | Политика §9; журнал содержит достаточно данных для repair или явный failed |

## 3. Новые и уточняющие инварианты

Нумерация продолжает Phase 7 (**P7-I\***). Нарушение инварианта §3 — **блок merge** PR, реализующего p7.1.

| ID | Инвариант |
|----|-----------|
| **P7-I6** | Событие **`live_position_open`** (и каждое последующее **`live_position_dca`**, увеличивающее экспозицию по mint) **не записывается** в `LIVE_TRADES_PATH`, пока для соответствующего увеличения нет **`execution_result`** со статусом **`confirmed`** и непустым **`txSignature`**, либо пока не выполнена явная политика repair §9.2, задокументированная в том же журнале. |
| **P7-I7** | В **`openTrade`** снимке для live обязательно сохраняются **цепочные ссылки**: минимум **`entryLegSignatures: string[]`** (или эквивалентное имя), пополняемые при каждой подтверждённой покупке (open/DCA). Значения — **реальные подписи транзакций** с Solana mainnet. Пустой массив при **`live_position_open`** — **запрещён** после merge p7.1. |
| **P7-I8** | Replay **игнорирует** (не включает в `open`) любое **`live_position_*`**, для которого §5.2 классифицирует строку как **`ghost`** — до явного операторского разрешения через режим **`LIVE_REPLAY_TRUST_GHOST_POSITIONS=1`** (дефолт **0**) или до записи корректирующего события §8. |
| **P7-I9** | При **`LIVE_STRATEGY_ENABLED=1`** и **`LIVE_EXECUTION_MODE=live`**: **`PAPER_POSITION_USD`** (или канонический SSOT размера ноги paper для Oscar в [`ecosystem.config.cjs`](../../../ecosystem.config.cjs)) **должен** совпадать с **`phase5`** аргументом `intendedUsd` для buy_open (§6). Расхождение → **`risk_block`** с кодом **`parity_notional_mismatch`** (новый `limit`), без «тихого» продолжения. |

**Уточнение P7-I2 (наследие):** политика «журнал vs цепь» дополняется: при конфликте **приоритет цепи** для определения **фактических** остатков; **журнал** определяет **намерение** стратегии только после прохождения фильтра §7.

## 4. Контракт данных (JSONL и снимок openTrade)

### 4.1 Расширение снимка `openTrade` в событиях `live_position_*`

В теле **`openTrade`** (как сейчас `Record<string, unknown>` с сериализацией из paper types):

| Поле | Тип | Обязательность | Смысл |
|------|-----|------------------|--------|
| `entryLegSignatures` | `string[]` | **обязательно** с p7.1 | Все подписи подтверждённых пополнений позиции (первая — initial open swap). |
| `chainAnchorVerifiedTs` | `number` (epoch ms), опционально | желательно | Время последней успешной верификации баланса против подписи (диагностика). |

**Совместимость:** старые строки журнала без `entryLegSignatures` → replay классифицирует как **legacy_unanchored** (§7.3).

### 4.2 Новые / уточняемые события (минимум)

Расширение **`liveSchema`**: если добавляются новые `kind`, следовать Phase 1: Zod в `events.ts`, политика fsync в `store-jsonl.ts`.

| `kind` | Назначение |
|--------|------------|
| **`live_position_anchor`** (опционально v1) | Явная запись «mint M якорён подписью S» для упрощения аудита (если не хотите дублировать только через `openTrade`). **Решение в PR:** реализовать или отложить; минимальный v1 может обойтись только полями в `openTrade`. |
| **`live_reconcile_quarantine`** | Одна или несколько строк при обнаружении ghost: `{ mint, reason, journalLineHint?, suggestedAction }`. |

Детали полей — в PR с приложенными примерами строк (без секретов).

## 5. Цепочный якорь (определение для G1)

**Цепочный якорь** для mint **M** на момент записи **`live_position_open`**:

1. Существует **`execution_result`** с `status=confirmed`, непустым `txSignature`, соответствующим **`execution_attempt`** с `side=buy` и `mint=M`.  
2. По **`getTransaction(signature)`** с политикой commitment как у Phase 6: `meta.err == null`.  
3. Дельта SPL по **M** на pubkey кошелька live (сумма по ATA owner=wallet) **≥ минимально ожидаемому** сырому количеству из исполнения с допуском **`LIVE_RECONCILE_TOLERANCE_ATOMS`** (или относительный режим — только если уже описан в runbook).

Для **DCA**: то же для каждой новой подписи; массив **`entryLegSignatures`** дополняется.

## 6. SSOT нотационного размера (P7-I9)

**Канон для процесса `live-oscar`:**

- **`X_usd`** = значение, используемое paper Oscar для размера первой ноги при записи в paper JSONL (**`PAPER_POSITION_USD`** в блоке env **`live-oscar`** в [`ecosystem.config.cjs`](../../../ecosystem.config.cjs)).  
- **`LIVE_MAX_POSITION_USD`** (и при наличии **`LIVE_ENTRY_NOTIONAL_USD`** — по политике [`phase5-runtime.ts`](../../../src/live/phase5-runtime.ts)) **должны** удовлетворять:

  \[
  \texttt{intendedUsd} = X\_usd \le \texttt{LIVE\_MAX\_POSITION\_USD}
  \]

**Проверка:** при старте (`live_boot` или сразу после загрузки конфигов) если не выполняется — записать **`risk_block`** `parity_notional_mismatch` с полями `{ paperPositionUsd: X_usd, liveMaxPositionUsd, liveEntryNotionalUsd }` и **не** разрешать новые входы до исправления конфига.

## 7. Изменения replay (`replay-strategy-journal.ts` и вызывающий код)

### 7.1 Порядок обработки

Сохранить **P7-I3** (сортировка по `ts`, tie-break по индексу строки).

### 7.2 Фильтрация ghost

После применения события **`live_position_open`** / **`live_position_dca`** к черновику `open`:

- Если у снимка **нет** непустого **`entryLegSignatures`** → класс **`legacy_unanchored`** → **не включать** mint в финальный `open` при дефолтном режиме (**P7-I8**).  
- Если массив есть: для каждой подписи выполнить **lazy** или **batched** проверку §5.1 (политика производительности — в PR: кэш по подписи на процесс старта). При недоступности RPC → см. §7.4.

### 7.3 Классификация строк для reconcile

| Класс | Условие | Вклад в ожидаемые остатки |
|-------|---------|---------------------------|
| **anchored_ok** | §5.1 выполнен для всех подписей ноги | да |
| **ghost** | подпись invalid / tx err / баланс не сходится | нет (quarantine §8) |
| **legacy_unanchored** | нет полей якоря | нет при дефолте |

### 7.4 RPC недоступен при старте

Если проверка подписи не удалась по сети:

- **Не** трактовать как «ghost» автоматически; статус **`anchor_rpc_pending`**.  
- Политика: при **`anchor_rpc_pending`** для любого mint в последнем применённом `live_position_open` — **`live_reconcile_report`** с `reconcileStatus: rpc_fail` или расширение существующего отчёта; **`LIVE_RECONCILE_MODE=block_new`** сохраняет блокировку до успешной верификации **или** операторского режима (§11).

## 8. Quarantine и наблюдаемость

При классификации **ghost**:

1. Записать **`live_reconcile_quarantine`** (или эквивалентный диагностический блок внутри существующего **`live_reconcile_report`** — выбрать один канон в PR).  
2. Heartbeat (`§` runbook) должен экспонировать счётчик **`quarantinedMints`** или список префиксов mint.

**Запрещено:** автоматическое удаление строк из файла журнала на проде без отдельного утверждённого инструмента миграции (§10).

## 9. Политика `confirm_timeout` и repair

### 9.1 Единый поток

После отправки swap и получения подписи:

1. Если confirm timeout — **обязательный** асинхронный или следующий тик **`recoverTxStatus(signature)`**.  
2. Если tx **успешна** на цепи — идемпотентно привести журнал live к состоянию «открыто» (расширить существующий [`repair-missed-live-buys.ts`](../../../src/live/repair-missed-live-buys.ts) или общий модуль **repair**):  
   - дописать / исправить **`live_position_open`** или связку attempt/result так, чтобы выполнялся **P7-I6**.  
3. Если tx **неуспешна** — **`execution_result`** с явной ошибкой; **не** писать **`live_position_open`**.

### 9.2 Связь с env

Сохранить и задокументировать **`LIVE_REPAIR_MISSED_OPENS`** и пороги возраста из текущего runbook; при расширении — новые ключи только с дефолтом, совместимым с канарейкой.

## 13. Acceptance criteria (обязательные к закрытию PR)

1. Имеется тест: при синтетическом журнале с **`live_position_open`** без якорей — replay **`open.size === 0`** при `LIVE_REPLAY_TRUST_GHOST_POSITIONS=0`.
