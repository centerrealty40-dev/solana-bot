# W9.0 — Dip-bot intelligence (`dip_bot`): спецификация продукта

**Статус:** normative (к исполнению агентом-исполнителем).  
**Продукт:** Solana Alpha (`solana-alpha`).  
**Ветка интеграции:** `v2`.  
**Зависимости:** Postgres с `*_pair_snapshots` и существующей моделью тегов кошельков (`bot` и др.); журналы сделок стратегий (JSONL); платный RPC (QuickNode и т.п.) для чтения транзакций.  
**Внешний индексатор:** не используется (явное решение заказчика).

**Связанные нормативы:** [`../release/RELEASE_OPERATING_MODEL.md`](../release/RELEASE_OPERATING_MODEL.md), [`../release/NORM_UNIFIED_RELEASE_AND_RUNTIME.md`](../release/NORM_UNIFIED_RELEASE_AND_RUNTIME.md).

---

## 1. Цель продукта

Построить **аналитический подпродукт**, который:

1. По **уже совершённым входам по проливу** (якорь известен; в MVP источник якорей — **только Live Oscar**, см. §6.1) находит на чейне **покупателей того же mint в окне до нашего входа**.
2. По **повторяемости** и объёму выделяет адреса/кластеры **системных ранних покупателей пролива** и помечает их узким тегом **`dip_bot`** в БД (в дополнение к широкому тегу **`bot`**, если он уже есть).
3. В **фазе 2** (после снижения чувствительности к RPC) дополняет профиль **`dip_bot`** таксономией **логики выхода** (для калибровки собственного входа — отдельный контур; не блокер фазы 1).
4. Обеспечивает **два режима по расписанию** (§11): **начальный сбор (bootstrap)** — прогоны **значительно чаще**, чтобы быстро накопить множество `dip_bot` и начать с ними работать; **steady state** — обновление актуальности **раз в неделю** с минимальной нагрузкой на RPC.

**Не цель:** монетизация «на ботах», публичные списки с обвинениями, фронтраннинг; только **внутренний риск-интеллект** для решений стратегии.

## 2. Определения

| Термин | Смысл |
|--------|--------|
| **Якорное событие** | Наш **фактический вход по проливу**: известны минимум `mint`, `entry_ts` (unix ms), `strategy_id`, опционально `pair_address`, DEX/source, `total_invested_usd` первой ноги, `signature` live при наличии. Источник строк: JSONL `open` / внутренняя таблица агрегатов — зафиксировать в реализации один первичный источник. |
| **`dip_bot` (тег)** | Узкая метка сущности (**адрес** или **кластер**): системно участвует в **ранних покупках пролива** относительно нашего якоря в типичных условиях. |
| **Пре-окно** | Интервал **`[entry_ts − T_pre, entry_ts)`** на оси времени чейна; **пост-окно фазы 1** по умолчанию **не используется** (или минимально — см. §6). |
| **Кластер** | Группа адресов, эвристически отнесённых к **одному организатору** (общий funding / коррелированное поведение — см. §8). |

## 3. Проверенная информация о задержке «увидеть пролив» (для выбора `T_pre`)

**Факт из кода репозитория:**

- DEX-коллекторы (`scripts-tmp/raydium-collector.mjs`, `meteora-`, `orca-`, `moonshot-`, `pumpswap-collector.mjs`) используют **`INTERVAL_MS = Number(process.env.*_COLLECTOR_INTERVAL_MS || 60_000)`** — по умолчанию **60 секунд** между тиками записи снимков в `*_pair_snapshots`.
- В **`ecosystem.config.cjs`** для части процессов явно задано **`60000`** ms (например Orca, Moonshot); для Raydium/Meteora/PumpSwap при отсутствии ключа в env действует **тот же дефолт 60s** из скрипта.
- Тик дискавери paper/live стратегий может быть **10s** (`PAPER_DISCOVERY_INTERVAL_MS: '10000'` в ecosystem для pt1-* / live-oscar), но **новая цена в Postgres** для пары появляется не чаще, чем коллектор вносит строку в снимки (и при условии, что пара попадает в выборку источника).

**Вывод для спеки:** операционная эвристика заказчика — **«мы можем опаздывать до порядка минуты»** относительно **мгновенной** реакции ончейн-ботов — **согласуется с дефолтным cadence снимков ~60s**. Это **не жёсткая гарантия**: добавляются гранулярность тика, лаг DexScreener/Gecko, границы окна.  

**Параметр по умолчанию:** `T_pre = 60_000` ms (1 минута). Допускается конфиг **`T_pre_ms`** в env джобы; заказчик может задать **ровно 60000** как старт без «запаса», с возможностью поднять до **90_000–120_000** ms после A/B на recall.

## 4. Объём «70 + 30» и дробление ног (контекст, вне scope логики `dip_bot`)

Заказчик уточнил: **70% / 30% с задержкой** — это **вынужденная мера против удара по тонкому стакану / проскальзывания**, а не основной антибот-механизм. В перспективе возможно **до 5 ног** при крупном входе.

**Инвариант спеки W9.0:** продукт **`dip_bot`** **не обязан** менять сплит ног; он поставляет **данные** (теги, scores, профили выхода в фазе 2). Любое изменение задержек между ногами — **отдельная задача** на базе метрик и, при необходимости, корреляции с присутствием `dip_bot`.

## 5. Архитектура (высокий уровень)

```
JSONL open / таблица входов
        │
        ▼
 Джоба по расписанию (bootstrap часто → steady раз в неделю)
        │
        ├─► RPC: свопы в пре-окне по mint
        ├─► Парсинг покупателей + USD-оценка
        ├─► Эвристики повторяемости → кандидат dip_bot
        ├─► Кластеризация (опционально фаза 1.5)
        └─► UPSERT в Postgres: тег dip_bot, метаданные, связь кластер↔адрес

Стратегии / дашборд (позже): READ тегов для скоринга входа
```

**Запрещено:** дергать тяжёлый RPC на **каждом** тике дискавери стратегии в фазе 1.

## 6. Фаза 1 — Идентификация `dip_bot` (обязательный MVP)

### 6.1 Вход джобы

- **MVP — только Live Oscar:** якоря берутся **исключительно** из live-журнала по **`LIVE_TRADES_PATH`** (типично `data/live/pt1-oscar-live.jsonl`), события фактического **открытия позиции / покупки** с `strategyId` **`live-oscar`** (или эквивалент в контракте JSONL — зафиксировать в коде парсером). Бумажные **`pt1-*`** в MVP **не** кормят джобу (снижение RPC и соответствие операционному решению).
- Множество **якорных событий** за период: с watermark **`last_processed_anchor_ts`** (или offset по последней обработанной подписи / monotonic id), опционально ограничение «последние N дней» для первого прогона.
- Для каждого якоря: `mint`, `entry_ts`, `strategy_id`, при наличии **`pair_address`** и **program/DEX** — для сужения RPC-запросов.

### 6.2 Пре-окно

- `window_start = entry_ts - T_pre_ms`, `window_end = entry_ts` (**полуоткрытый интервал** `[start, end)` или согласованная политика границ — зафиксировать в коде один вариант).
- **`T_pre_ms` по умолчанию:** `60000` (см. §3).

### 6.3 Пост-окно (фаза 1)

- **По умолчанию:** **не анализировать** покупки после `entry_ts` (боты выходят долго и по-разному; дорого и шумно).
- Исключение: опциональный флаг **`DIP_BOT_PHASE1_INCLUDE_POST_MS=0`** (дефолт 0); если позже >0 — только для экспериментов, не для прод-гейта без отдельного решения.

### 6.4 Источник истины на чейне

- **Реализация v1 (продукт 1.11.96):** покупатели в пре-окне читаются из **Postgres `swaps`** (`side='buy'`, `base_mint`, `block_time`), уже собранных контурами ingest/backfill/sigseed — **0 billable RPC** на запуск джобы; если данных нет, якорь всё равно помечается обработанным с **0** покупателей (см. лог).
- **Расширение (опционально):** платный RPC (QuickNode) для пробелов покрытия — отдельный флаг и версия после замеров; парсер должен поддерживать те же семейства DEX, что и торговый контур.

### 6.5 Критерий присвоения тега адресу `dip_bot`

Параметризуемые пороги (env):

| Параметр | Назначение | Стартовое значение (рекомендация) |
|----------|------------|-----------------------------------|
| `DIP_BOT_MIN_HITS` | Минимум якорных событий, на которых адрес был **покупателем в пре-окне** | `3` |
| `DIP_BOT_MIN_USD_ONE_EVENT` | Игнорировать «пыль» | `50` USD за событие (регулируется) |
| `DIP_BOT_MIN_SUM_USD` | Альтернатива/дополнение: суммарный объём по всем хитам | опционально |

**Логика:** адрес попадает в кандидаты, если на **≥ DIP_BOT_MIN_HITS** разных `mint`/временах выполнено условие ранней покупки в пре-окне с объёмом ≥ порога.

**Снятие / оспаривание тега:** вне scope MVP; достаточно поля `confidence` и `last_verified_at`.

### 6.6 Запись в БД

- **Не удалять** и не переопределять широкий тег **`bot`**.
- Добавить узкую сущность **`dip_bot`**:
  - либо **новое значение enum / строка** в существующей таблице тегов (если схема `wallet_tags`/`tags` позволяет несколько тегов на кошелёк),
  - либо отдельная таблица **`dip_bot_wallets`** с полями ниже.

**Минимальные поля:**

- `wallet_pubkey` (text, PK или уникальный с `source`)
- `tag` = константа `dip_bot` (если универсальная таблица тегов)
- `first_seen_at`, `last_verified_at`
- `hit_count` (количество подтверждённых якорей)
- `total_estimated_usd` (суммарная оценка покупок в пре-окнах)
- `confidence` numeric [0,1] (простая функция от hit_count и стабильности)
- `cluster_id` nullable (FK на таблицу кластеров, см. §8)
- `notes` / `provenance` jsonb (последний batch id, версия парсера)

**Миграция:** новый SQL-файл в `src/core/db/migrations/` + запись в [`CHANGELOG`](../release/CHANGELOG.md) продукта по модели релиза.

## 7. Фаза 2 — Таксономия выхода (отложено, контракт на будущее)

**Триггер включения:** явный флаг продукта и бюджет RPC (например `DIP_BOT_EXIT_ANALYSIS_ENABLED=1`).

**Идея:** для адресов с уже присвоенным `dip_bot` периодически строить **профиль выхода**: время до первой продажи, долями, лестница и т.д., в окне **после их покупки** (параметризуемая глубина, напр. до 24h или до события).

**Выход:** таблица **`dip_bot_exit_profiles`** или jsonb-поле у `dip_bot_wallets`; потребитель — **офлайн** рекомендации по задержкам между ногами входа (не автомат без human review в первой версии).

## 8. Кластеризация (организатор)

**Цель:** ответить, **один ли хозяин** у нескольких адресов и отслеживать смену кошельков.

**Минимальная реализация (MVP):**

- Таблица **`dip_bot_clusters`** (`cluster_id`, `created_at`, `confidence`).
- Таблица связей **`dip_bot_cluster_members`** (`cluster_id`, `wallet_pubkey`, `joined_at`).
- Эвристика v1: **общий funding source** — разбор **нескольких** входящих транзакций на SOL/USDC для каждого адреса (лимит глубины и RPC calls — конфиг); кластеризация по **совпадению непосредственного funder** или по известному списку биржевых кошельков (опционально).
- Эвристика v2 (позже): совместная активность по времени на множестве якорей.

**Вывод для UI/стратегии:** `cluster_id` + число активных членов + дата последнего нового адреса в кластере.

## 9. RPC-бюджет и идемпотентность

- Каждый запуск джобы имеет **`batch_id`** (uuid); результаты помечаются для возможности **повторного прогона** без дублей (UPSERT по `wallet_pubkey` + обновление счётчиков).
- Лимиты: **`DIP_BOT_MAX_TX_FETCH_PER_ANCHOR`**, **`DIP_BOT_MAX_ANCHORS_PER_RUN`** — обязательны; при превышении — остановка с кодом ≠ 0 и алерт в лог.
- Логирование: суммарное число RPC credits **не обязательно** в MVP, но желательно считать **число `getTransaction` / `getSignaturesForAddress`** на run.

## 10. Интеграция со стратегией (после MVP данных)

**Фаза данных:** только Postgres + отчёты.

**Фаза потребления (отдельный PR):**

- Опциональный флаг стратегии **`PAPER_DIP_BOT_SCORE_ENABLED`** / live-эквивалент: перед записью `open` выполняется **дешёвый** запрос «есть ли известные `dip_bot` среди покупателей в последнем пре-окне» — только если для этого mint **уже есть закэшированный снимок** от джобы (materialized column или таблица **`dip_bot_anchor_scores`** по `(mint, entry_ts)`), **без** RPC в hot path.

Запрещено в первом релизе W9.0: блокировать вход без человеческого review, если метрики качества тега не измерены.

## 11. Операции и расписание (cron, RPC-окно, два режима)

### 11.1 Пользователь и окружение

- Процессы в **`/opt/solana-alpha`**, от имени **`salpha`** (как остальные cron Solana Alpha).
- Лог: **`data/logs/dip-bot-live-oscar.log`** (или согласованное имя в install-скрипте).
- Состояние watermark: файл **`data/state/dip_bot_live_oscar_watermark.json`** или строка в Postgres — **один** канон на среду.

### 11.2 Почему минута **02:25 UTC**

По [`deploy/RUNTIME.md`](../../../deploy/RUNTIME.md) и блоку **`install-detective-data-plane-salpha.sh`** на проде:

- Ежедневный тяжёлый коридор детектива: **03:10–04:35 UTC** (backfill, pilot, funding, bot-bucket, scam-farm, graph).
- Почасовые пики **sigseed**: минуты **:08** и **:18** UTC.

Слот **02:25 UTC** — **после** **02:08** / **02:18** и **до** **03:10**, без пересечения с **пн 06:15** (`unknown-dex:report`) и **вс 04:35** (`rpc-features:prune`). Фоновые **`*/3`** / **`*/15`** задачи остаются; отдельный **пиковый** batch по новому продукту сюда не накладывается сознательно.

### 11.3 Режим **A — начальный сбор (bootstrap)**

**Цель:** как можно быстрее накопить первые устойчивые **`dip_bot`** и кластеры, чтобы начать ими пользоваться (дашборд, ручной review, будущие гейты).

| Параметр | Значение |
|----------|----------|
| Cron (UTC) | **`25 2 * * *`** — **каждый день** в **02:25** |
| Поведение | Один вызов джобы: обработать новые якоря Live Oscar с watermark до **`DIP_BOT_MAX_ANCHORS_PER_RUN`** (жёсткий потолок за run, §9). |
| Допускается | В первые дни — **второй ручной** прогон вне cron при нулевом трафике Live Oscar (догон истории); не в hot path стратегии. |

**Выход из bootstrap** (любое из условий по решению оператора + зафиксировать в [`deploy/RUNTIME.md`](../../../deploy/RUNTIME.md) или внутреннем runbook):

- набрано ≥ **`DIP_BOT_BOOTSTRAP_MIN_TAGGED`** различных адресов с тегом **`dip_bot`** (рекомендация старт **`5`**), **или**
- прошло ≥ **`DIP_BOT_BOOTSTRAP_MAX_DAYS`** календарных дней с первого прогона (рекомендация **`14`**), **или**
- явная команда оператора **`DIP_BOT_BOOTSTRAP_COMPLETE=1`** в env на VPS / флаг в состоянии.

После выхода из bootstrap cron **заменяют** на режим B (ниже).

### 11.4 Режим **B — steady state (актуализация)**

**Цель:** держать базу **`dip_bot`** свежей при **низкой** стоимости RPC.

| Параметр | Значение |
|----------|----------|
| Cron (UTC) | **`25 2 * * 2`** — **каждый вторник** в **02:25** |
| Поведение | Инкремент по watermark за прошедшую неделю + переоценка топ-K уже помеченных адресов (опционально, в пределах того же **`DIP_BOT_MAX_ANCHORS_PER_RUN`** на «новые» якоря). |

### 11.5 Установка cron

- Идемпотентный скрипт **`scripts/cron/install-dip-bot-cron-salpha.sh`**, маркеры **`# DIP_BOT_CRON_BEGIN`** … **`# DIP_BOT_CRON_END`**, правка только этого блока в **`crontab -u salpha`**.
- В репозитории по умолчанию коммитить **вариант для режима B** (вторник weekly); в **`deploy/RUNTIME.md`** описать: при запуске продукта сначала вставить **ежедневную** строку **`25 2 * * *`**, после критериев §11.3 — заменить на **`25 2 * * 2`**.

### 11.6 Переменные окружения расписания (рекомендация)

| Переменная | Смысл |
|------------|--------|
| `DIP_BOT_ANCHOR_STRATEGY_IDS` | В MVP: **`live-oscar`** (единственное значение). Расширение списка — отдельная версия спеки. |
| `DIP_BOT_MAX_ANCHORS_PER_RUN` | Потолок якорей за один запуск (bootstrap и steady). |
| `DIP_BOT_BOOTSTRAP_MIN_TAGGED` | Порог адресов с **`dip_bot`** для выхода из bootstrap. |
| `DIP_BOT_BOOTSTRAP_MAX_DAYS` | Максимум дней daily cron перед принудительным переходом на weekly (или ручным решением). |

### 11.7 Деплой

- По [`NORM_UNIFIED_RELEASE_AND_RUNTIME.md`](../release/NORM_UNIFIED_RELEASE_AND_RUNTIME.md); миграции Postgres до первого запуска джобы на VPS.

## 12. Acceptance criteria (DoD)

1. Миграция(и) применяются локально и на staging; **`npm run typecheck`** зелёный.
2. На фикстурном наборе из **≥10** реальных якорных событий (без секретов в git) джoba завершается и создаёт **≥1** запись `dip_bot` при наличии синтетического или анонимизированного ввода — **или** документированный «пустой результат» при отсутствии совпадений.
3. Повторный прогон того же периода **не дублирует** строки тегов (идемпотентность).
4. Документ **обновлён** [`INDEX.md`](./INDEX.md) строкой на W9.0.
5. Запись в [`CHANGELOG.md`](../release/CHANGELOG.md) и bump [`VERSION`](../release/VERSION) — **только интегратором** по [`RELEASE_OPERATING_MODEL.md`](../release/RELEASE_OPERATING_MODEL.md).
