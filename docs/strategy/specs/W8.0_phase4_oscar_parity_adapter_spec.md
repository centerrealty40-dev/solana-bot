# Spec W8.0-p4 — Паритет Oscar: решения стратегии → adapter исполнения (live-oscar, Phase 4)

**Статус:** нормативный черновик для реализации (**до merge кода — ревью + bump `docs/strategy/release/VERSION` по [`RELEASE_OPERATING_MODEL.md`](../release/RELEASE_OPERATING_MODEL.md)**).  
**Продукт:** `solana-alpha`, процесс **`live-oscar`** (отдельный от **`pt1-oscar`**).  
**Зависит от:**

- [`W8.0_live_oscar_trading_bot.md`](./W8.0_live_oscar_trading_bot.md) — §6 паритет, §5 свежий quote, §7 JSONL;
- [`W8.0_phase1_live_jsonl_contract.md`](./W8.0_phase1_live_jsonl_contract.md) — `liveSchema: 1`, `intentId`, `execution_attempt` / `execution_result` / `execution_skip`;
- [`W8.0_phase3_wallet_simulate_spec.md`](./W8.0_phase3_wallet_simulate_spec.md) — lazy keypair, подпись, `qnCall` + `simulateTransaction` (**без** `sendTransaction`);
- Реализация paper Oscar: [`main.ts`](../../../src/papertrader/main.ts) (`discoveryTick`, таймеры), [`tracker.ts`](../../../src/papertrader/executor/tracker.ts), [`open.ts`](../../../src/papertrader/executor/open.ts), конфиг [`config.ts`](../../../src/papertrader/config.ts).

**Не входит в Phase 4:**

- Исполняемые гейты **§3.3–§3.4** родительской спеки (**риск / капитал / 2X**) — **Phase 5** (adapter остаётся вызываемым только после того, как Phase 5 обернёт его).
- **`sendTransaction`** и подтверждение в сети — **Phase 6**.
- Полный **replay / reconcile** live-журнала с ончейн-остатками — **Phase 7** (Phase 4 может заложить минимальное in-memory состояние для трекера).
- Стратегии **≠ Oscar (dip/post)** в live — вне scope, пока `LIVE_STRATEGY_PROFILE !== oscar`.

---

## 1. Цели Phase 4

1. **Один источник правды для логики «открыть / вести / закрыть»** относительно бумажного Oscar: те же входы (PG-снимки, интервалы, пороги), тот же **порядок гейтов** на входе, та же модель позиции (**`OpenTrade`**, DCA legs, ladder), что и у `papertrader` для профиля Oscar.
2. Вынести **исполнение** (Jupiter quote → swap build → подпись → симуляция по Phase 3) за **adapter**, чтобы ядро стратегии не содержало вызовов `sendTransaction` и не путало бумажный JSONL с live.
3. При **`LIVE_EXECUTION_MODE=simulate`** и успешной симуляции фиксировать пару **`execution_attempt` → `execution_result`** (`sim_ok` / `sim_err`) и достаточные поля для последующего replay (Phase 7).
4. При **`dry_run`** для live — не загружать keypair (**P3-I1**); поведение «на грани исполнения» согласовать с Phase 2–3 (quote/build без подписи или политика self-test — без регресса).
5. Добиться **измеримого** паритета: на одном наборе ENV и одной БД решение «пропустить кандидата» по гейту X совпадает с paper при том же коде гейта (см. §10).

## 2. Инварианты (нарушение — блок merge)

| ID | Инвариант |
|----|-----------|
| **P4-I1** | Процесс **`live-oscar`** **никогда** не вызывает **`appendEvent`** из [`papertrader/store-jsonl.ts`](../../../src/papertrader/store-jsonl.ts) и **никогда** не пишет в **`PAPER_TRADES_PATH`** / путь паритетного бумажного журнала (`LIVE_PARITY_PAPER_TRADES_PATH`). Читать бумажный журнал допускается **только read-only** и только если явно зафиксировано в §8 (по умолчанию — **не требуется**). |
| **P4-I2** | Любой **buy** (открытие позиции live), который доходит до Jupiter/simulate, проходит **тот же упорядоченный конвейер гейтов**, что и paper Oscar перед событием **`open`** в [`main.ts`](../../../src/papertrader/main.ts): safety → impulse (если вкл.) → live mcap best-effort → price verify (если вкл.) → sim-audit (если вкл.; unsigned в paper — в live не подменяет подписанный sim Phase 3). Исключения и ослабления **запрещены** без отдельной спеки и bump MINOR+. |
| **P4-I3** | Один swap intent → один **`intentId`** (**P3-I4** / W8.0-p1). Трекер и DCA не генерируют второй intent без нового UUID. |
| **P4-I4** | Параллельный **`pt1-oscar`** и **`live-oscar`** — **разные процессы**; модульные singleton papertrader в памяти **не разделяются** между процессами. Запрещено вводить общий глобальный mutable state на диске для «синхронизации решений» без отдельной спеки. |
| **P4-I5** | До Phase 6 в конфиге остаётся **fail-fast** при **`LIVE_STRATEGY_ENABLED=1`** и **`LIVE_EXECUTION_MODE=live`** (как сейчас в [`config.ts`](../../../src/live/config.ts)). |

## 3. Архитектура: adapter и граница модулей

### 3.1 Концепция

- **Strategy core (Oscar dip)** — существующие функции papertrader: discovery, построение **`OpenTrade`**, обновление позиций, решения трекера (**partial / full exit**, DCA).
- **Journal sink (live)** — только [`appendLiveJsonlEvent`](../../../src/live/store-jsonl.ts) + контракт W8.0-p1.
- **Execution adapter** — модуль `src/live/` (например `execution-adapter.ts`, `oscar-loop.ts`), который:
  - получает **каноническое намерение исполнения** (`LiveExecutionIntent`, см. §4);
  - для **`buy`**: вызывает цепочку Phase 2–3 (`liveBuyQuoteAndPrepareSnapshot` → sign → `liveSimulateSignedTransaction`);
  - для **`sell`** (подэтап §9): строит Jupiter swap token→SOL с **`userPublicKey`** кошелька, далее та же симуляция;
  - пишет **`execution_attempt` / `execution_result`** (и при необходимости **`execution_skip`** с кодами из W8.0-p1 §6).

### 3.2 Рефакторинг papertrader (минимально достаточный)

Допустимые паттерны реализации (выбрать **один** в PR и зафиксировать в CHANGELOG):

1. **Hooks / callbacks:** извлечь из `discoveryTick` (и позже из `trackerTick`) функцию вида `runOscarEntryPipeline(cfg, deps)`, где `deps.appendPaperOpen` / `deps.liveExecuteBuy` передаются снаружи; в **`live-oscar`** передаётся no-op для paper и реальный adapter для live.
2. **Параметр `executionProfile`:** один бинарь `papertrader` с режимом `live-oscar` — **не обязателен** в Phase 4 (родительская спека §12.1 остаётся открытой); текущий отдельный entrypoint **`live-oscar`** сохраняется.

**Запрещено:** копировать на 300+ строк дублирующий `discoveryTick` в `src/live/` без плана сходимости с paper (drift).

### 3.3 Конфигурация паритета ENV

- Родительская спека §3.1: дубли **`PAPER_*` → `LIVE_*`** или общий фрагмент **`LIVE_INHERIT_ENV_FILE`**. Phase 4 **обязан** документировать выбранный механизм в [`CHANGELOG.md`](../release/CHANGELOG.md) и в **`.env.example`**.
- Минимум: все ключи, влияющие на **решение** открытия (lane filters, safety, impulse, price verify, sim-audit, position size, slippage, интервалы discovery/track), доступны live-процессу с **теми же семантиками**, что у pt1-oscar.
- Опционально (**рекомендуется на канарейке**): **`LIVE_PARITY_ENV_SNAPSHOT=1`** — при старте одна строка в JSONL (`live_boot` или отдельное диагностическое событие) с **хэшем/перечнем** ключевых параметров (без секретов), чтобы сравнить с paper.

#### 3.3.1 Операционная модель «канарейка vs baseline» (**принято**)

- **`pt1-oscar` (paper):** зона частых мелких правок и экспериментов; отдельный профиль ENV / блок в PM2.
- **`live-oscar`:** только **baseline** — отдельный закреплённый набор `LIVE_*` (или наследование через **отдельный** фрагмент, который **не** является тем же файлом, что у канарейки paper).
- **Запрещено:** автоматически подключать один и тот же «живой» include / общий экспериментальный env-файл к **обоим** процессам без явного решения о **промоуте** (ручное копирование нужных ключей в baseline + запись в CHANGELOG при необходимости).

## 4. Контракт `LiveExecutionIntent` (логический)

Нормативные поля (точные типы — в коде):

| Поле | Описание |
|------|----------|
| `intentKind` | `'buy_open'` \| `'sell_partial'` \| `'sell_full'` \| `'dca_add'` (если DCA мапится на отдельный swap). |
| `mint` | SPL mint токена. |
| `side` | Для JSONL: **`buy`** \| **`sell`**. |
| `intendedUsd` | Нотация в USD, если определима; иначе согласовать с `intendedAmountAtomic`. |
| `intendedAmountAtomic` | Опц.; строка uint для входного актива в минимальных единицах. |
| `executionMode` | Копия `LIVE_EXECUTION_MODE`. |
| `strategyMeta` | Опц. сжатый контекст: `lane`, `source`, `dex`, ссылка на цену входа / причину выхода (**без** секретов). |
| `targetPriceUsd` | Опц.; для сверки §8.4 родительской спеки (**fill tolerance** реализуется в Phase 5–6 вместе с риском или явным подэтапом Phase 4c по решению исполнителя). |

Генерация **`intentId`**: [`newLiveIntentId`](../../../src/live/intent.ts) в момент фиксации намерения **после** прохождения всех гейтов §7.

## 5. Поведение по режимам

| `LIVE_EXECUTION_MODE` | Buy path (после гейтов) |
|----------------------|-------------------------|
| `dry_run` | Не загружать keypair. Quote/swap build по политике Phase 2; подпись и `simulateTransaction` **не** выполняются для «боевого» adapter-пути (допускается только изолированный self-test по флагам Phase 2–3). JSONL: при необходимости **`execution_skip`** с явным `reason`, без пары attempt/result если intent не создавался. |
| `simulate` | Keypair по **P3-I1**, реальный Jupiter swap body, подпись, **`liveSimulateSignedTransaction`**; **`execution_attempt`** обязателен перед результатом. |
| `live` | Недопустимо до Phase 6 (**P4-I5**). |

## 6. JSONL (аддитивно к `liveSchema: 1`)

Без bump **`liveSchema`**, если достаточно существующих видов:

- **`execution_attempt`**: заполнять `quoteSnapshot`, `targetPriceUsd` (если есть), `intendedUsd` / `intendedAmountAtomic`.
- **`execution_result`**: для успешной симуляции — `status: 'sim_ok'`, `simulated: true`, `unitsConsumed`; опционально **`executedPriceUsd`** и производные (если вычисляются из quote до Phase 7 reconcile).

Если потребуется явное событие «позиция live открыта только в симуляции» — новый **`kind`** вводится только с решением о **`liveSchema: 2`** или расширением контракта в W8.0-p1 (отдельный PR).

## 7. Объём релиза Phase 4 (buy, sell, DCA)

**Принято:** целевой **один** релиз Phase 4 включает полный контур Oscar для live: **открытие (buy)**, **DCA add**, **частичные и полные продажи (sell)** по решениям трекера — через общий execution adapter (Phase 3: подпись + `simulateTransaction`; без `sendTransaction` до Phase 6).

Логическая декомпозиция работ (ветки разработки), а не разные prod-cut без sell:

| Блок | Содержание |
|------|------------|
| **Вход** | **`buy_open`** после полного конвейера гейтов (**P4-I2**); in-memory **`open`** для live. |
| **DCA** | **`dca_add`** → отдельный swap intent; свежий quote обязателен (**L4** родительской спеки). |
| **Выход** | **`sell_partial` / `sell_full`** из [`tracker.ts`](../../../src/papertrader/executor/tracker.ts) (ladder, SL, trail, timeout, liq-watch и т.д.); Jupiter token→SOL; те же шаги подписи и симуляции. |

### 7.1 Дашборд PaperTrader2 (`/papertrader2`)

Нормативно для UI (реализация: [`dashboard-server.ts`](../../../scripts-tmp/dashboard-server.ts), [`dashboard-paper2.html`](../../../scripts-tmp/dashboard-paper2.html)):

| Элемент | Требование |
|---------|------------|
| Порядок колонок | Жёстко **слева направо**: **Live Oscar** (`strategyId` **`live-oscar`**) → **Paper Oscar** (`pt1-oscar`) → **Deep Runner** (`pt1-diprunner`) → **Dno** (`pt1-dno`). Порядок **не** зависит от PnL и **не** ротируется. |
| Источник колонки Live | JSONL по **`DASHBOARD_LIVE_OSCAR_JSONL`** (дефолт: каталог `live/` рядом с `PAPER2_DIR`, файл **`pt1-oscar-live.jsonl`**). Файл с тем же именем **исключается** из сканирования `PAPER2_DIR`, чтобы не было дубликата строки. |
| Константа порядка | Экспорт **`DASHBOARD_PANEL_ORDER`** в `dashboard-server.ts`; ответ **`GET /api/paper2`** всегда содержит ровно четыре слота (плейсхолдер с нулями, если журнал стратегии отсутствует). |
| Сводка PnL Live | До появления в live JSONL paper-совместимых позиций или явных полей PnL в **`execution_result`** колонка Live может показывать **нулевые** агрегаты PnL; счётчики heartbeat / исполнений за 1h — из событий `channel: live`. |

## 8. Опционально: read-only сверка с paper JSONL

Нормативно **не требуется**. Если включается флагом (**например `LIVE_PARITY_DIFF_INSTRUMENTAL=1`** — только имя для спеки, финальные имена в коде):

- только **сравнение** решений (pass/fail по mint+ts window) между процессами;
- никаких побочных эффектов на исполнение.

Иначе источник истины — **общий код гейтов**, а не журнал paper.

## 9. Зависимости от других модулей

- **Postgres / коллекторы:** только read-only, как у paper.
- **Pricing:** `refreshSolPrice`, `getSolUsd`, Jupiter quote — те же модули papertrader, если не нарушает **P4-I1**.
- **QN:** `SA_RPC_HTTP_URL` для simulate (**Phase 3**).

## 10. Acceptance criteria (измеримые)

- [ ] При **`LIVE_STRATEGY_ENABLED=0`** live-oscar **не** выполняет adapter-путь buy и **не** загружает keypair.
- [ ] При включённой стратегии и **`simulate`** для кандидата, прошедшего все гейты, в JSONL есть **`execution_attempt`** и ровно один терминальный **`execution_result`** с тем же **`intentId`**.
- [ ] Нет записей в бумажный журнал Oscar от процесса live (**P4-I1**).
- [ ] Юнит-тесты: мок adapter или мок Jupiter/`qnCall` — проверка порядка вызовов и кодов **`execution_skip`** при «сломанном» quote/build/sign/sim.
- [ ] **`npm run typecheck`** и релевантные тесты зелёные; **`CHANGELOG`** + bump **`VERSION`** (MINOR для Phase 4 рекомендуется).

---

## 11. Forbidden actions

- Открывать позицию live **до** прохождения конвейера гейтов входа (**P4-I2**) без отдельной явной спеки «канарейка отключения гейта» (не Phase 4).
- Использовать цену только из Postgres для решения о размере/выходе **без** свежего quote на пути исполнения (**нарушение L4** родительской спеки).
- Писать секреты или полный путь keypair с содержимым в JSONL.

---

## 12. Чеклист перехода Phase 4 → Phase 5

- [ ] Adapter buy/sell стабилен в `simulate`.
- [ ] Задокументированы метрики расхождений paper vs live (если включена диагностика §8).
- [ ] Готов обёртывающий слой для **§3.3–§3.4** без изменения семантики intent (**Phase 5**).

## 13. Принятые продуктовые решения (зафиксировано)

1. **Объём Phase 4:** в одном целевом релизе — **buy + sell (+ DCA)**; см. §7.
2. **ENV:** рекомендуемый паттерн — **два профиля** (канарейка paper vs baseline live), §3.3.1; точный механизм (`LIVE_*` дубликаты vs отдельный include только для live) — по выбору при внедрении, с записью в CHANGELOG.
3. **Дашборд:** четыре фиксированные колонки, Live первая слева; без зеркальных paper-событий в live JSONL на этом этапе — достаточно **`execution_*`** + heartbeat для базовой видимости; см. §7.1.
