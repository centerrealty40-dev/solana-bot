# W6.12 — S04: Заполнение `wallets.funding_source` для правил scam-farm-detective

**Тип:** normative implementation spec  
**Зависимости:** желательны минимальные **`money_flows`** или результат S02; может работать на частичных данных  
**Связанный обзор:** [`W6.12_OVERVIEW_detective_without_chain_firehose.md`](./W6.12_OVERVIEW_detective_without_chain_firehose.md)

---

## 1. Проблема

SQL-правила **`rug_cohort`** и **`orchestrated_split`** (`src/intel/scam-farm-detective/queries.ts`) используют **`wallets.funding_source`**:

- cohort группирует ранних покупателей по **общему фандеру**;
- split связывает пары buyer/seller через **совпадение funding_source** или через **`money_flows`**.

Без заполненного `funding_source` часть сигналов **молчит**, даже при наличии `swaps`.

## 2. Определение «фандера» (normative MVP)

Для адреса кошелька **`W`**:

**Primary definition (предпочтительно):** минимальный по времени вход **нативного SOL** на **`W`** из внешнего источника, зафиксированный в **`money_flows`**:

- актив **`SOL`** (или эквивалент согласованной строки в вашей схеме);
- `target_wallet = W`;
- `source_wallet` не в ignore-list (системные программы — исключить);
- выбрать строку с **минимальным `tx_time`** среди кандидатов за окно **`SA_FUNDING_LOOKBACK_DAYS`** (default **30**).

**Fallback:** если для `W` нет подходящих потоков, оставить `NULL` и пометить в логах метрикой `funding_miss`.

Запрещено в MVP: рекурсивный «полный граф» финансирования без лимита RPC.

## 3. Алгоритм бэкфилла

Периодичность: **не чаще 1×/сутки** на кошелёк (watermark в таблице или колонке `funding_source_updated_at` — добавить миграцией если отсутствует).

Шаги:

1. Выбрать batch адресов из **`wallets`** приоритетно тех, у кого `funding_source IS NULL` и есть хотя бы одна строка в **`money_flows`** за lookback.
2. Для каждого адреса вычислить фандера по §2 SQL-агрегатом (предпочтительно **один запрос на batch** с `DISTINCT ON` или оконной функцией), без RPC.
3. `UPDATE wallets SET funding_source = ?, funding_source_updated_at = now()` (имена колонок времени — по факту миграции).
4. Если **`money_flows` пусты**, но S02 включён — сначала дождаться прогона S02 для этой когорты (операторский порядок в S05).

### Опционально Phase 2 (не блокирует MVP)

Если flows отсутствуют, но есть только RPC: **один** узкий запрос истории (последние K подписей) исключительно для извлечения первого входа SOL — только под компонентом `wallet_backfill` и лимитами S01. Явно пометить как **optional** в задаче имплементации.

## 4. ENV

| Переменная | Назначение |
|------------|------------|
| `SA_FUNDING_BACKFILL_ENABLED` | `0/1` |
| `SA_FUNDING_LOOKBACK_DAYS` | окно поиска в `money_flows` |
| `SA_FUNDING_BATCH_SIZE` | размер batch SQL update |

## 5. Критерии приёмки

1. Миграция: при необходимости `funding_source_updated_at` + индекс для выборки `NULL`.
2. Команда `npm run wallet-funding:backfill` (или аналог) с `--dry-run`, печатающая сколько строк будет обновлено.
3. После прогона на стейджинге: выборочно `EXPLAIN` запросов детектива с участием `funding_source` выполняется без full scan таблиц (при наличии индексов на `money_flows.tx_time`, `money_flows.target_wallet`).
4. Документирован конфликт с privacy/legal — не выходит за рамки публичных chain данных (внутренняя заметка оператора).

## 6. Связь с Wallet Intel

Promote `wallets → entity_wallets` остаётся отдельным контуром ([`W6.9_IMPLEMENTATION_CHECKLIST.md`](./W6.9_IMPLEMENTATION_CHECKLIST.md)); данная спека **не заменяет** entity-слой, а закрывает **SQL-детектив** на canonical `wallets`.
