# W6.10 — Umbrella-тег `bot`, пайплайн Intel и закрытие пробелов таксономии

**Расположение:** `docs/Smart Lottery V2/` — см. [`README.md`](./README.md).  
**Статус:** normative spec (целевая архитектура; реализация по фазам).  
**Контекст продукта:** `solana-alpha`, ветка **`v2`**.

**Закрывает пробелы обсуждений:**

| Пробел | Как закрывает W6.10 |
|--------|---------------------|
| Нет единого класса «бот» для downstream-модуля | Вводится **`wallet_tags.tag = 'bot'`** (семантика Bot) с явными **слоями** и **`source`**. |
| Много узких тегов (`mev_bot`, `bot_farm_*`) без общего запроса | **Umbrella-правило** + сохранение узких тегов как доказательной базы. |
| Риск «снести все теги» vs «навечно запутаться» | Политика: **не массовый DELETE**; **версионирование** (`source`, `context.rule_set`); опционально view «текущее мнение». |
| Seed **`wallets`** не покрываются **`tagWallet`** | Обязательная стратегия **promote / кандидаты из `swaps`** (согласовано с W6.9 checklist). |
| Детектив / боты / торговля разъехались | Единый **порядок пайплайна** и точка **политики** ([`W6.9`](./W6.9_wallet_intel_detective_trading_spec.md)). |
| Наследие «speed-refresh» в RUNTIME без кода в дереве | Явное разделение: **speed = операционная очередь на `tagWallet`**; логика ботов в **`wallet-tagger` + W6.10**; восстановление cron — отдельная задача. |

**Связанные документы:** [`W6.9_wallet_intel_detective_trading_spec.md`](./W6.9_wallet_intel_detective_trading_spec.md), [`W6.9_IMPLEMENTATION_CHECKLIST.md`](./W6.9_IMPLEMENTATION_CHECKLIST.md), [`CONCEPT_DATA_PLANE_STAGING_TAGS_LIFECYCLE.md`](./CONCEPT_DATA_PLANE_STAGING_TAGS_LIFECYCLE.md), [`ROADMAP_SMART_MONEY_DATA_PLANE.md`](./ROADMAP_SMART_MONEY_DATA_PLANE.md).

## 1. Цели и не-цели

### 1.1 Цели

1. Выделить кошельки с **нечеловеческой / высокочастотной / координированной машинной** логикой (в смысле торгового риска, не юридического).
2. Пометить их **общим тегом `bot`** для запросов «все боты» и для **будущего модуля подклассов** (arb, sandwich, …).
3. **Не удалять** исторические теги; новые прогоны добавляют строки с новым **`source` / rule_set**.
4. Согласовать выход с **торговой политикой** W6.9 (`wallet_intel_decisions`, `mintDecision`).

### 1.2 Не-цели v0

- Идеальная классификация arb vs sandwich (отложено на **W6.11+** или отдельный модуль).
- Доказательность «бот в суде».
- Обязательная зависимость от Helius / отдельного платного RPC (см. W6.9 §8).

---

## 2. Принципы данных

1. **`wallet_tags`** — append-friendly журнал: `(wallet, tag, source)` уникальны в PK; **не стирать** прошлые строки ради «чистоты».
2. Каждый новый модуль пишет теги с **`source`** вида **`bot_umbrella_v0`**, **`bot_rule_high_swap_rate_v0`**, **`detector_w69`**, и т.д.
3. **`context` (jsonb/text)** обязан содержать минимум: **`rule_set`**, **`triggers`** (какие узкие теги или метрики сработали), при необходимости пороги.
4. **`primary_tag` в `entity_wallets`** — производное; изменение приоритета **без удаления** строк в `wallet_tags`. Отдельное решение: включать ли **`bot`** в [`PRIMARY_TAG_PRIORITY`](../../src/intel/wallet-tagger.ts) и на какой позиции (§6).

---

## 3. Архитектура модулей

```
  [facts] wallets, swaps, money_flows, tokens, entity_wallets
       │
       ▼
  [M1] wallet-tagger.tagWallet  ──► узкие теги (mev_bot, bot_farm_*, sniper, …)
       │
       ▼
  [M2] bot-umbrella + bot-sql   ──► wallet_tags.tag = 'bot' (several sources)
       │
       ▼
  [M3] scam-farm-detective      ──► scam_farm_candidates (+ optional Atlas)
       │
       ▼
  [M4] wallet-intel-policy      ──► wallet_intel_decisions, mintDecision (W6.9)
```

**Торговый бот / стратегия** читает преимущественно **M4**; отладка и будущие модели — **M1–M2** и сырые факты.

## 4. Модуль M2 — Umbrella `bot`

### 4.1 Слой A (наследование от узких тегов)

Если у кошелька есть **любой** из триггеров из таблицы ниже — upsert (идемпотентно по PK) строку:

- **`tag`** = `'bot'`
- **`source`** = `'bot_umbrella_v0'` (bump версии при смене набора триггеров)
- **`confidence`** = функция от max(confidence триггеров) или фиксированное 80
- **`context`** = JSON: `{ "rule_set": "2026-05-01", "trigger_tags": ["mev_bot", ...] }`

**Триггеры v0 (рекомендуемый минимум):**

| Узкий тег | Включать в v0? |
|-----------|----------------|
| `mev_bot` | да |
| `bot_farm_boss` | да |
| `bot_farm_distributor` | да |
| `sniper` | **опционально** (много ложных с человеческими терминалами); если да — сниженный confidence umbrella |

**Не дублировать** узкие теги: umbrella только **добавляет** `bot`.

### 4.2 Слой B (SQL / метрики без узкого тега)

Цель — ловить машинные паттерны до того, как сработает узкое правило, или если **`tagWallet`** не отрабатывал (нет профиля).

Примеры эвристик (пороги — через env, дефолты зафиксировать в коде при реализации):

- **`BOT_RULE_SWAP_BURST`:** за последние 24h число строк в **`swaps`** по `wallet` ≥ `BOT_SWAP_COUNT_24H_MIN` и медианный интервал между сделками ≤ `BOT_MEDIAN_GAP_SEC_MAX`.
- **`BOT_RULE_MANY_MINTS`:** за 24h ≥ `BOT_DISTINCT_MINTS_MIN` при среднем `amount_usd` ≤ `BOT_AVG_TRADE_USD_MAX`.
- **`BOT_RULE_FLOW_FANOUT`:** при наличии **`money_flows`** — экстремальный fan-out/fan-in (пороги согласовать с существующими эвристиками `bot_farm_*`, чтобы не дублировать 1:1 — либо ослабить порог здесь, либо оставить только «пропуски»).

Каждое правило — отдельный **`source`** (`bot_rule_swap_burst_v0`, …) и своя строка **`tag='bot'`** (разные source в PK — несколько строк `bot` допустимы и полезны для аудита).

### 4.3 CLI / npm

Рекомендуемые имена (согласовать с реализацией):

- **`npm run intel:bot-bucket`** — полный M2 (A+B) с лимитами батча.
- Флаги: `--dry-run`, `--since-hours`, `--wallet <pk>` (spot), `--rule-set bot_v0`.

Зависимость: для слоя A желательно предварительно прогнать **`tagWallet`** по тем же кошелькам (отдельная команда или шаг pipeline §8).

## 5. Связь с W6.9 (политика торговли)

1. Модуль **wallet-intel-policy** при материализации **`wallet_intel_decisions`** должен учитывать:
   - наличие **`wallet_tags` где tag='bot'** → решение уровня **`IGNORE_FOR_SIGNAL`** или отдельный код **`BOT_CLASS`** (зафиксировать в rule_set W6.9);  
   - **не путать** с **`BLOCK_TRADE`** (ферма / scam-linked): бот ≠ обязательно запрет монеты, это политика продукта — но **рекомендуемый дефолт**: ранний покупатель с **`bot`** **не считается** «smart-кворумом»; для denylist фермы по-прежнему **scam-farm + теги scam_***.

2. Явная матрица (зафиксировать в W6.9 при следующем bump rule_set):

| Сигнал | Влияние на mint (рекомендация) |
|--------|--------------------------------|
| `BLOCK_TRADE` / scam cluster | **NO_TRADE** |
| `bot` | Не участвует в smart-кворуме; опционально флаг «высокая доля ботов среди ранних покупателей» |
| `IGNORE_FOR_SIGNAL` (CEX, terminal, …) | Исключить из наблюдателей раннего входа |

## 6. `primary_tag` и конфликты

**Пробел:** umbrella `bot` может конфликтовать с `smart_money` в интерпретации.

**Решение spec:**

- Либо **не включать** `bot` в приоритет как «сильнейший» до калибровки,
- Либо вставить **`bot`** в [`PRIMARY_TAG_PRIORITY`](../../src/intel/wallet-tagger.ts) **ниже** scam и **выше** `retail` только после измерения ложных срабатываний.

До этого продакт может использовать **только `wallet_intel_decisions`** как финальный слой для бота.

---

## 7. Покрытие `wallets` без `entity_wallets`

**Пробел:** [`tagWallet`](../../src/intel/wallet-tagger.ts) использует профиль **`entity_wallets`** и **`money_flows`/`swaps`**.

**Обязательные меры (одна или комбинация):**

1. **Promote:** минимальный upsert в **`entity_wallets`** для адресов из **`wallets`** (seed), чтобы `tagWallet` мог отработать.
2. **Кандидаты из `swaps`:** отдельный SQL список активных pubkey → батч **`tagWallet`**.
3. **Слой B M2** может пометить **`bot`** только по **`swaps`** даже без полного профиля Atlas — но тогда **`confidence`** ниже и **`context.note = 'swaps_only'`**.

Это **дублирует и уточняет** W6.9 checklist §1.4 — при реализации W6.10 не ослаблять.

---

## 8. Рекомендуемый порядок пайплайна (production-oriented)

1. **`wallet-intel:doctor`** (W6.9) — наличие свежих `swaps` / sanity counts.
2. **Promote / swap-candidates** (§7) при необходимости.
3. **`tagWallet`** батчами (M1).
4. **`intel:bot-bucket`** (M2).
5. **`scam-farm:detect`** (M3, сначала dry-run при новых правилах).
6. **`wallet-intel:policy`** (M4).

Частота: бот-слой может обгонять scam-farm по расписанию чаще (например каждые 6–24h), политика — после обоих.

---

## 9. Наследие RUNTIME: speed-refresh / speed_tag_queue

В [`deploy/RUNTIME.md`](../../deploy/RUNTIME.md) описаны **`intel:speed-refresh`** и очередь **`speed_tag_queue`** как способ **приоритизировать высокоактивные кошельки** для вызова **`tagWallet`**.

**Если исходников нет в текущем дереве репозитория:** W6.10 **не требует** восстановления speed-conveyor для существования M2; достаточно SQL-кандидатов по **`swaps`**. Восстановление speed — **отдельная задача** «parity с VPS/runbook».

## 10. Будущий модуль подклассов ботов (задел)

- Вход: подмножество **`wallet_tags` WHERE tag='bot'`**, joined с **`swaps`/`money_flows`** по окну.
- Выход: новые теги **`bot_arb`**, **`bot_sandwich_candidate`**, … **или** одна таблица **`wallet_bot_subtypes`** с версией.
- Версионирование: **`source = bot_subtype_v1`**, не перезапись истории.
