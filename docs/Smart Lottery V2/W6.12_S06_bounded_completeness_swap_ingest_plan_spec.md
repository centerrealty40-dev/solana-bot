## 1. Проблема и позиционирование

**Проблема:** поток **`sa-parser` / chain-wide stream** даёт высокий либо неконтролируемый расход QuickNode и **не является необходимым** для целей детектива по таблицам `swaps`, `money_flows`, если определить **ограниченную вселенную адресов U** и добиться **монотонного исчерпания очереди** в горизонте **T** (например 30 суток) при фиксированном дневном бюджете.

**Позиция S06:** формализовать **budget-first bounded completeness**: не полнота всей сети Solana, а **гарантированная обработка каждого элемента U в пределах очереди и бюджета**, с измеримыми метриками сходимости.

**Негативная цель (non-goals S06):**

- Обещать «все свапы Solana за месяц» без индексатора или без расширения U на весь чейн.
- Включать без отдельной спеки неограниченный **Raydium/Orca/Jupiter** декод в `swaps` (сейчас normative путь pump.fun через существующий декодер).
- Заменять S01 глобальным «ручным» учётом кредитов вне ledger.

## 2. Определения

| Термин | Определение |
|--------|-------------|
| **U (universe)** | Множество адресов, для которых продукт **обязуется** накапливать факты в `swaps`/`money_flows` через контура S02+S03 в пределах конфигурации. Normative минимум v1: адреса в **`wallet_backfill_queue`** в статусах `pending` или `done` (повторные проходы для углубления по watermark допускаются). Расширения U оформляются отдельным патчем спеки или дочерним параграфом. |
| **Drain rate** | Ожидаемое число адресов очереди, переводимых из `running`→`done` (или эквивалентное завершение попытки) за сутки при текущих env cron и отсутствии `QN_GLOBAL_DAY_CAP`. |
| **Enqueue rate** | Число новых строк, попадающих в очередь за сутки (типично из `wallets` через `--enqueue-from-wallets=N`). |
| **Сходимость очереди** | Инвариант: среднее **`pending_count` не растёт без верхней границы** при стабильном бюджете; доля адресов с устаревшим `last_run_at` уменьшается на горизонте T. |
| **Bounded completeness по U** | Для каждого `a ∈ U` существует конечная последовательность прогонов S02 при неизменном бюджете, такая что либо адрес помечен **устойчивым терминальным статусом** (`done` с исчерпанием страниц подписей в пределах лимитов, или `error` с явным сообщением), либо очередь признана оператором застрявшей по §10.3. |

## 3. Нормативные инварианты

**I-S06-1 (бюджет первичен):** любой новый код или cron не может обходить **S01**; при `QN_GLOBAL_DAY_CAP` прогон завершается корректно (как в S02), состояние очереди консистентно.

**I-S06-2 (идемпотентность):** повторные прогоны только добавляют новые факты через существующие уникальные ключи (`swaps`, `money_flows`); не раздувают дубликаты.

**I-S06-3 (монотонность watermark):** для каждого адреса курсор подписей (`sig_cursor` / поле в `wallet_backfill_queue` по факту схемы) не откатывается назад без явной операторской команды и записи в CHANGELOG (исключение — баг-фикс миграции).

**I-S06-4 (разделение контуров):** **S03 (sigseed)** и **S02 (wallet-centric)** остаются независимыми источниками строк `swaps` с разным `source`; планирование бюджета суммирует их вклад в [`sa-qn-budget-check`](../../package.json) / W6.13.

**I-S06-5 (align с детективом):** параметры **`SCAM_FARM_LOOKBACK_DAYS`**, **`BOT_LAYER_B_SINCE_HOURS`** и горизонт **T** согласуются операторски: ingest должен **опережать** или **совпадать** с окном SQL-правил (см. §8).

## 4. Алгоритм контроллера enqueue (gate)

**Цель:** не допускать ситуации, когда **enqueue_rate ≫ drain_rate** без явного решения о расширении бюджета или горизонта T.

**Normative v1 (реализация одним из способов, эквивалентность по эффекту):**

1. **Измерение раз в сутки** (после ночного прогона backfill): выполнить SQL из §7.1, получить `pending_count`, `median_age_pending_hours`, `done_last_24h` (или эквивалент).
2. **Условие gate:** если  
   `pending_count > SA_BACKFILL_ENQUEUE_GATE_PENDING_MAX`  
   (новый env, см. §9), то **следующий** запланированный **enqueue** (`wallet-backfill:run --enqueue-from-wallets=…`) **пропускается** или **`N` уменьшается** до  
   `max(0, SA_BACKFILL_ENQUEUE_SOFT_CAP - pending_count)`  
   (новый env), пока оператор не снимет блокировку вручную (env flag или таблица-флаг — на усмотрение реализации, по умолчанию env).

**Замечание:** текущий cron в `install-detective-data-plane-salpha.sh` выполняет enqueue и pilot независимо; **внедрение gate** — отдельная задача реализации (скрипт-обёртка или расширение `wallet-backfill-run.ts`), фиксируется в CHANGELOG при появлении кода.

**Normative minimum без кода (до gate):** оператор **вручную** сравнивает §7.1 раз в неделю и при `pending_count` выше порога **уменьшает `N`** в cron или отключает строку enqueue на время разгрузки (запись в runbook / тикет).

## 5. Оценка кредитов и горизонта T

**Верхняя оценка кредитов за один прогон S02 (billable RPC):**

\[
C_{\text{run}} \leq N_{\text{wallets}} \times (S_{\text{sig}} + T_{\text{tx}}) \times C_{\text{rpc}}
\]

где:

- \(N_{\text{wallets}}\) = `SA_BACKFILL_MAX_WALLETS_PER_RUN` (или значение из cron для pilot),
- \(S_{\text{sig}}\) = `SA_BACKFILL_SIG_PAGES_MAX` (страницы `getSignaturesForAddress`; **1 страница ≥ 1** billable вызов в учёте),
- \(T_{\text{tx}}\) = `SA_BACKFILL_MAX_TX_PER_WALLET` (`getTransaction`),
- \(C_{\text{rpc}}\) = `QUICKNODE_CREDITS_PER_SOLANA_RPC` (дефолт **30**).

**Оценка дней до обработки текущего хвоста очереди** (песочница планирования):

\[
T_{\text{drain}} \approx \left\lceil \frac{P}{R_{\text{day}}} \right\rceil,\quad
R_{\text{day}} = \sum_{\text{слоты cron}} N_{\text{wallets,slot}}
\]

где \(P\) = `pending_count`, сумма по всем backfill-слотам за сутки.

**Согласование с W6.13:** суммарный вклад S02+S03 за сутки не должен систематически превышать выделенный компонентный потолок (`SA_BACKFILL_MAX_CREDITS_PER_DAY` и фактический расход ledger); иначе увеличить T или снизить N/S/T параметры.

## 6. Расписание (cron) и PM2

**Normative для VPS solana-alpha:**

- SSOT строк cron — **`scripts/cron/install-detective-data-plane-salpha.sh`**; после изменения — идемпотентная переустановка от root, пользователь **`salpha`**.
- **Не нормативно** держать расходящиеся копии cron вне репозитория без записи в CHANGELOG.

**Рекомендуемая структура дня (пример, уже близка к репозиторию):**

1. Enqueue ограниченной партии из `wallets` (утро UTC).
2. Один или два прогона **pilot/run** с явными env в строке cron (ширина полосы).
3. Отдельно: sigseed (hourly), scam-farm, intel-bot-bucket с **`BOT_LAYER_B_SINCE_HOURS`** согласованным с §8.

**PM2:** оркестратор **`sa-wallet-orchestrator`** пополняет `wallets`; он **не** пишет в `swaps` — это нормально для модели S06.

## 7. Метрики и наблюдаемость

### 7.1 Обязательный еженедельный SQL-пакет (оператор)

Выполнить на прод-DSN (значения интерпретировать в динамике недели):

```sql
-- Глубина очереди
SELECT status, count(*) AS n
FROM wallet_backfill_queue
GROUP BY status;

-- Свежесть ingest swaps по источнику (копия логики scripts/diag/swaps-pipeline-diagnostics.sql)
SELECT coalesce(source, '(null)') AS source,
       count(*) FILTER (WHERE block_time > now() - interval '24 hours') AS rows_24h,
       count(*) FILTER (WHERE block_time > now() - interval '7 days') AS rows_7d,
       max(block_time) AS newest_swap_chain_time
FROM swaps
GROUP BY source
ORDER BY rows_7d DESC;
```

**Критерий тревоги:** `sa-parser` при включённом стриме должен иметь ненулевой `rows_24h`; если стрим **намеренно выключен**, нулевой `rows_24h` для `sa-parser` **ожидаем**, а рост `wallet_backfill` / `sigseed` обязателю подлежит проверке.

### 7.2 Логи прогона

Структурированный JSON итога прогона `wallet-backfill` (см. S02 §6) сохраняется в файлах cron (`data/logs/…`). Оператор архивирует **последнюю строку JSON** за день для учёта `swapsInserted`, `txFetched`.

## 8. Согласование окон: scam-farm, bot-bucket, U

| Потребитель | Параметр окна | Требование к ingest |
|-------------|---------------|---------------------|
| scam-farm phase A | `SCAM_FARM_LOOKBACK_DAYS` (S05) | Наличие строк в `swaps`/`money_flows` с **`block_time` / `tx_time`** внутри окна для представительной доли U. |
| intel-bot-bucket layer B | `BOT_LAYER_B_SINCE_HOURS` | Должен быть **≥** типичного окна наблюдения детектива или согласован с оператором; при разреженных данных **увеличивать** окно (как в cron **168h**), а не бесконечно поднимать RPC без gate. |

**Normative:** при планировании T=30 суток **`BOT_LAYER_B_SINCE_HOURS` ≥ 168** считается допустимым дефолтом для ночного bot-bucket до появления стабильного дневного потока `swaps`.

## 9. ENV-контракт (расширение к S02)

| Переменная | Назначение | Дефолт v1 S06 |
|------------|------------|----------------|
| `SA_BACKFILL_ENQUEUE_GATE_PENDING_MAX` | Если `pending_count` выше — не выполнять полный enqueue или снизить N | **пусто** = gate выключен до реализации; рекомендуемое стартовое **1500** после калибровки |
| `SA_BACKFILL_ENQUEUE_SOFT_CAP` | Верхняя целевая глубина очереди после enqueue | **пусто**; рекомендуемое **2000** |
| `SA_BACKFILL_ENQUEUE_BATCH_DEFAULT` | Замена литерала `N` в cron при переходе на обёртку | **500** (как текущий enqueue) |

Уже существующие ключи S02/S03/S01 **не переопределяются** данной спекой.

Публикация в **`.env.example`**: при появлении кода gate — добавить строки с комментарием «W6.12 S06».

## 10. Критерии приёмки

### 10.1 Документ и навигация

1. Файл S06 присутствует в репозитории по пути настоящего документа.
2. В [`W6.12_OVERVIEW_detective_without_chain_firehose.md`](./W6.12_OVERVIEW_detective_without_chain_firehose.md) в таблице §3 добавлена строка **S06**.
3. В [`README.md`](./README.md) каталога Smart Lottery V2 есть ссылка на S06 (оглавление или блок W6.12).

### 10.2 Операционные (после реализации gate — отдельный PR)

1. При `pending_count > SA_BACKFILL_ENQUEUE_GATE_PENDING_MAX` очередной enqueue **не увеличивает** `pending_count` более чем на `SA_BACKFILL_ENQUEUE_SOFT_CAP` относительно измерения до прогона (или enqueue пропущен — по выбранной ветке реализации).
2. За 4 последовательные недели при стабильном бюджете **медиана `pending_count` не имеет тренда роста** > 10%/неделю без одновременного роста `enqueue` (SQL отчёт оператора).

### 10.3 Застрявшая очередь

Если адрес в статусе `error` > **7** суток с одним и тем же `error_message`, оператор открывает разбор (RPC, лимиты, адресная аномалия). Это **не** блокер semver, но блокер объявления «month complete».
