# W6.12 — S05: Операционная готовность scam-farm-detective после budget-first ingest

**Тип:** normative operator + acceptance spec  
**Зависимости:** S01–S04 реализованы или частично (минимальный набор указан ниже)  
**Связанный обзор:** [`W6.12_OVERVIEW_detective_without_chain_firehose.md`](./W6.12_OVERVIEW_detective_without_chain_firehose.md)

---

## 1. Цель

Зафиксировать **минимальный состав данных и порядок cron/PM2**, при которых **`npm run scam-farm:detect`** даёт осмысленный результат **без** chain-wide stream/parser, и как оператор проверяет здоровье контура.

## 2. Минимальные данные для правил фазы A

| Правило детектива | Требование к данным |
|-------------------|---------------------|
| `sync_fund` | Достаточное покрытие **`money_flows`** за `SCAM_FARM_LOOKBACK_DAYS` по SOL-переводам между различными кошельками |
| `rug_cohort` | **`swaps`** ранних покупок по минтам из множества rug + **`wallets.funding_source`** заполнен (S04) |
| `orchestrated_split` | **`swaps`** buy/sell пары + **`money_flows`** или совпадение **funding_source** |

**Минимальный v1 запуск:** хотя бы **`money_flows`** (S02) + **`swaps`** (S02 и/или S03) за согласованное окно.

## 3. Согласование lookback

- **`SCAM_FARM_LOOKBACK_DAYS`** (env детектива, default 14) не должен превышать **фактическую глубину** данных, которую обеспечивают S02/S03 за последние дни без провалов.
- Если nightly backfill даёт только последние **48h** активности на кошелёк, держать **`SCAM_FARM_LOOKBACK_DAYS` ≤ 2** до расширения лимитов — **или** наращивать `SA_BACKFILL_*` (операторское решение, зафиксировать в RUNTIME).

Нормативное требование: при деплое оператор заполняет таблицу «ожидаемая глубина vs env» в одном месте ([`deploy/RUNTIME.md`](../../deploy/RUNTIME.md) § appendix W6.12).

## 4. Порядок задач (cron / PM2)

Рекомендуемый порядок в пределах UTC-суток:

1. **Оркестратор кошельков** (непрерывно) — пополняет `wallets`.
2. **Sigseed enqueue/parse** по расписанию (уже в RUNTIME) — узкий `swaps`.
3. **Wallet backfill** (S02) — `money_flows` + доп. `swaps`.
4. **Funding backfill** (S04) — после того как появились flows.
5. **`scam-farm:detect`** — не раньше пункта 3 для `sync_fund`; для cohort/split желательно после пункта 4.

Частота detect: сохранить существующий **`run-scam-farm.sh`** (например 1×/сутки) до появления отдельной спеки на intra-day.

## 5. Критерии готовности (go / no-go)

**GO**, если одновременно:

1. `npm run wallet-intel:doctor` (или эквивалентная проверка) показывает **ненулевые** `swaps_last_24h` и **`money_flows_last_24h`** при включённом S02/S03 (допускается порог «> 0» без целевого объёма на старте).
2. Доля **`wallets` с непустым `funding_source`** в свежей когорте ≥ порога оператора (начать с **отчёта**, порог задать после первой недели).
3. Глобальный ledger S01 не стабильно упирается в cap за **< 12h** суток — иначе перераспределить компоненты или поднять лимиты по бизнес-решению.

**NO-GO:** запускать массовый detect «вхолостую» без проверки doctor — допускается только `DRY_RUN` для отладки SQL.

## 8. Риски

- Ложные отрицательные: узкий backfill пропускает старые связи → смягчать порогами detect или увеличивать приоритетные кошельки.
- Ложные положительные: неполные flows искажают sync_fund → мониторить качество данных до включения auto-confirm в Atlas.
