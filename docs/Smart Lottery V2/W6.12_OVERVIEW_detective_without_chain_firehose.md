# W6.12 — Обзор: scam-farm detective без chain-wide stream/parser

**Статус:** normative (целевая архитектура для реализации поэтапно)  
**Продукт:** solana-alpha (схема `solana_alpha`, см. платформенный `DB_TOPOLOGY`)  
**Аудитория:** агенты реализации, оператор VPS  

---

## 1. Проблема

Полный контур **`sa-stream` + `sa-parser`** (подписка на поток событий программы и массовый `getTransaction`) при биллинге QuickNode **не укладывается** в разумный дневной бюджет (порядка **~1.5M кредитов/сутки**): наблюдалась трата **до ~1M кредитов в час** без жёсткого ограничения охвата.

При этом **фаза A** `scam-farm-detective` (`src/intel/scam-farm-detective/queries.ts`) — **чистый SQL** по таблицам **`swaps`**, **`money_flows`**, **`tokens`**, **`wallets`**. Ей **не нужен** круглосуточный firehose всей сети; нужен **актуальный срез фактов** за **`SCAM_FARM_LOOKBACK_DAYS`** (по умолчанию 14) с **контролируемой стоимостью** добычи этих фактов.

## 2. Целевой продукт (что создаём в итоге)

**Ingest-плоскость «budget-first»:**

1. Единый учёт и потолок **billable RPC / кредитов QuickNode** для всех «тяжёлых» джоб (оркестратор кошельков, бэкфилл по кошелькам, sigseed worker, wallet trace — см. дочерние спеки).
2. **Wallet-centric nightly (или реже) backfill:** выборка ограниченного числа адресов из **`wallets`** → ограниченная глубина подписей / транзакций → нормализованные строки в **`swaps`** и **`money_flows`** (без подписки на всю сеть).
3. **Mint-scoped sigseed** как **второй** узкий источник **`swaps`** (очередь + лимиты), без stream.
4. Заполнение **`wallets.funding_source`** минимально достаточным способом для правил **rug_cohort** и **orchestrated_split**.
5. Операционная схема: после стабильного притока фактов — **`npm run scam-farm:detect`** по cron; метрики/alerts через существующие doctor/отчёты.

**Негативная цель (non-goals данного трека):**

- Не возвращать **безлимитный** stream/parser на прод без отдельного бюджетного контура и капов.
- Не требовать **$1k+/мес** RPC как обязательное условие v1 этого трека.
