**Расположение:** `docs/Smart Lottery V2/` — см. [`README.md`](./README.md).  
**Статус:** normative — зафиксированные решения оператора + требования к UI; реализация в кодовой базе **Solana Alpha** (ветка продукта по [`RELEASE_OPERATING_MODEL.md`](../strategy/release/RELEASE_OPERATING_MODEL.md)); контур **Smart Lottery** — согласованная «дочерняя» линия того же продукта.

## 1. Решения оператора (зафиксировано)

### 1.1 Ранние покупатели и нагрузка (вопросы A1–A2)

- Окно **`T_early`** и объём выборок должны подбираться так, чтобы **не перегружать Postgres и RPC**. Конкретные числа — **после профилирования**; все пороги выносим в **ENV** с консервативными дефолтами.
- Множество ранних покупателей по умолчанию: до **`K = 1000`** уникальных покупателей (или эквивалент в пределах окна времени — см. §3). Если нагрузка высокая — снижаем **`K`** только через ENV, без правки кода.

### 1.2 Политика блокировки и входа (A3–A5)

- **Любой адрес с достаточно уверенным «скамерным» сигналом** → **отказ от покупки** (блок монеты при участии такого адреса в раннем наборе). Приоритет **`BLOCK_TRADE`** над smart-tier — **да**.
- Если **нет отрицательного сигнала** — **допускается вход** (в рамках прочих гейтов стратегии). Это поведение должно быть **переключаемым через ENV** (в т.ч. режим «строгий»: без положительного покрытия данных не заходить).
- Списки **`IGNORE_FOR_SIGNAL`** (исключения из smart-кворума без блока) **в v1 не вводим** — вопрос B7 не применяется; при необходимости добавим отдельным патчем спеки.

### 1.3 Smart-tier (B8)

- Вводим **`SMART_TIER_*`**. На первом этапе все промотируемые адреса попадают в **`SMART_TIER_A`**; расширение уровней — последующими версиями правил.

### 1.4 Umbrella «bot» (B9)

- Подтверждено: ранний покупатель с umbrella-меткой **`bot`** **не считается** несущим smart-кворум, но **сам по себе не блокирует** монету без scam-сигнала (см. W6.10).

### 1.5 Хранилище решений (C10–C12)

- Таблицы **`wallet_intel_decisions`** и **`wallet_intel_runs`** — как в [`W6.9_IMPLEMENTATION_CHECKLIST.md`](./W6.9_IMPLEMENTATION_CHECKLIST.md) §2.
- **`mint_intel_cache`** — **отложено**.
- Поле версии правил: **`rule_set_version`** = **semver продукта** из [`docs/strategy/release/VERSION`](../strategy/release/VERSION) (единый SSOT для оператора).

### 1.6 Scam-farm и Atlas (E16–E19)

- **Все промежуточные и итоговые метрики** детектива обязательны к экспонированию (состав — §5).
- Режим **`WRITE_ATLAS`** — **включать сразу** (не ждать длительного DRY_RUN), при соблюдении лимитов SQL/RPC из ENV.
- Аналитика аномалий — **максимально полная на дашборде** (§6), включая разрезы по правилам и прогонам.

### 1.7 Частота и лимиты (F20–F22)

- Целевая частота интел-пайплайна: **ориентир ~1 раз в 4 часа**; точное расписание и лимиты **`SCAM_FARM_*` / батчи** — после тестовых прогонов по кредитам и нагрузке на БД.
- Нужна **единая точка входа** **`wallet-intel:pipeline`** (или эквивалент), вызывающая этапы в согласованном порядке (tagger → scam-farm → policy и т.д. по W6.10).

## 2. Пояснения к ранее неясным вопросам

### 2.1 D13 — «MINT_DECISION и swaps»

**Смысл:** без строк в **`swaps`** система не видит факт «кто купил mint рано». В классической формулировке W6.9 это даёт **`NEED_MORE_DATA`**.

**Ваше решение:** при отсутствии **негативного** сигнала вход разрешён. В спецификации реализации это означает режим **permissive** (ENV):  
`WALLET_INTEL_REQUIRE_SWAP_COVERAGE=0` — отсутствие свежих `swaps` **не** является автоматическим запретом;  
`WALLET_INTEL_REQUIRE_SWAP_COVERAGE=1` — строгий режим W6.9 для отладки.

### 2.2 D14 — `money_flows`

**Смысл:** часть SQL **scam-farm-detective** (синхронное funding и т.д.) читает **`money_flows`**. Если таблица пуста или редка, эти правила дают мало кандидатов — это **не ошибка**, а сигнал для дашборда «низкое покрытие потоков».

### 2.3 D15 — `wallets` vs `entity_wallets`

**Смысл:** **`wallet-tagger.tagAtlas`** исторически опирается на **`entity_wallets`**. Поток **W6.8** пишет в **`wallets`**, где строки могут **не дублироваться** в Atlas.

**Рекомендация спеки v1:** батч **`promote`** (минимальный upsert профиля) из **`wallets` → `entity_wallets`** для адресов, попавших в очередь тегирования, **либо** явный режим теггера по таблице **`wallets`** — выбор фиксируется одним ENV **`WALLET_TAGGER_INPUT_MODE=`** `entity_only|wallets_promote|...`. Дефолт настраивается при первом merge после метрик покрытия.

### 2.4 B7 — что означало «IGNORE_FOR_SIGNAL»

Это был класс адресов (CEX hot и т.п.), которые **не блокируют монету**, но **не участвуют** как «smart» в кворуме. Вы решили **не использовать** такие исключения в v1 — раздел можно не реализовывать до отдельного решения.

### 2.5 E17 — «общие метрики прогона»

Для однозначности закрещаем минимальный набор **итогов прогона** scam-farm (дашборд + лог):

| Метрика | Описание |
|---------|-----------|
| `candidates_new` | Новые строки в **`scam_farm_candidates`** за прогон |
| `candidates_open` | Открытые кандидаты по статусу (если применимо схеме) |
| `wallets_touched` | Уникальные адреса, затронутые правилами |
| `atlas_writes` | Записи в Atlas (`wrote_to_atlas`, кластера/теги) за прогон |
| `by_rule` | Счётчики срабатываний **по каждому имени правила/SQL-фазы** |
| `duration_ms` | Длительность прогона |

Промежуточные метрики — те же разрезы **во время** этапов (до финального persist), чтобы отлавливать узкие места.

## 3. Параметры, обязательные к выносу в ENV

| Переменная | Назначение |
|------------|------------|
| `WALLET_INTEL_EARLY_BUYERS_K` | Макс. число ранних покупателей (дефолт **1000**, снижать при нагрузке) |
| `WALLET_INTEL_T_EARLY_MINUTES` | Окно «ранности» сделок по mint (подбор под RPC/БД) |
| `WALLET_INTEL_REQUIRE_SWAP_COVERAGE` | `0` — permissive (нет негатива → можно входить при прочих гейтах); `1` — строгий W6.9 |
| `WALLET_INTEL_ALLOW_ENTRY_WITHOUT_NEGATIVE` | Явный флаг разрешения входа без негатива (дублирует смысл permissive, если нужна гранулярность) |
| `WALLET_TAGGER_INPUT_MODE` | Источник адресов для тегера (см. §2.3) |
| `SCAM_FARM_*` / существующие лимиты | Как сейчас + новые лимиты батчей после нагрузочных тестов |
| `WALLET_INTEL_PIPELINE_INTERVAL_HINT` | Документируемый ориентир cron (напр. 4h) — сам cron в `deploy/RUNTIME.md` |

## 4. Классификация Wallet Tags и Primary Tag (предложение к согласованию)

**Primary tag** — один денормализованный тег на кошелёк (`entity_wallets.primary_tag`), выбирается **строгим приоритетом** из множества текущих тегов. Реализация уже задаёт порядок в **`PRIMARY_TAG_PRIORITY`** ([`src/intel/wallet-tagger.ts`](../../src/intel/wallet-tagger.ts)).

### 4.1 Слои таксономии (логические классы)

| Класс | Назначение | Примеры тегов из кода |
|-------|------------|------------------------|
| **P0 — Стоп-сигнал для торговли** | Организаторы схем, скам-операции, выводы | `scam_operator`, `scam_proxy`, `scam_treasury`, `scam_payout` |
| **P1 — Инфраструктура / не персональный alpha** | CEX, терминалы, крупные потоки без вердикта «скам» | `cex_hot_wallet`, `terminal_distributor`, `terminal_user`, `gas_distributor` |
| **P2 — Бот-сети и MEV** | Координированные боты, фермы | `bot_farm_boss`, `bot_farm_distributor`, `mev_bot`, umbrella **`bot`** (W6.10) |
| **P3 — Поведенческие** | Снайпинг, мем-флип, киты | `sniper`, `meme_flipper`, `whale` |
| **P4 — Позитив для smart-path** | Кандидаты smart money | `smart_money`, `insider`, `rotation_node`, `lp_provider` |
| **P5 — Базовые** | Розница, неактив | `retail`, `inactive` |

### 4.2 Маппинг на торговую политику (v1)

| Условие | Исход для mint / адреса |
|---------|-------------------------|
| Любой тег из **P0** на раннем покупателе | **BLOCK_TRADE** / отказ от покупки |
| Umbrella **`bot`** или теги **P2** на раннем покупателе | **Не учитывать** в smart-кворуме; **не блокировать** без P0 |
| Теги **P4** + отсутствие P0/P2-блока | **`SMART_TIER_A`** (после отдельных порогов уверенности, если появятся) |
| Только **P5** или данные скудны | Разрешение входа при **отсутствии негатива** (ENV) |

Вы просили «всё, что связано со скамерами и организаторами — блокировать»: к **организаторам** для v1 относим явно **P0** и сильные сигналы **scam-farm** (**`scam_farm_candidates`** + записи в Atlas), пока не появится отдельный тег `organizer` — тогда добавим его в P0 с bump semver правил.

**Ваш ревью:** подтвердить или скорректировать список P0/P2 и необходимость поднять `farm_linked` / теги из scam-farm в P0 при появлении в **`wallet_tags`**.

- По **каждому правилу / фазе SQL** (как в [`src/intel/scam-farm-detective/queries.ts`](../../src/intel/scam-farm-detective/queries.ts)): счётчики срабатываний, отклонённые строки, время выполнения.

## 6. Дашборд `https://etonne-moi.com/SmartLottery`

### 6.1 Контекст развёртывания

- DNS и хостинг домена описаны в продукте **Solana Alpha**; реализация страницы и API — из этой кодовой базы. Линия **Smart Lottery** — согласованная ветка/контур того же продукта.
- Доступ: **существующая** схема **логин/пароль** на сайте.

### 6.2 Обновление данных

- Частота автообновления UI **как у текущего дашборда торговых стратегий** (тот же интервал polling / SSE / reload — унифицировать с существующим фронтом).

### 6.3 Агрегаты по времени

- Основной операторский такт для сводок: **каждые 60 минут**, с привязкой к **наименее загруженному** cron-слоту на сервере (зафиксировать время в `deploy/RUNTIME.md` после выбора).

### 6.4 Источники кошельков

- Отображение прироста уникальных кошельков — по **нормализованной матрице**: строки = комбинации **`collector_id` × источник/pool-type × при необходимости `seed_lane`**, столбцы = метрики за окно **1h** и кумулятив.

### 6.5 Обязательные виджеты

1. **База данных:** ключевые таблицы (`wallets`, `swaps`, `money_flows`, `wallet_tags`, `entity_wallets`, `scam_farm_candidates`, …) — **последняя активность**, объёмы прироста за час / всего, опционально размер (если не дорого по запросу).
2. **Коллекторы:** все процессы, связанные с **поиском кошельков** — для каждого: последний успех, ошибки, **сколько новых уникальных кошельков за час** и за прогон, **разрез по источнику и по пулу** (насколько позволяет `metadata`/логи).
3. **Анализаторы:** scam-farm + теггер + политика (когда появится) — статус последнего прогона, §5 метрики.
4. **Scam-farm:** блок **«последний час»** и блок **«всего / накопление»** для оценки скорости роста базы кандидатов и Atlas.
5. **Теги за час:** число **новых присвоений** (или затронутых кошельков) по категориям:
   - боты (включая umbrella bot и узкие `bot_farm_*`, `mev_bot` — уточнить группировку в UI),
   - скамеры (P0),
   - организаторы (как только выделены тегом/кластером — до этого прокси через scam-farm + P0),
   - treasury (напр. `scam_treasury` и связанные),
   - **всего по всем тегам** и **кумулятивно уникальные теги / распределение**.
6. **Аномалии:** выделение всплесков/провалов (пороги через ENV после калибровки).

## 7. Критерии приёмки (черновик)

1. Все параметры §3 доступны через ENV и задокументированы в `.env.example`.
2. Запуск **`npm run wallet-intel:pipeline`** (имя финализировать в PR) выполняет цепочку без ручного порядка вызовов.
3. Дашборд `/SmartLottery` после авторизации показывает §6.5 без ручного SQL.
4. Semver продукта пробрасывается в **`wallet_intel_decisions.rule_set_version`** / Run record.
