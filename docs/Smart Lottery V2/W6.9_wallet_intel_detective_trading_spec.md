# W6.9 — Wallet Intel Detective (торговый контур): архитектура и спецификация

**Расположение:** `docs/Smart Lottery V2/` — см. [`README.md`](./README.md).  
**Статус:** normative spec (целевая архитектура; реализация по фазам).  
**Контекст продукта:** `solana-alpha`, ветка **`v2`**.  
**Связь:** продолжает [`ROADMAP_SMART_MONEY_DATA_PLANE.md`](./ROADMAP_SMART_MONEY_DATA_PLANE.md), [`CONCEPT_DATA_PLANE_STAGING_TAGS_LIFECYCLE.md`](./CONCEPT_DATA_PLANE_STAGING_TAGS_LIFECYCLE.md), использует **`wallets`** из W6.7/W6.8 и существующий код **`src/intel/scam-farm-detective/`**, **`src/intel/wallet-tagger.ts`**, схему Atlas в [`src/core/db/schema.ts`](../../src/core/db/schema.ts).

---

## 1. Цель (что строим и зачем)

Построить **детектив минимально достаточной строгости для торговли**, не для юридической доказательности:

1. **Denylist / красный флаг:** если ранний покупатель нового mint связан со **схемой фермы / оператором** с достаточной уверенностью → **монета не покупается** (избегание rug-пулов и координированных разводов).
2. **Foundation для smart money:** из того же контура данных формируется основа для **allowlist / скоринга** кошельков, на которых позже строится сигнал «смотреть покупку новой монеты».

**Не цели первой версии:** вердикты «скамер в суде», 100% покрытие сети, идеальная кластеризация всех пользователей.

---

## 2. Принципы

| Принцип | Формулировка |
|--------|----------------|
| **Trading-first precision** | Для denylist допустимы **ложные запреты** легче, чем ложные разрешения перед рагом; для smart — наоборот, консервативный promote. |
| **Два независимых выхода** | **BLOCK_TRADE** и **SMART_CANDIDATE** — разные правила и пороги; пересечение обрабатывается политикой приоритета (§7). |
| **Версионирование** | Каждое решение привязано к **`rule_set_id` / semver правил**, **`detector_run_id`**, **`as_of`**; возможен откат правил без потери сырых фактов. |
| **Разделение фактов и вердиктов** | Факты (ребра потоков, сделки, seed provenance) хранятся отдельно от **policy outcome** (blocked / promoted / unknown). |
| **Не «чистим БД» удалением** | Ретейл и боты остаются в данных; меняется **tier / использование в сигнале**, а не обязательное удаление строк. |

## 3. Архитектура (слои)

```
┌─────────────────────────────────────────────────────────────────┐
│ L0  Источники: Gecko seed (W6.8), Dex/collector swaps (существ.), │
│     опционально ончейн backfill (RPC / Helius-enhanced APIs)      │
└───────────────────────────────┬─────────────────────────────────┘
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│ L1  Canonical facts                                             │
│     • wallets (+ metadata provenance из коллекторов)               │
│     • swaps (факты сделок по mint/wallet/time)                     │
│     • money_flows (направленные переводы SOL/SPL — когда есть)   │
│     • tokens / blacklist поля (когда есть)                       │
└───────────────────────────────┬─────────────────────────────────┘
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│ L2  Обогащение и классификация (батчи)                           │
│     • wallet-tagger: эвристики → wallet_tags / primary_tag       │
│     • scam-farm-detective SQL rules → scam_farm_candidates       │
│     • (новое) wallet_intel_policy: материализация BLOCK/SMART    │
└───────────────────────────────┬─────────────────────────────────┘
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│ L3  Графовые метрики (опционально фаза 2)                         │
│     • подграф по окну времени + компоненты / роли узлов           │
│     • признаки: fan-in/out, bundle funding, co-mint touch       │
└───────────────────────────────┬─────────────────────────────────┘
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│ L4  Trading surfaces (для стратегии «новые монеты»)               │
│     • mint_early_buyers ∩ blocked_wallets → NO_TRADE             │
│     • mint_early_buyers ∩ smart_tier → сигнал к рассмотрению    │
└─────────────────────────────────────────────────────────────────┘
```

**Критический путь:** L1 должен содержать минимум **`swaps` или эквивалент** по ранним покупкателям mint и/или **`money_flows`** для связей funding. Только **`wallets`** из Gecko-сидера **недостаточно** для denylist по «ранним покупателям» — нужен факт покупки и время.

## 4. Состояние на старт («через неделю много новых кошельков»)

### 4.1 Что уже даёт оркестратор W6.8

- Массовое пополнение **`wallets`** с **`metadata`** (`seed_lane`, `job_type`, `gecko_multi_seed`, …).
- Это **хороший входной граф** для «кого дообогатить» и «кого проверить funding», но **не заменяет** таблицы сделок.

### 4.2 Минимальный набор данных, чтобы детектив начал работать

| Данные | Назначение |
|--------|------------|
| **`wallets.address`** | Идентификация; связка с тегами и кластерами. |
| **`swaps`** (wallet, base_mint, side, block_time) | Ранние покупатели mint; поведенческие признаки. |
| **`money_flows`** | Связь buyer↔seller, синхронное funding, графовые эвристики. |
| **`tokens`** (blacklist, liquidity_usd, first_seen_at — по мере наличия) | Якоря «rug-like» для SQL-детектива. |
| **`entity_wallets` / `wallet_tags` / `wallet_clusters`** | Уже поддерживаются записью из scam-farm-atlas при включённом режиме. |

**Спека-требование:** до включения торгового denylist по mint выполнить **ингест или бэктилл** минимум **`swaps`** для наблюдаемых mint (хотя бы из ваших коллекторов), иначе слой «ранний покупатель» пустой.

## 5. Политики выхода детектива

### 5.1 Классы решений по кошельку

| Код | Смысл для торговли |
|-----|---------------------|
| **`BLOCK_TRADE`** | Участие этого адреса в раннем наборе mint → **запрет входа** (или жёсткий штраф до отказа). |
| **`SMART_TIER_*`** | Уровень доверия для последующего кворума (не обязательно бинарный). |
| **`IGNORE_FOR_SIGNAL`** | Адрес не считается ни smart, ни farm для сигнала (CEX hot, терминал-пеймастер, известный сервис). |
| **`UNKNOWN`** | Недостаточно фактов; стратегия по умолчанию **не полагается** на такой адрес как на smart. |

### 5.2 Политика при конфликте

Зафиксировать в конфиге (рекомендуемый дефолт):

- Если **`BLOCK_TRADE`** по адресу или по **`cluster_id`** → **приоритет над SMART**, монета **не покупается**.
- Если только **`UNKNOWN`** среди ранних покупателей → поведение стратегии: **нет позитивного сигнала** (не эквивалент «скама»).

## 6. Модули реализации (маппинг на код)

### 6.1 Существующие компоненты (расширять, не дублировать вслепую)

| Компонент | Роль |
|-----------|------|
| **`src/intel/scam-farm-detective/`** | SQL-эвристики → **`scam_farm_candidates`**; при **`WRITE_ATLAS`** → кластера и теги. |
| **`src/intel/wallet-tagger.ts`** | Поведенческие теги; приоритет **`primary_tag`**. |
| **`src/core/db/schema.ts`** | `wallets`, `swaps`, `money_flows`, `wallet_tags`, `wallet_clusters`, `scam_farm_candidates`. |

### 6.2 Новый модуль (рекомендуемое имя в репозитории)

**`wallet-intel-policy`** (или подпакет в `src/intel/`):

- Читает факты + кандидатов + теги.
- Материализует **торговые поверхности**:
  - таблица или view **`wallet_intel_decisions`** (wallet, decision, score, reasons, `rule_set_version`, `computed_at`), **или**
  - таблица **`mint_risk_flags`** (mint, flag, evidence_ref).
- Идемпотентные upsert по `(wallet, rule_set_version)` или `(mint, flag_type, version)`.

Детали схемы оставить на этап миграции (не блокировать утверждение архитектуры).

### 6.3 Графовый слой (фаза 2)

- Включается, когда **объём рёбер** и ложные срабатывания SQL перестают устраивать.
- Вход: **`money_flows`** (+ опционально ребра co-trade из `swaps`).
- Выход: **`wallet_graph_metrics`** / `component_id` для усиления BLOCK/SMART.

## 7. Спецификация торгового правила «новый mint»

### 7.1 Входы

- **`mint`**, окно **`T_early`** (например первые 10–30 минут или первые **K** уникальных покупателей).
- Множество **`early_buyers`** из **`swaps`** (`side = buy`), после фильтра **`IGNORE_FOR_SIGNAL`**.

### 7.2 Выход

- **`MINT_DECISION`:** `NO_TRADE | ALLOW_SCAN | NEED_MORE_DATA`
  - **`NO_TRADE`:** ∃ buyer ∈ **`BLOCK_TRADE`** или buyer ∈ **cluster** с **`BLOCK_TRADE`** (если кластеризация включена).
  - **`ALLOW_SCAN`:** нет BLOCK и выполнены условия стратегии по smart-кворуму (вне scope этой спеки).
  - **`NEED_MORE_DATA`:** нет покрытия `swaps`/flows в окне — **не трактовать как безопасный**.

### 7.3 Версионирование правил

- Каждый прогон логирует **`rule_set_version`**, **`inputs_hash`** (версия источников данных), **`counts`** (сколько адресов классифицировано).

## 8. Helius (отдельный платный RPC): нужен ли и когда

### 8.1 Функциональная необходимость

**Helius не обязателен для принятия архитектуры детектива.** Он становится полезен, когда:

- нужно **быстро и полно** наполнить **`money_flows` / сырые legs** для адресов из **`wallets`** (backfill истории);
- нужны **enhanced transaction APIs** / webhooks, чтобы снизить число сырых RPC-вызовов и ошибки парсинга;
- QuickNode **перегружен** торговыми процессами и вы хотите **изолировать** аналитический burn.

### 8.2 Экономика (~80M кредитов/месяц)

- **Не ставить цель «израсходовать всё».** Цель — **полнота фактов под denylist** при заданном SLA и потолке затрат на 1 actionable решение.
- Перед покупкой плана зафиксировать **бюджеты:**
  - **`HELIUS_MONTHLY_CREDIT_CAP`** (hard),
  - **`MAX_CREDITS_PER_NEW_WALLET`** / **`MAX_CREDITS_PER_MINT_RISK_CHECK`**,
  - **ежедневный лимит** и очередь приоритетов (сначала адреса из ранних покупателей и кандидаты фермы).

### 8.3 Рекомендуемая стратегия провайдеров

| Этап | Провайдер |
|------|-----------|
| **Phase 0** | Только данные из существующих коллекторов + QuickNode там, где уже есть контур. |
| **Phase 1** | Подключить Helius **точечно** для очереди **обогащения** seed-кошельков (лимиты, отдельный ключ). |
| **Phase 2** | Расширить на streaming/webhooks при росте объёма mint. |

**Вывод по покупке:** заказывать Helius **имеет смысл параллельно с появлением очереди обогащения и метрик расхода**, а не «наперёд» без пайплайна записи в `money_flows`/`swaps`.

## 9. План внедрения (выравнивание на «неделю данных»)

### Фаза 0 (0–3 дня): контракты

- Утвердить **`rule_set_version` v0.1**, политику конфликтов §5.2, окно **`T_early`** §7.
- Проверить наличие потока в **`swaps`** для целевых mint; если нет — спека на минимальный ingest (отдельная задача, зависимость W6.9).

### Фаза 1 (параллельно накоплению wallets): детектив на фактах

- Запуск **`scam-farm-detective`** в **`DRY_RUN`** / ограниченном **`WRITE_ATLAS`** по runbook.
- Запуск **`wallet-tagger`** батчами по новым адресам из **`wallets`**.
- Материализация **`wallet_intel_decisions`** с консервативными порогами BLOCK.

### Фаза 2: торговая поверхность

- Реализовать **`MINT_DECISION`** §7 как чистую функцию от таблиц + версии правил.
- Дашборд/лог: доля `NO_TRADE`, `NEED_MORE_DATA`, причины.

### Фаза 3: опционально Helius + граф

- Очередь backfill → метрики полноты → решение о масштабе Helius.

## 10. SLI / контроль качества

| Метрика | Зачем |
|---------|--------|
| **Coverage:** доля ранних покупателей mint с известным tier | Не торговать в слепую |
| **BLOCK precision proxy:** доля `NO_TRADE` с последующим резким падением ликвидности (по вашим данным) | Калибровка aggressiveness |
| **BLOCK harm:** доля отказов от mint с последующим «хорошим» исходом (упущенная прибыль) | Цена ложных запретов |
| **Latency:** время от появления mint до `MINT_DECISION` | Жизнеспособность для новых монет |
| **RPC cost per decision** | Окупаемость Helius |

---

## 11. Границы и запреты

1. Не смешивать **юридические формулировки** с торговыми флагами в одном поле без версии контекста.
2. Не удалять **`wallets`** только из-за «мусора»; использовать tier и исключения из сигнала.
3. Изменения **семантики тегов** — только с bump **`rule_set_version`** и записью в changelog модуля.
