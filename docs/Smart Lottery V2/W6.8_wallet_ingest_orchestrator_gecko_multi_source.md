# W6.8 — Оркестратор пополнения `wallets` (Gecko multi‑source, без DexScreener)

**Расположение:** `docs/Smart Lottery V2/` — см. [`README.md`](./README.md).  
**Статус:** normative spec; **реализация v0.1:** `scripts-tmp/sa-wallet-orchestrator.mjs` + `wallet-orchestrator-lib.mjs`, `npm run sa-wallet-orchestrator`, PM2 **`sa-wallet-orchestrator`** в `ecosystem.config.cjs`.  
**Контекст продукта:** `solana-alpha`, ветка **`v2`**.  
**Связь с W6.7:** развивает идею **`sa-grws-collector`**: Gecko → фильтр DEX → QuickNode JSON-RPC → **`wallets`**, расширяя до **нескольких источников** и **единого оркестратора** с глобальными лимитами и недельной калибровкой весов.

---

## 1. Цель

Ввести **один параллельный торговым ботам** процесс **коллектора‑оркестратора**, который:

1. **Не использует DexScreener** (в отличие от торговых `*-collector.mjs`).
2. Использует **только Gecko Terminal (HTTP)** для обнаружения пулов/ликвидности по выбранным DEX‑lane и **QuickNode (JSON-RPC)** для извлечения pubkey и записи в **`wallets`**.
3. Соблюдает **глобальные** ограничения:
   - минутный ориентир Gecko Public API (**~30 HTTP‑вызовов/мин** на ключ; целевой троттлинг с запасом, напр. **≤28/мин** суммарно по всем lane);
   - дневной бюджет QuickNode (**ориентир 1 500 000 кредитов/сутки**, пересчёт в billable RPC через **`QUICKNODE_CREDITS_PER_SOLANA_RPC`**).
4. **Распределяет** HTTP и RPC между источниками по **весам** (не равномерно): учитывается доминирование экосистемы **Pump / PumpSwap** в потоке новых токенов относительно Raydium / Meteora / Orca / Moonshot.
5. Раз в **неделю** пересматривает веса по **фактической эффективности** (`новые строки wallets / потраченные кредиты`), не допуская монополии одного lane на весь бюджет и не «голодания» узких lane.

Торговые коллекторы (DexScreener + их расписание) **не изменяются** этой спека; пересечение с ними допускается **только** через общую таблицу **`wallets`** (и согласованный контракт `metadata`).

---

## 2. Границы и запреты

1. **Не модифицировать** скрипты и env торговых коллекторов (`raydium-collector.mjs`, `orca-collector.mjs`, …) в рамках W6.8.
2. **Не вызывать DexScreener** из оркестратора.
3. **Не писать** в торговые таблицы (`swaps`, очереди ботов и т.п.) — только **`wallets`** (+ опционально служебные таблицы метрик оркестратора, если введены явно отдельной миграцией).
4. **Не параллелить** два тяжёлых RPC‑набора по одному и тому же lane в одном процессе без явной очереди (один оркестратор — последовательное выполнение jobs или явный mutex на RPC‑бюджет).
5. **Не пушить в `main`**; изменения кода — через **`v2`** по [`RELEASE_OPERATING_MODEL.md`](../strategy/release/RELEASE_OPERATING_MODEL.md).

---

## 3. Архитектурное решение: один процесс‑оркестратор

**Зафиксировано для W6.8:** один долгоживущий процесс (**daemon**) с внутренним **планировщиком задач** (cron‑подобные слоты UTC) и **общими** компонентами:

- HTTP‑клиент Gecko с **глобальным** меж‑lane троттлингом (интервал между **любыми** исходящими запросами Gecko).
- Счётчик billable JSON-RPC и оценка кредитов QuickNode — **единый дневной файл состояния** (аналогично `sa-grws-budget-state.json`, но с разбиением по lane по желанию).
- Модуль извлечения pubkey из `getTransaction` / контракт `metadata` — общий для всех lane (как в W6.7 §6.3 / PI‑6), с различием **`metadata.source`** или **`metadata.seed_lane`**.

**Допустимый вариант реализации:** supervisor запускает **`node orchestrator.mjs --once --job=...`** из systemd/pm2 cron — эквивалентно, если **между запусками** сохраняется общий state Gecko/QN и нет гонок двух процессов одновременно.

---

## 4. Источники (lane) и идентификаторы

Минимальный набор lane первой очереди (расширяется только через дополнение к этой спеки):

| `lane_id`     | Описание | Gecko |
|---------------|----------|--------|
| `pumpswap`    | Экосистема Pump / PumpSwap (высокий вес) | network `solana`, фильтр DEX id / эвристики по контракту Gecko API |
| `raydium`     | Raydium AMM / CLMM / связанные id | то же |
| `meteora`     | Meteora (DLMM и др. по доступным спискам) | то же |
| `orca`        | Orca Whirlpool | то же |
| `moonshot`    | Moonshot | то же |

Каждый job логирует: `lane_id`, `job_type` (`new_pools` \| `trending_pools` \| `extended` \| `daily_deep`), `gecko_pages`, `rpc_cap_tick`, результат (`wallets_inserted`, `wallets_unique`, `rpc_billable`, `gecko_http`).

---

## 5. Расписание задач (базовое правило)

Для **каждого** lane задаются **типовые** интервалы (пример из обсуждения; точные cron выражения — в конфиге):

| `job_type`      | Периодичность | Назначение |
|-----------------|---------------|------------|
| `new_pools`     | каждые **1 ч** | основной поток «свежих» пулов |
| `trending_pools`| каждые **6 ч** | иная выборка, снижение пересечения с `new_pools` |
| `extended`      | каждые **12 ч** | больше страниц / вторичный endpoint при наличии |
| `daily_deep`    | раз в **24 ч** | максимальная глубина страниц редко |

**Фазовый сдвиг между lane:** чтобы не класть все HTTP в одну минуту, каждому lane назначается смещение  
`phase_minutes = lane_index × 9` (или эквивалент в секундах) относительно «ровного» часа; при **N** lane индекс фиксируется в конфиге.

**Асимметрия Gecko‑глубины:** lane **`pumpswap`** имеет право на **+1…+2 страницы** за цикл относительно прочих при том же `job_type`, отражая больший поток новых монет (см. §7 — веса согласованы с этим).

---

## 6. Глобальные лимиты и анти‑hogging RPC

### 6.1 Gecko

- Перед **каждым** HTTP‑запросом Gecko вызывается **глобальный** throttle (целевой CPM, напр. **28**).
- Учёт **`gecko_http_calls_day`** в state‑файле (UTC‑сутки); опциональный soft‑cap **`SA_ORCH_MAX_GECKO_HTTP_PER_DAY`**.
- При приближении к минутному лимиту провайдера — backoff + лог предупреждения.

### 6.2 QuickNode

- Дневной потолок кредитов: **`SA_ORCH_MAX_QUICKNODE_CREDITS_PER_DAY`** (дефолт согласовать с **`SA_GRWS_MAX_QUICKNODE_CREDITS_PER_DAY`** или заменить единым ключом для оркестратора).
- Перед каждым `rpcCall` — проверка остатка дня и **лимита на job** (ниже).
- Оценка кредитов: `rpc_billable × QUICKNODE_CREDITS_PER_SOLANA_RPC`.

### 6.3 Анти‑«углубился в пустой пул»

Зафиксировать в реализации:

1. **`max_rpc_per_pool_per_job`** — верхняя граница RPC на один пул в рамках одного job.
2. **`early_exit_new_wallet_rate`** — если за первые **K** RPC доля новых pubkey (относительно уже известных в БД) ниже порога, lane переключается на следующий пул **без** исчерпания глубины.
3. **Ротация пулов:** не опрашивать один и тот же `pool_address` в каждом часовом слоте подряд, если последние **M** проходов дали нулевые вставки (конфигурируемый TTL исключения).

## 7. Веса lane и доли дневного бюджета RPC

### 7.1 Стартовые веса (эвристика рынка, не ончейн‑аудит)

Публичные обзоры (порядок величин) связывают доминирование **Pump.fun / PumpSwap** с **большинством новых запусков токенов** на Solana и высокой долей связанной DEX‑активности; **Raydium** — второй полюс (AMM + LaunchLab); **Meteora**, **Orca**, **Moonshot** — меньший вклад в «массовый выпуск». Поэтому **равномерное** распределение не используется.

**Стартовое распределение дневного RPC‑бюджета оркестратора** (после выделения резерва):

| Lane       | Доля дневного RPC‑бюджета | Комментарий |
|------------|---------------------------|-------------|
| `pumpswap` | **50%**                   | максимальный приоритет по потоку новых монет |
| `raydium`  | **22%**                   | второй полюс ликвидности/лаунчей |
| `meteora`  | **11%**                   | DLMM‑слой |
| `orca`     | **10%**                   | whirlpool‑слой |
| `moonshot` | **7%**                    | узкий лаунчпад |
| **Резерв** | **5%**                    | ретраи, всплески, перераспределение по weekly |

Сумма без резерва **100%** назначаемого пула; **резерв** хранится отдельно и не участвует в quotidianых job до решения оркестратора или оператора.

Эквивалент в billable RPC при **1 500 000** кредитов и **30** кредитах за вызов: **~50 000** RPC/день → ориентиры: Pump **~22 500**; Raydium **~11 000**; Meteora **~5 500**; Orca **~5 000**; Moonshot **~3 500**; резерв **~2 500**.

### 7.2 Пол Gecko

HTTP не обязан совпадать с долями RPC один‑в‑один: для **`pumpswap`** допускается больший **`max_pages`** на те же слоты времени (§5). Суммарный **RPM** остаётся глобально ограниченным.

### 7.3 Floor и cap по lane

Чтобы узкий lane не «умер» при недельной калибровке:

- **`weight_floor_pct`** — минимум **3%** назначаемого RPC‑пула (не считая резерва), пока lane включён в конфиге.
- **`weight_cap_pct`** — максимум **60%** одного lane после пересчёта (защита от одной аномальной недели).

## 8. Недельный пересмотр стратегии (автоматический + ручной знак)

### 8.1 Окно и метрики

Раз в **7 календарных дней** (UTC) для каждого `lane_id` собираются:

- **`new_wallets_lane`** — число строк в **`wallets`**, у которых `first_seen_at` попадает в окно и  
  `metadata.seed_lane = lane_id` (или согласованный ключ).
- **`credits_spent_lane_est`** — сумма оценочных кредитов по счётчику RPC оркестратора за окно (или billable × коэффициент).
- **`gecko_http_lane`** — суммарные HTTP за окно (для контроля «дешёвых» lane, не раздувающих RPC).

**Эффективность:**

\[
\text{score}_i = \frac{\text{new\_wallets}_i}{\max(\varepsilon,\ \text{credits\_spent}_i)}
\]

рекомендуемое \(\varepsilon\) — эквивалент **минимального порога кредитов** за неделю (напр. 10 000 кредитов), чтобы не делить на ноль и не взвешивать шум.

### 8.2 Перераспределение

1. Нормализовать `score_i` по всем активным lane.
2. Вычислить новые доли **`target_pct_i`** как выпуклую комбинацию:
   - **70%** — от текущих стартовых весов §7.1 (якорь стабильности);
   - **30%** — от нормализованных `score_i`.
3. Применить **floor/cap** из §7.3.
4. Нормализовать сумму к **100%** назначаемого пула; излишек/недостачу положить в **резерв** или распределить пропорционально.

### 8.3 Артефакты

- Файл **`data/wallet-orchestrator-weekly-report.json`** (или запись в БД) с таблицей метрик и новыми весами.
- Лог **`orch_weekly_rebalance`** с полями `old_weights`, `new_weights`, `scores`.

### 8.4 Ручное вето

Оператор может зафиксировать **`ORCH_WEIGHT_OVERRIDE_JSON`** (время ограничения до следующего цикла) — documented в RUNBOOK; спека не запрещает.

## 9. Контракт `metadata` для строк `wallets`

Минимум ключей (наследие W6.7 + расширение):

- `gecko_multi_seed: true`
- `seed_lane`: `pumpswap` \| `raydium` \| …
- `job_type`: `new_pools` \| `trending_pools` \| …
- `seed_pool`, `seed_base_mint`, `seed_quote_mint`, `seed_ts`
- `collector_id`: константа оркестратора, напр. **`sa-wallet-orch`**
- `collector_semver`
- `batch_id` — по правилам PI‑5 W6.7 **на процесс** или **на calendar day** для оркестратора (выбрать одно в реализации и зафиксировать здесь при первом коммите).

**Идемпотентность:** сохранить **`ON CONFLICT (address) DO NOTHING`** до отдельной спеки на merge `metadata`.

## 10. Наблюдаемость и пилотная диагностика

1. Каждый job — структурированный JSON‑лог: lane, тип, длительность, gecko_http, rpc, wallets_inserted, wallets_unique, ошибки.
2. Переиспользовать идеи **`sa-grws-pilot-diagnose.mjs`** / **`sa-grws-analytics.mjs`**: сценарные прогоны и отчёты по **`seed_lane`**.
3. Алерты (опционально): серия job с **нулевыми** пулами после фильтра; исчерпание **80%** дневного RPC до UTC‑полуночи.

---

## 11. Критерии приёмки

1. Один процесс оркестратора в PM2 (или эквивалент) **не** конфликтует по имени с торговыми `sa-*` коллекторами.
2. DexScreener **нигде** не вызывается в коде оркестратора.
3. Суммарный Gecko RPM за минуту при типовом расписании **не превышает** целевой CPM с запасом (измерение логами за 24 ч).
4. Дневной счётчик RPC **не превышает** лимит, выведенный из **`SA_ORCH_MAX_QUICKNODE_CREDITS_PER_DAY`**.
5. Веса lane соответствуют §7.1 на старте и §8 после первой недели тестового прогона (отчёт приложен).
6. Документ **`README.md`** в этом каталоге содержит ссылку на W6.8.
