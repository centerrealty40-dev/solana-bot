# Концепт: staging, версии коллекторов, теги, архивирование

**Контур:** Smart Lottery V2 — см. [`README.md`](./README.md).

## 1. Staging vs canonical («отдельный пул»)

**Пилотный коллектор W6.7 (v0.1.0):** запись **напрямую в `wallets`** без отдельной staging-таблицы; конфликт PK — **`DO NOTHING`**. Подробности — §6.6 в [`W6.7_gecko_raydium_wallet_seed_collector_local.md`](./W6.7_gecko_raydium_wallet_seed_collector_local.md).

**Дальнейшие фазы (выбрать при масштабировании):**

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **Отдельная таблица** (`wallet_seed_raw` и т.п.) | Чистое разделение; canonical не засоряется | Доп. JOIN при промоушене |
| **Та же `wallets` + metadata** (`lifecycle: seed`) | Проще запросы | Риск смешения с прод-смыслом строки |

**Промоушен:** явное правило перевода `seed → verified` после минимальных проверок или после первого успешного тегирующего прохода.

## 2. Версионирование коллекторов (обязательно)

Каждый прогон записывает:

- **`collector_id`** — стабильное имя (`sa-grws`, …).  
- **`collector_semver`** — `MAJOR.MINOR.PATCH`:  
  - **MAJOR** — меняется семантика извлечения адресов или фильтра источника; старые батчи несопоставимы с новыми для некоторых метрик.  
  - **MINOR** — новые поля, дополнительный источник в том же контуре.  
  - **PATCH** — исправления без смены смысла.  
- **`batch_id`** — уникальный id прогона (UTC + хеш параметров или serial).  
- **`ingested_at`**.

Аналитика и отладка: фильтры вида «исключить batch X», «только semver ≥ 1.2.0».

Расширение контракта `metadata` в спеке W6.7 — см. §6.3 там после дополнения при имплементации.

- `wallet`, `source_type`, `collector_semver`, `batch_id`, `observed_at`, опционально `payload_ref`.

## 4. Система тегов

### 4.1 Классы

| Класс | Примеры | Заметки |
|-------|---------|---------|
| Structural / negative | `high_frequency_bot`, `farm_linked`, `cex_hot` | Высокий приоритет при фильтрации кворума |
| Behavioral | `raydium_active`, ранние паттерны | Пересчёт чаще |
| Quality | `human_likely`, `needs_review` | Временные |
| Outcome | альфа по истории минтов | Зависят от лейблов успеха |

### 4.2 Версии правил

У каждого правила: **`rule_id`**, **`rule_semver`**, **`effective_from`**.  
Запись тега хранит **`rule_version`** (или hash конфига), чтобы пересчитывать выборочно при смене логики.

### 4.3 Обработка

- Очередь **`wallet_tag_jobs`** (или аналог) с приоритетом.  
- Воркер: батч → признаки (Atlas, граф, частота, кластер) → upsert.  
- **Идемпотентность:** уникальность `(wallet, tag, rule_version)` или явный флаг `is_current`.

### 4.4 Current vs history

- **Текущее состояние** — быстрые фильтры для продукта.  
- **История** — append-only или поля `revoked_at`, `superseded_by` для аудита и отката моделей.

### 4.5 Приоритеты при конфликте

Документ уровня продукта: например `farm_linked` подавляет `human_likely`. Не размазывать по скриптам без SSOT.

## 5. Жизненный цикл: неактивность (~90 дней)

**Определение:** нет релевантных событий в **ваших** данных (`swaps`, опционально обновления `entity_wallets`/трейса) за **90 суток** — не путать с «тишиной в блокчейне, который мы не индексируем».

**Эскалация:**

1. Флаг **`stale_candidate`** / тег `inactive_90d`.  
2. **Archive:** перенос тяжёлых исторических полей в архивную таблицу или партицию по времени.  
3. **Удаление:** только по явной политике; для canonical чаще **`archived_at`** + исключение из активных выборок; жёсткое DELETE — только без FK-боли и с бэкапом.

**Реактивация:** повторный ingest или новый swap → снять архивный статус, поставить в очередь тегирования.
