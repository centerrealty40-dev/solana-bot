# W6.12 — S01: Единый учёт QuickNode-кредитов для «тяжёлых» ingest-джоб

**Тип:** normative implementation spec  
**Зависимости:** нет (фундамент для S02–S05)  
**Связанный обзор:** [`W6.12_OVERVIEW_detective_without_chain_firehose.md`](./W6.12_OVERVIEW_detective_without_chain_firehose.md)

---

## 1. Цель

Исключить ситуацию, когда несколько процессов (оркестратор кошельков, sigseed worker, wallet trace, будущий nightly backfill) **независимо** потребляют QuickNode до превышения **одного** согласованного дневного лимита (**по умолчанию 1_500_000 кредитов/сутки UTC**), без общего учёта.

После S01 оператор должен видеть **один источник правды**: сколько кредитов израсходовано за UTC-день и сколько осталось, с разбивкой по **классам джоб** (labels).

## 2. Область (scope)

**Входит:**

- Общий **persisted state** (JSON или строка в Postgres — решение в имплементации) с полями минимум:
  - `day` — ключ UTC-даты `YYYY-MM-DD`;
  - `credits_used_day` — сумма зарезервированных/учтённых кредитов за день;
  - `by_component` — объект `{ component_id: credits_used }` для диагностики;
  - `updated_at_ms`.
- Библиотека-хелпер **или** общий модуль, который **перед** billable RPC-вызовом проверяет остаток и **атомарно** (с учётом гонок между процессами — см. §4) инкрементирует usage.
- Конвенция env:
  - `SA_QN_GLOBAL_CREDITS_PER_DAY` — потолок кредитов (default **1500000**, `Math.max` с разумным минимумом на уровне кода);
  - `SA_QN_CREDITS_PER_RPC` — коэффициент перевода RPC→кредиты (default согласовать с существующим `QUICKNODE_CREDITS_PER_SOLANA_RPC`, типично **30**);
  - опционально `SA_QN_GLOBAL_STATE_PATH` — путь к файлу state (если файл).

**Не входит (явные non-goals):**

- Точный биллинг со стороны QuickNode (учёт по факту провайдера); только **консервативная оценка** по числу billable RPC.
- Объединение с не-RPC расходами (Gecko HTTP и т.д.) — только QuickNode JSON-RPC, помеченные как billable.

## 3. Нормативные требования

1. **Единый потолок:** сумма учтённых кредитов за UTC-день не превышает `SA_QN_GLOBAL_CREDITS_PER_DAY` (с допуском на округление вниз при `floor`).
2. **Сброс по UTC:** при смене календарного дня UTC счётчики обнуляются; `by_component` обнуляется или архивируется — зафиксировать в имплементации одно поведение и документировать.
3. **Компоненты:** каждый потребитель регистрирует `component_id` из фиксированного enum строк:
   - `wallet_orchestrator`
   - `wallet_backfill` (резерв под S02)
   - `sigseed_worker`
   - `wallet_trace_worker`
   - `scam_farm_rpc_probe` (если включён отдельный RPC в детективе)
4. **Graceful stop:** при исчерпании дневного лимита джоба **не падает бесконечным retry**; логирует предупреждение с кодом вроде `QN_GLOBAL_DAY_CAP` и завершает текущий batch/tick.
5. **CLI отчёт:** одна команда или флаг `--qn-global-report`, печатающая JSON с `credits_used_day`, `credits_cap`, `remaining`, `by_component`.

## 4. Конкуренция процессов (обязательное решение)

Зафиксировать один из подходов в имплементации и в этом документе (после выбора — дописать подпункт):

- **A)** Postgres advisory lock + строка `qn_usage_ledger` (предпочтительно при нескольких PM2-процессах); или  
- **B)** один координирующий процесс (нежелательно); или  
- **C)** файл с `fsync` + lock (допустимо только если гарантирован один хост и согласован lock).

Критерий приёмки: два параллельных процесса не могут совместно превысить дневной cap в интеграционном тесте или сценарии dry-run.

## 5. Интеграция с существующим кодом

- Оркестратор (`scripts-tmp/sa-wallet-orchestrator.mjs`) уже ведёт локальный state RPC; **цель S01** — либо **делегировать** учёт в общий модуль, либо **дублировать запись** в глобальный ledger (избегать расхождения — предпочтительно единый модуль).
- Sigseed / wallet trace — подключить к тому же модулю в отдельных задачах (ссылки на PR).

## 6. Критерии приёмки (acceptance)

1. Документированы env и путь/state.
2. Реализован общий модуль + отчёт CLI.
3. Как минимум **один** существующий потребитель (рекомендуется оркестратор) использует глобальный ledger; остальные — в задачах follow-up с отсылкой на S02–S03.
4. В логах при cap есть стабильный machine-readable `code`.

## 7. Риски

- Расхождение коэффициента кредитов с реальным тарифом QuickNode → занизить/завысить расход; mitigation: env override и quarterly review.
