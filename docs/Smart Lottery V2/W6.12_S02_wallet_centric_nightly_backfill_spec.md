# W6.12 — S02: Wallet-centric nightly backfill → `swaps` и `money_flows`

**Тип:** normative implementation spec  
**Зависимости:** [`W6.12_S01_unified_rpc_budget_counter_spec.md`](./W6.12_S01_unified_rpc_budget_counter_spec.md) (глобальный ledger обязателен до прод-включения)  
**Связанный обзор:** [`W6.12_OVERVIEW_detective_without_chain_firehose.md`](./W6.12_OVERVIEW_detective_without_chain_firehose.md)

---

## 1. Цель

Регулярно (по умолчанию **раз в сутки UTC**) обогащать аналитические таблицы **`swaps`** и **`money_flows`** **только для адресов, уже присутствующих в БД** (`wallets` и/или очередь), с **жёсткими верхними границами** на:

- число кошельков за прогон;
- страницы `getSignaturesForAddress`;
- число `getTransaction` за кошелёк за прогон;
- глобальный дневной лимит кредитов (через S01).

Это заменяет по смыслу **неограниченный** chain-wide parser/stream для целей scam-farm SQL-фазы.

## 2. Источники адресов (приоритет обработки)

Нормативный порядок выборки кандидатов в один прогон (реализовать как SQL или материализованный список):

1. **`wallets`** с `first_seen_at` в последние **N** суток (default **3**) и/или с `metadata.seed_lane` из оркестратора (если колонка JSON доступна — фильтр опционален).
2. Пересечение с адресами, встречающимися в **pair_snapshots** / торговых снимках за последние **24h** (если есть дешёвый join по creator/volume — уточнить в имплементации по фактической схеме).
3. Хвост очереди **`wallet_backfill_queue`** (новая таблица — см. §5), если адрес ещё не обработан с watermark.

Исключать программы/системные аккаунты по списку (переиспользовать ignore-list оркестратора или вынести общий env `SA_BACKFILL_IGNORE_PUBKEYS`).

## 3. RPC-алгоритм (normative)

Для каждого выбранного адреса `W`:

1. Вызвать `getSignaturesForAddress(W, { limit: L, before })` не более **`SA_BACKFILL_SIG_PAGES_MAX`** страниц (default **2–3**).
2. Для каждой подписи в порядке от новых к старым:
   - если уже есть в дедуп-кэше (`processed_signatures` или уникальный индекс целевых таблиц) — skip;
   - вызвать `getTransaction(signature, { encoding: 'jsonParsed', maxSupportedTransactionVersion: 0 })`;
   - перед каждым billable вызовом — **резерв кредитов** через S01.
3. Из результата транзакции извлечь:
   - **swaps:** переиспользовать существующий парсер (`src/intel/parse-rpc-tx-swap.ts` и/или путь sigseed parse), чтобы формат строк **`swaps`** совпадал с текущей схемой (никаких вторых форматов).
   - **money_flows:** переиспользовать логику Atlas writer или эквивалентный нормализатор переводов SOL/SPL, совместимый с колонками `money_flows` (см. `src/atlas/writer-flows.ts`).

**Запрет:** не вызывать неограниченный deep scan «до генезиса» для каждого кошелька в одном прогоне.

## 4. ENV (минимальный контракт)

| Переменная | Назначение | Пример default |
|------------|------------|----------------|
| `SA_BACKFILL_ENABLED` | `0/1` включает cron/PM2 джобу | `0` до готовности |
| `SA_BACKFILL_MAX_WALLETS_PER_RUN` | верхняя крышка адресов за прогон | `500` |
| `SA_BACKFILL_SIG_PAGES_MAX` | страницы подписей на кошелёк | `3` |
| `SA_BACKFILL_MAX_TX_PER_WALLET` | лимит `getTransaction` на кошелёк за прогон | `40` |
| `SA_BACKFILL_RPC_SLEEP_MS` | троттлинг между RPC | `200` |
| `SA_BACKFILL_PRIORITY_FRESH_HOURS` | окно «свежих» из §2 | `72` |

Все лимиты должны подчиняться **S01** (глобальный дневной cap имеет приоритет).

## 5. Очередь и идемпотентность

**Новая таблица** `wallet_backfill_queue` (имена могут быть скорректированы миграцией, но семантика обязательна):

- `address` PK или unique;
- `priority` numeric;
- `last_run_at`, `last_signature_cursor` (opaque), `runs_count`;
- `status`: `pending | running | done | error`;
- `error_message` text nullable.

Идемпотентность вставок: **`swaps`** и **`money_flows`** только через существующие `ON CONFLICT` / уникальные ключи; повторный прогон не должен раздувать дубликаты.

## 6. Наблюдаемость

- Структурированный лог JSON: `component=wallet_backfill`, `run_id`, `wallets_processed`, `tx_fetched`, `swaps_inserted`, `flows_inserted`, `credits_est`.
- Метрика для doctor (опционально в S05): счётчики за 24h по таблицам.

## 7. Критерии приёмки

1. Поднята миграция очереди + watermark полей.
2. Реализован скрипт `npm run wallet-backfill:run` (или эквивалент) с dry-run режимом.
3. Интеграция с S01: при исчерпании дня прогон завершается корректно.
4. Документирован cron (в [`deploy/RUNTIME.md`](../../deploy/RUNTIME.md) после деплоя — отдельная задача оператора).

## 8. Зависимости на другие шаги

- **S04:** для полноты правил детектива желательно запускать **после** появления минимальных `money_flows` или совместно (порядок в операторском чеклисте S05).
