# W6.14 — Спецификация: оптимизация scam-farm analyzer (граф, treasury, мета-кластеры)

**Статус:** нормативная спека для последующей реализации (design-first).  
**Область:** продукт **solana-alpha**, модуль `src/intel/scam-farm-detective/**`, таблицы Atlas (`wallet_clusters`, `entity_wallets`, `wallet_tags`), факты **`money_flows`** / **`swaps`** / **`tokens`** / **`scam_farm_candidates`**.  
**Связанные документы:** [W6.12_OVERVIEW](./W6.12_OVERVIEW_detective_without_chain_firehose.md), [W6.12_S05](./W6.12_S05_detective_operational_readiness_spec.md), [W6.11](./W6.11_intel_policy_dashboard_operator_spec.md), `deploy/RUNTIME.md`.

---

## 1. Зачем этот документ

Текущий детектив (фаза **A**, SQL) находит **локальные** паттерны (`sync_fund`, `rug_cohort`, `orchestrate_split`), создаёт **кандидата** и при выполнении условий записывает в Atlas **одну строку `wallet_clusters` на кандидата** и теги `scam_operator` / `scam_proxy` с `source = scam_farm_detective`.

Операторский опыт и объём данных показывают:

- много **мелких** кластеров (часто 2–3 кошелька на кандидата) — это **ожидаемо**, а не признак «сломанной модели»;
- **не хватает** второго слоя: **общие стоки (treasury / sink)**, **склейки кандидатов в «фермы»**, **цепочек relay**, **временной синхронности**, опционально — осторожных эвристик **CEX deposit**.

Эта спека фиксирует **цели**, **таксономию новых тегов**, **алгоритмы**, **техническую реализацию**, **лимиты**, **порядок внедрения** и **критерии успеха**.

## 2. Термины (единый словарь)

| Термин | Значение в системе |
|--------|---------------------|
| **Кандидат** | Запись в `scam_farm_candidates`; ключ `candidate_id` = SHA256 от `(funder \| отсортированные кошельки \| отсортированные якорные mint)`. Правила **сливаются** в одну строку при совпадении ключа. |
| **Кластер Atlas (строка `wallet_clusters`)** | Создаётся **один раз на успешную запись Atlas** для данного кандидата (`detected_by = scam_farm_detective`). Это **не** «вся ферма», а **одна зафиксированная гипотеза**. |
| **Мета-кластер (новое)** | Логическая группа, объединяющая несколько кандидатовов/кластеров по графовым признакам (общий sink, общий funder-компонент, связность по потокам). Представление в БД — см. §7. |
| **Фаза A** | Существующий SQL-детектив + опциональный RPC-probe (без изменения семантики ключей кандидата в MVP мета-слоя). |
| **Фаза B** | Новые **агрегирующие** шаги поверх фактов и/или поверх уже помеченных кошельков (treasury, merge, relay, …). |
| **Sink / treasury** | Адрес-получатель, на который в окне наблюдения сходится поток от **многих** источников из заданного множества (подозрительные кошельки, участники кандидатов и т.д.). |

## 3. Цели и ограничения

### 3.1 Цели

1. **Выявлять общий treasury** и родственные стоки (единый «кошелёк-сборщик» для многих операторов).
2. **Снижать фрагментацию** восприятия: склеивать связанные кандидаты в **мета-кластеры** для аналитики и UI (без обязательного удаления мелких кластеров).
3. **Расширить таксономию тегов** осмысленными, **политически разделёнными** `source` (см. §6), чтобы политика mint-gate и отчёты могли отличать «роль в схеме» от «локального паттерна».
4. Сохранить **budget-first**: тяжёлые запросы только на суженных множествах, с **жёсткими лимитами** строк/узлов за прогон.
5. Обеспечить **идемпотентность** записи в `wallet_tags` / обновлений графа (повторный прогон не размножает шум).

### 3.2 Не-цели (явно вне MVP)

- Построение **полного** chain-wide графа Solana.
- Высокоточная **идентификация CEX** без allowlist / без ручной калибровки.
- Автоматический **бан** mint или автоматические торговые действия только по новым тегам (только интеллект + политика оператора).

## 4. Архитектура пайплайна (целевое состояние)

```
[ингест: money_flows, swaps, wallets, tokens]
        │
        ▼
┌───────────────────────┐
│ Фаза A (существует)    │  sync_fund / rug_cohort / orchestrate_split
│ scam-farm-detective    │  → scam_farm_candidates → wallet_clusters (мелкие)
└───────────┬───────────┘
            │
            ▼
┌───────────────────────┐
│ Фаза B1 — Treasury     │  агрегация исходящих потоков → sink-кандидаты
└───────────┬───────────┘

            ▼
┌───────────────────────┐
│ Фаза B2 — Meta-merge   │  компоненты связности / склейка кластеров
```

## 5. Принципы данных и конфликтов в Atlas

### 5.1 `entity_wallets.cluster_id`

Сегодня при записи кандидата выполняется upsert с **перезаписью** `cluster_id` на последний кластер. Для мета-слоя это ограничение UI/аналитики.

**Требование спеки:**

- **Не ломать** существующий путь без миграции оператора.
- Ввести **параллельную** привязку к мета-кластеру (§7.2) **или** документированный операторский режим «primary cluster = meta» (флаг ENV, по умолчанию off).

### 5.2 Идемпотентность тегов

Использовать существующий ключ `(wallet, tag, source)` в `wallet_tags`. Новые теги — **новые значения `tag` и/или `source`**, чтобы не пересекаться с `scam_farm_detective`.

## 6. Таксономия новых тегов и источников

### 6.1 Новые значения `wallet_tags.source`

Рекомендуемая гранулярность (один исполняемый модуль может писать несколько тегов, но source разделяет политику):

| `source` | Назначение |
|----------|------------|
| `scam_farm_graph` | Агрегаты по графу потоков: sink, relay-hub (если выделены как тег на кошелёк). |
| `scam_farm_meta` | Мета-кластер: принадлежность к «ферме» как к надгруппе. |
| `scam_farm_temporal` | Временная синхронность (когда появятся данные). |

`scam_farm_detective` **остаётся** только за фазой A (оператор / политика не смешивают поколения сигналов).

### 6.2 Предлагаемые `wallet_tags.tag`

Все имена — латиница, нижний регистр, длина ≤ 32 (как в схеме).

| Тег | Кому ставится | Смысл | Фаза |
|-----|----------------|-------|------|
| `farm_sink` | Адрес получателя | Много **различных** источников из множества S отправляют в один target за окно T (SOL или агрегированно). | B1 |
| `farm_treasury` | Подмножество `farm_sink` | Усиленный порог + опционально связь с уже помеченными `scam_operator` / кандидатами. | B1 |
| `relay_hub` | Промежуточный кошелёк | Высокая степень «many-to-one-to-many» в короткой цепочке A→H→B (ограниченная глубина). | B3 |
| `farm_time_cohort` | Кошелёк | Участник сжатого временного окна совместной активности с другими (по `swaps`). | B4 |
| `cex_deposit_hint` | Адрес | Эвристика совпадения с allowlist депозитных адресов (очень низкая автономная сила). | B5 |

**Политика confidence:** целое 0–100; для B5 по умолчанию **низкая** (например 25–40) + обязательный `context` с причиной и id списка.

## 7. Представление мета-кластеров в БД

### 7.1 Вариант A (рекомендуемый MVP): отдельная таблица связей

**Новая таблица** `scam_farm_meta_clusters` (имя обсуждаемо):

- `meta_id` UUID или bigserial PK  
- `label` text (опционально)  
- `created_at`, `updated_at`  
- `detection_reason` jsonb (sink seeds, edge counts, параметры прогона)  
- `confidence` integer  

**Таблица членства** `scam_farm_meta_cluster_members`:

- `meta_id` FK  
- `wallet` varchar(64)  
- `role` varchar(24) — например `sink`, `operator`, `funder`, `relay`, `unknown`  
- PK `(meta_id, wallet)`  

**Связь кандидат ↔ мета:**

- `scam_farm_meta_cluster_candidates(meta_id, candidate_id)` с PK `(meta_id, candidate_id)`.

Плюсы: не ломает текущую семантику `wallet_clusters`; удобно для дашборда и последующего слияния.

### 7.2 Вариант B: использовать `wallet_clusters.merged_into_id`

Подходит для **явного дерева слияний** мелких кластеров в крупный «родительский» кластер.

- Мелкие строки `wallet_clusters`: `merged_into_id → id` родителя.  
- Родитель: `kind = farm_meta` (новое значение), `detected_by = scam_farm_meta`.

Минусы: перегрузка смысла поля `merged_into_id`; сложнее хранить не-кластерные узлы (sink без собственного детективного кластера).

**Рекомендация:** начать с **варианта A**; вариант B — опциональная визуальная оптимизация.

### 7.3 Теги членства в мета-кластере

Для каждого кошелька в мета-кластере:

- `wallet_tags`: `tag = farm_meta_member`, `source = scam_farm_meta`, `context` = `{ metaId, role, candidateIds[], rules[] }` (усечь до лимита поля).

Для **sink**:

- `tag = farm_treasury` или `farm_sink`, `source = scam_farm_graph`.

## 8. Фаза B1 — Treasury / Sink

### 8.1 Входные множества

Параметры ENV (префикс `SCAM_FARM_SINK_*`):

| ENV | Смысл | Стартовый default (подбор оператором) |
|-----|--------|-------------------------------------|
| `SCAM_FARM_SINK_LOOKBACK_HOURS` | Окно по `money_flows.tx_time` | 72 |
| `SCAM_FARM_SINK_MIN_SOURCES` | Минимум **различных** `source_wallet` на один `target_wallet` | 8 |
| `SCAM_FARM_SINK_MIN_TAGGED_SOURCES` | Альтернативно: минимум источников из union тегов `scam_operator` | 3 |
| `SCAM_FARM_SINK_ASSET` | `'SOL'` или список | `SOL` (MVP) |
| `SCAM_FARM_SINK_MAX_TARGETS_PER_RUN` | Лимит строк-ответа | 500 |
| `SCAM_FARM_SINK_MIN_TOTAL_SOL` | Порог суммарного объёма (анти-шум пыли) | настраивается |
| `SCAM_FARM_SINK_EXCLUDE_SYSTEM` | Список/паттерны исключений (программы, burn, известные сервисы) | постепенно |

**Два режима** (оба могут работать в одном прогоне, разные INSERT):

1. **Широкий:** все потоки в lookback, фильтр только по структуре графа (дороже).  
2. **Узкий:** только исходящие рёбра от кошельков из:

   - `wallet_tags` где `tag IN ('scam_operator','scam_proxy')` и `source = scam_farm_detective`, **или**
   - `participant_wallets` из `scam_farm_candidates` за активные статусы (конфигурируемо).

Режим 2 — **основной для продакшена** из соображений стоимости.

### 8.2 Ядро SQL (черновик логики)

На узком множестве источников `S` (материализовать во временную таблицу или CTE с лимитом):

```sql
-- Псевдокод: агрегация исходящих SOL-потоков от S
SELECT target_wallet,
       COUNT(DISTINCT source_wallet) AS n_sources,
       SUM(amount) AS total_sol
FROM money_flows
WHERE asset = 'SOL'
  AND tx_time > now() - (:lookback_hours || ' hours')::interval
  AND source_wallet IN (SELECT wallet FROM seed_wallets LIMIT :seed_cap)
  AND source_wallet <> target_wallet
GROUP BY target_wallet
HAVING COUNT(DISTINCT source_wallet) >= :min_sources
ORDER BY n_sources DESC, total_sol DESC
LIMIT :max_targets;
```

**Усиление до `farm_treasury`:** повторить с более строгими порогами **или** пересечь с тем, что `target_wallet` принимает от ≥ K кошельков, уже состоящих в **разных** `candidate_id` (через джойн `scam_farm_candidates` и `jsonb_array_elements_text(participant_wallets)` — осторожно с производительностью; для прод — вынести участников в нормализованную витрину §9).

### 8.3 Выходы

- Upsert строк в `scam_farm_meta_clusters` + роль `sink` для target.  
- `wallet_tags` для target: `farm_sink` или `farm_treasury`.  
- Лог JSONL: `{ ok, phase: "B1", targets: N, seeds: M, durationMs }`.

### 8.4 Ложные срабатывания

- Централизованные сервисы (миксеры, мосты) — **exclude list**.  
- Один спам-бот, размножающий мелкие кошельки → mitigated через `MIN_TOTAL_SOL` и минимальный размер перевода (опционально ENV).

## 10. Фаза B2 — Мета-кластер (склейка)

### 10.1 Граф

**Вершины:** кошельки из объединения:

- участники кандидатов (желательно из витрины §9.1),  
- sink-кандидаты из B1.

**Рёбра (вес опционален):**

- прямой SOL-flow между двумя вершинами за окно T с `amount ≥ min_edge_sol`;  
- или «совместная связь с одним sink»: два оператора → один и тот же target (порог по времени).

### 10.2 Алгоритм MVP

- Построить разреженный список рёбер (лимит `SCAM_FARM_META_MAX_EDGES`).  
- Найти **connected components** (Union-Find / BFS в памяти после выборки из SQL).  
- Отфильтровать компоненты с `|V| < SCAM_FARM_META_MIN_WALLETS` (например 4).  
- Для каждой компоненты создать `meta_id`, связать кандидатов, попавших в компонент.

### 10.3 Выходы

- Записи в таблицы §7.1.  
- Теги `farm_meta_member` для участников (не дублировать `scam_operator`, а **дополнять**).

## 11. Фаза B3 — Relay hub

### 11.1 Определение

Кошелёк H — relay-кандидат, если за окно T существует множество пар (A,B), где есть поток A→H и H→B (SOL), с ограничениями:

- `COUNT(DISTINCT A) ≥ n1`, `COUNT(DISTINCT B) ≥ n2`;  
- медианный лаг между ногами ≤ `SCAM_FARM_RELAY_MAX_LAG_SEC`.

### 11.2 Реализация

- MVP: SQL с self-join по `money_flows` на фильтрованном множестве (дорого) **или** только для кошельков из B2-компонент.  
- Тег: `relay_hub`, `source = scam_farm_graph`.

---

## 12. Фаза B4 — Временная синхронность (cohort)

### 12.1 Данные

Требуется достаточное покрытие **`swaps`** (время, signer/initiator, mint). Спека **не** дублирует S03; фиксирует зависимость: B4 включается только если doctor подтверждает глубину.

### 12.2 Эвристика MVP

Для каждого mint (или top-N mint по активности за сутки):

- кластеризовать первые сделки покупателей по `floor(tx_time to minute)`;  
- если в одном минутном бакете ≥ K уникальных buyer wallets — пометить их **слабым** тегом `farm_time_cohort`, `source = scam_farm_temporal`, низкая confidence.

Пороги через `SCAM_FARM_TEMPORAL_*`.

---

## 13. Фаза B5 — CEX deposit hint

### 13.1 Подход

- Только **allowlist** известных депозитных адресов (файл/таблица `cex_deposit_allowlist`), версионируемый оператором.  
- Если `target_wallet` из B1 совпадает с allowlist → `cex_deposit_hint`, очень низкая confidence + ссылка на id записи списка.

### 13.2 Риски

Ложные и стратегические изменения адресов бирж — документировать как **информационный** сигнал.

## 18. Критерии успеха продукта

1. Оператор находит **общий treasury** для групп уже известных операторов **без ручного SQL**.  
2. Число **осмысленных «ферм»** на дашборде (мета-кластеры) **меньше**, чем число сырых кандидатов, при сохранении трассируемости к `candidate_id`.  
3. Ложноположительные новые теги **контролируются** порогами и лимитами; есть отчёт «сколько новых тегов за сутки».  
4. Политика ([W6.11](./W6.11_intel_policy_dashboard_operator_spec.md)) может **различать** `scam_farm_detective` vs `scam_farm_graph` vs `scam_farm_meta`.

## 20. Rollback

- `SCAM_FARM_GRAPH_ENABLED=0` — отключает фазу B.  
- Удаление тегов с `source IN ('scam_farm_graph','scam_farm_meta','scam_farm_temporal')` — по операторскому скрипту (отдельный maintenance SQL, не часть регулярного прогона).  
- Таблицы мета-кластеров можно оставить пустыми без влияния на фазу A.
