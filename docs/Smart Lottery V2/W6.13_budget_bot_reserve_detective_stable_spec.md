# W6.13 — Бюджет 1.5M кредитов/сутки: резерв под ботов, стабильный оркестратор + детектив

**Статус:** normative (целевая архитектура и операторские инварианты)  
**Продукт:** solana-alpha  
**Основание:** прогон VPS после W6.12 (ledger, smoke backfill, `wallet-funding:backfill`, `scam-farm:detect`), оценка расхода кредитов и пробелов по `swaps` / компонентам.

---

## 1. Цель

Построить **устойчивый** контур на платном RPC QuickNode при жёстком лимите **1 500 000 кредитов за календарные сутки UTC**, с разделением:

| Блок | Доля от лимита | Кредиты / сутки (типично) | Назначение |
|------|----------------|---------------------------|------------|
| **Операционный пул** | **≤ 70%** | **≤ 1 050 000** | Оркестратор кошельков, nightly wallet-backfill, sigseed/trace (когда подключены к ledger), редкий RPC детектива, служебные CLI |
| **Резерв** | **≥ 30%** | **≥ 450 000** | Будущий **анализатор ботов** (нечеловеческие паттерны), непредвиденные пики, буфер под другие фичи на том же endpoint |

**Инвариант:** сумма учтённых кредитов за UTC-день по **`sa_qn_global_daily`** (все `component_id`) **не превышает 1 050 000** в штатном режиме; целевой остаток к концу типичного дня **не ниже ~450 000** «неиспользованных» от полного лимита 1.5M (т.е. не выжигать резерв проактивно).

**Примечание:** 70% — это **целевой потолок потребления**, а не обязательство потратить ровно столько. Недобор допустим и желателен в спокойные дни.

---

## 2. Перевод кредитов в RPC (для планирования)

При коэффициенте **`QUICKNODE_CREDITS_PER_SOLANA_RPC = 30`** (дефолт в коде W6.12):

| Пул | Кредиты / сутки | Эквивалент billable RPC / сутки |
|-----|-----------------|----------------------------------|
| Полный лимит | 1 500 000 | 50 000 |
| Операционный потолок (70%) | 1 050 000 | **35 000** |
| Резерв (30%) | 450 000 | **15 000** |

Фактический смок на VPS: **45 RPC → 1 350 кредитов** за ограниченный backfill — масштабируется линейно по числу `getSignaturesForAddress` / `getTransaction`.

---

## 3. Рекомендуемое распределение операционного пула (~1.05M кредитов)

Нормативные **мягкие потолки по компонентам** (задаются ENV / политикой; сумма не должна превышать 1.05M при одновременном «полном газе» всех сервисов):

| Компонент | `component_id` | Рекомендуемый потолок кредитов / сутки | Комментарий |
|-----------|----------------|----------------------------------------|-------------|
| Оркестратор кошельков | `wallet_orchestrator` | **550 000 – 700 000** | Основной потребитель; держать **`SA_ORCH_MAX_QUICKNODE_CREDITS_PER_DAY`** согласованным с глобальным пулом (см. §7) |
| Wallet backfill | `wallet_backfill` | **80 000 – 180 000** | Nightly или реже; лимиты `SA_BACKFILL_*`; приоритет адресов с pump/DEX-метаданными |
| Sigseed worker | `sigseed_worker` | **100 000 – 250 000** | После подключения к **`jsonRpcWithQnLedger`**; узкий enqueue (`ATLAS_ENQUEUE_*`) |
| Wallet trace | `wallet_trace_worker` | **50 000 – 150 000** | Когда процесс на ветке; очередь + batch |
| Scam-farm RPC probe | `scam_farm_rpc_probe` | **≤ 30 000** | Только если **`SCAM_FARM_ENABLE_RPC=1`**; по умолчанию SQL-фаза без RPC |
| **Про запас внутри 70%** | — | **≥ 50 000** | Джиттер, ретраи, сезонные пики Gecko без RPC |

Если сумма нижних границ превышает 1.05M — уменьшать **sigseed** или **backfill**, а не резерв 30%.

## 4. Резерв под анализатор ботов (30%)

**Назначение:** отдельный продуктовый контур (очередь кошельков → признаки поведения → теги / скоринг), который **не имеет права** по умолчанию исчерпывать общий лимит 1.5M совместно с оркестратором.

Нормативные требования к будущей реализации анализатора:

1. Собственный **`component_id`** в ledger, например **`bot_pattern_analyzer`**.
2. Дневной потолок **`SA_BOT_ANALYZER_MAX_CREDITS_PER_DAY`** ≤ **450 000** (или ниже, если операционный пул в этот день близок к 1.05M — приоритет у оркестратора задаёт оператор).
3. При конфликте: **глобальный hard stop** на **1.5M** остаётся в `sa_qn_global_daily`; анализатор обязан корректно завершать batch при `QN_GLOBAL_DAY_CAP`.
4. Очередь и приоритеты: не делать full-scan всех `wallets` без лимита; инкрементальные окна (`first_seen_at`, метки оркестратора).

Детальная спека правил ботов — отдельный документ (не входит в W6.13).

## 5. Недочёты, выявленные при оценке (факт прогона)

1. **`swaps_last_24h = 0`** при живых потоках **`money_flows`** → правила **`rug_cohort`** и **`orchestrated_split`** не получают входные данные по свопам.
2. **Wallet-backfill** сейчас декодирует **в основном pump.fun** → для «случайных» адресов из очереди **вставка в `swaps` часто нулевая**; расход RPC есть, полезный выход по свопам мало.
3. **Sigseed / wallet-trace** на VPS могут обходить общий ledger → риск «тихого» превышения 1.5M без единой картины.
4. **`wallet-funding:backfill`** эффективен при наличии **`money_flows`**, но UI вывода JSON был исправлен (не смешивать аргументы `console.log`).
5. Детектив **`sync_fund`** может давать **тысячи кандидатов** при больших таблицах — нужны **операторские потолки** (`SCAM_FARM_MAX_SQL_ROWS`, качество данных), чтобы не перегружать Atlas/Postgres и не размывать продукт шумом.

## 6. Доработки по приоритетам

### P0 — стабильность бюджета и наблюдаемости

| ID | Задача | Критерий приёмки |
|----|--------|------------------|
| P0.1 | Жёстко связать **`SA_QN_GLOBAL_CREDITS_PER_DAY`** с суммой операционных потолков: при превышении целевого **70%** логировать **warn** при старте процессов | Документированная таблица ENV в RUNTIME |
| P0.2 | Ежедневный снимок: **`npm run sa-qn-global-report`** в лог/метрики (cron или hourly wrapper) | Файл или Telegram одной строкой: used / cap / by_component |
| P0.3 | Проверка: оркестратор + backfill не обходят ledger | Интеграционный тест или чеклист деплоя |

### P1 — полнота данных для детектива без firehose

| ID | Задача | Критерий приёмки |
|----|--------|------------------|
| P1.1 | Подключить **sigseed worker** и **parse** к **`jsonRpcWithQnLedger`** (`sigseed_worker`) | В отчёте ledger видна строка **`sigseed_worker`** при работе cron |
| P1.2 | Расширить **политику выбора адресов** для backfill: приоритет **`seed_lane`**, пересечение с mint из коллекторов, опционально whitelist программ | Рост **`swaps`** за 24h при том же лимите RPC vs случайная очередь |
| P1.3 | (Опционально) второй этап декодера свопов под **основные DEX**, используемые в `parse-rpc-tx`-семействе репозитория | Ненулевой `swaps_last_24h` при отключённом stream |

### P2 — качество и шум детектива

| ID | Задача | Критерий приёмки |
|----|--------|------------------|
| P2.1 | Настраиваемые пороги **`sync_fund`** (минимум целей, окно, исключения CEX) для снижения ложных массовых кандидатов | Документ + ENV |
| P2.2 | Связка **`wallet-intel:doctor`** с **`sa-qn-global-report`** (одна секция отчёта) | Один экран для оператора |

## 7. Согласование ENV (конкретика для оператора)

Рекомендуемый набор (подгонять под фактический трафик):

```env
# Глобально
SA_QN_GLOBAL_CREDITS_PER_DAY=1500000
QUICKNODE_CREDITS_PER_SOLANA_RPC=30

# Оркестратор — не выше ~65% полного лимита как цель (~975k), оставляя место прочим
SA_ORCH_MAX_QUICKNODE_CREDITS_PER_DAY=975000

# Backfill — жёсткий подпул (пример)
SA_BACKFILL_ENABLED=1
# SA_BACKFILL_MAX_WALLETS_PER_RUN / SIG_PAGES / TX подгонять так, чтобы средний день ≤ 120k кредитов

# Будущий анализатор ботов (пример верхней границы резерва)
SA_BOT_ANALYZER_MAX_CREDITS_PER_DAY=450000
```

**Правило:** если **`SA_ORCH_MAX_QUICKNODE_CREDITS_PER_DAY`** и локальные лимиты других сервисов в сумме дают **> 1.05M**, оператор обязан снизить один из потолков или ужесточить частоту cron.

---

## 8. Критерии «стабильный работающий продукт»

1. **Бюджет:** 7 дней подряд средний **`credits_used`** по **`sa_qn_global_daily`** ≤ **1.05M**, минимальный **`credits_remaining`** ≥ **450k** хотя бы в 5 из 7 дней (или объяснённый инцидент).
2. **Детектив:** `scam-farm:detect` выполняется по расписанию без OOM; **`rug`/`orch`** дают ненулевой вклад после P1 **или** явный операторский режим «только sync_fund» задокументирован.
3. **Doctor:** предупреждение по **`swaps_last_24h`** снято либо классифицировано как принятый режим (SQL-only детектив).
4. **Резерв:** код анализатора ботов не включён в прод без собственного **`component_id`** и потолка ≤ **450k**.
